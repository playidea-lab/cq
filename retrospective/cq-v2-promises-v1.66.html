약속 1 — 인스톨 원라인 + 위저드 ​

"원라인 설치, 위저드가 안내, 다른 걸 생각할 필요 없다"

v1.62.0 baseline: 90/100 ​

  • install.sh 원라인 동작
  • runFirstRunWizard Q1-Q4 + 한국어 + Enter 기본값 Yes
  • 한 가지 갭: GPU 서버에서 OAuth 브라우저 실패

v1.66.0 현재: 95/100 (+5) ​

개선:

  • ✅ Headless SSH 자동 감지 → --link 로그인 자동 분기 (auth.go ensureCloudAuth, v1.62)
  • ✅ wizard.go hasDisplay() 판정 + 워커 onboarding 메시지
  • ✅ cq update 자동으로 글로벌 skills + hooks + MCP config 갱신 (refreshGlobalAssets, v1.62)
  • ✅ cq intro 명령으로 첫 실행 후 "이제 뭐 하지?" 갭 해소 (T-PRO-3)

남은 갭 (-5점):

  • Windows 설치 경로 미검증 (powershell 등 별도)
  • Polar 결제 → Pro 활성화 sandbox E2E 측정 안 됨 (T-PRO-2 보류)
  • 신규 사용자 5명 UX 베타 피드백 미수집 (Phase 3 R3 deferred)

다음 마일스톤: T-PRO-2 (Polar) + T-PRO-3 deferred items.

약속 2 — 모든 LLM에서 MCP 연결 ​

"워커 / 노트북 모두 Claude/Codex/Gemini 등 모든 MCP 클라이언트로 접근"

v1.62.0 baseline: 60/100 ​

  • mcp.pilab.kr Streamable MCP 운영 중
  • Claude Code / Cursor stdio MCP 동작
  • ChatGPT remote MCP 동작
  • Codex / Gemini 검증 부재 (희망사항)

v1.66.0 현재: 75/100 (+15) ​

개선:

  • ✅ MCP initialize Instructions에 에이전트 컨벤션 자동 주입 — Edge declaration + Evidence URL 두 컨벤션
  • ✅ MCP 도구 추가 (cq_edge_declare, cq_evidence_url, cq_audit, cq_intro)
  • ✅ share=true 패턴 — MCP가 사용자 JWT 없이도 share 흐름 안내 (T-EVW-6)
  • ✅ Streamable MCP의 share token 시스템 (T-EVW-5)

남은 갭 (-25점):

  • Codex / Gemini OAuth 2.1 호환성 여전히 검증 안 됨 — 희망사항
  • Streamable MCP의 도구 수가 stdio 대비 여전히 적음 (15+ vs 130+)
  • "단순한 MCP 연결로 실험을 지시"는 stdio 한정 — Streamable은 propose만 + Telegram approval 필요
  • 외부 LLM에서 직접 실험 실행은 여전히 3계층 방어 + 사용자 승인 필수

다음 마일스톤: Codex/Gemini 실사용 검증, Streamable 도구 증설 (정책 결정 필요 — Phase 5+).

개선:

  • ✅ Phase 1 Context Injection — 개인 지식이 새 세션에 자동 흘러감 (개인 순환 완성)

새로 등장한 가치 (원래 약속에 없던 것) ​

이 4개월의 작업이 만든 부산물 가치 — 원래 5 약속 프레임에는 없었지만 product의 핵심 자산이 됨:

A. 에이전트 컨벤션 (Edge + Evidence) ​

  • 단순한 "도구"가 아니라 에이전트가 일관된 행동을 하게 만드는 프로토콜
  • MCP initialize Instructions 자동 주입 → 모든 클라이언트에서 동일 행동
  • docs/conventions/agent-conventions.md 통합 가이드 — 마케팅 자산

B. Evidence Web (사람-에이전트 검증 채널) ​

  • 약속 4의 부산물로 시작했지만 별도 product가 됨
  • "코드 안 짜는 연구자"의 검증 책임을 1분으로 단축
  • shareable 토큰 → 외부 협업 가능

C. Lighthouse / 회귀 가드 시스템 ​

  • T-EVW-7의 Lighthouse CI
  • T-CI2-16의 7-invariant E2E
  • T-EVW-5의 share-token 12 tests + revoke enforcement
  • Phase 0의 5-edge E2E

D. 운영 가시성 (audit / intro / troubleshooting) ​

  • cq audit (T-PRO-1) — 4표면 헬스 체크
  • cq intro (T-PRO-3) — 5 capability 발견
  • cq audit --alert — Telegram breach 알림
  • troubleshooting.md — 30+ 흔한 문제

교훈 (Lessons learned) ​

Wire validation 없으면 dead code 가능 (재확인) ​

  • v1.60.1 hotfix 이후에도 동일 패턴 발생 (Phase 1 워커 브랜치 미머지 사건)
  • 코드 작성 후 실사용 검증 필수 — 단위 테스트 통과 ≠ wire 연결
  • 회귀 가드: T-CI2-16 같은 E2E 패턴을 phase별 표준화 필요

Phase별 메모리 갭 발생 ​

  • Phase 1 메모리 누락 (오늘 보강)
  • 다음 phase 종료 시 즉시 memory 작성하는 워크플로우 필요 (T-XXX-7 릴리즈 태스크에 포함)

외부 의존 작업의 격리 ​

  • T-PRO-2 (Polar), T-PRO-6 (CF Pages) 가 Phase 3 진척을 막음
  • 외부 환경 셋업 작업은 별 트랙으로 분리, 코드 작업과 분리

Side-track 가치 큼 ​

  • 사이드 트랙 4개 (컨벤션 통합 / help 카테고리 / audit alert / troubleshooting) 가 Phase 3 본 작업보다 ROI 높았음
  • 운영/문서 작업의 누적 효과 = Pro 가치 증명

정리는 commit과 분리 ​

  • Worktree / branch 정리는 git status 변경 없음 — 별 commit 안 함
  • DB cleanup도 마찬가지 — application-level state 관리
