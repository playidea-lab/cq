Agent Edge Declaration Convention ​

CQ v2 Phase 0 — agents should declare semantic edges in the Timeline Graph via cq_edge_declare so the system can reconstruct research history.

Why ​

The Timeline Graph (cq_graph_edges) lets future sessions answer questions like "why did we choose lr=1e-4?" or "has anyone tried this hypothesis before?" by traversing relationships between conversations, experiments, and knowledge documents.

Without explicit edges, the graph stays empty and Context Injection (Phase 1) has nothing to surface. Declaration is your responsibility — the system will not infer edges in Phase 0.

When to call cq_edge_declare ​

Call the tool immediately after any of these actions:

ActionEdge typeExample
Submit an experiment that varies one parameter from a prior onederives_from"Same as exp-abc but lr=1e-4"
Submit an experiment that scales/extends a prior oneextends"Same as exp-abc but with full ImageNet"
Reproduce a published or prior resultreproduces"Re-run exp-abc to verify"
Record a finding that opposes a prior conclusioncontradicts"Result disagrees with insight-xyz"
Compare experiments side-by-side in a knowledge doccompared_to (one per baseline)"doc-cmp-001 compares exp-a, exp-b, exp-c"

Do not call this tool for produced_in or authored_by — those are auto-generated by the Hub completion hook.

Endpoint types ​

Both from_type and to_type accept exactly four values:

  • session — a conversation session (id is ai_sessions.session_id)
  • job — a Hub job (id is hub_jobs.id)
  • doc — a knowledge document (id is c4_documents.doc_id, e.g. exp-abc12345)
  • persona — a user (id is the user's UUID)

Example calls ​

Example 1 — derive ​

json
cq_edge_declare({
  "from_type": "job", "from_id": "<new-job-id>",
  "to_type":   "job", "to_id":   "<base-job-id>",
  "edge_type": "derives_from",
  "metadata":  {"changed": ["lr"], "from": "1e-3", "to": "1e-4"}
})

Example 2 — compare ​

After producing a comparison knowledge document doc-cmp-001:

json
cq_edge_declare({"from_type":"doc","from_id":"doc-cmp-001",
                 "to_type":"job","to_id":"job-a","edge_type":"compared_to"})
cq_edge_declare({"from_type":"doc","from_id":"doc-cmp-001",
                 "to_type":"job","to_id":"job-b","edge_type":"compared_to"})
cq_edge_declare({"from_type":"doc","from_id":"doc-cmp-001",
                 "to_type":"job","to_id":"job-c","edge_type":"compared_to"})

One call per baseline.

Example 3 — contradict ​

json
cq_edge_declare({
  "from_type": "doc", "from_id":  "doc-finding-002",
  "to_type":   "doc", "to_id":    "doc-finding-001",
  "edge_type": "contradicts",
  "metadata":  {"reason": "opposite trend on val_loss"}
})

Rules ​

  1. One call per relationship. Don't batch edges into a single call — declare them sequentially. Failures are per-edge.
  2. metadata is optional and ≤ 2 KB serialized. Use it for the reason the edge exists, not for full content.
  3. Edges are immutable. Wrong edge? Call cq_graph_edge_delete (Phase 1+) and re-declare. Do not retry the same edge.
  4. Don't try to declare structural edges (produced_in, authored_by). The tool will return structural_edge_forbidden.
  5. Don't invent endpoint types. Only session, job, doc, persona.

Failure modes ​

The tool returns {"success": false, "error": ..., "hint": ...} on validation failure. Common errors:

ErrorMeaning
structural_edge_forbiddenYou tried to declare produced_in or authored_by; only hooks may.
invalid_edge_typeUnknown semantic edge; see hint for the 5 allowed values.
invalid_from_type / invalid_to_typeUnknown endpoint kind; see hint.
missing_endpoint_idfrom_id or to_id was empty.
metadata_too_largeSerialized metadata > 2 KB.
not_authenticated / session_expiredUser must run cq auth login.
supabase_errorServer-side failure (RLS, FK, etc.). Inspect hint.

On any failure, continue your primary task — edge declaration is best-effort context enrichment, not a blocking step.
