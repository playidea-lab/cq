CQ Agent Conventions — Unified Guide ​

Single source of truth for every convention an agent must follow when talking to CQ. Each convention is also injected into the MCP initialize Instructions field so agents learn it before their first tools/list call — but this page is the canonical reference.

Why conventions exist ​

CQ records relationships between conversations, experiments, and knowledge documents (the Timeline Graph, Phase 0) and surfaces them to researchers via verifiable URLs (Evidence Web, Phase 2). Both depend on the agent doing two small things consistently:

  1. Declaring edges when it derives, extends, reproduces, contradicts, or compares prior work.
  2. Attaching URLs to its answers when it references a specific experiment, doc, or thread.

Without these conventions, the graph stays empty and the Evidence Web has nothing to link to. CQ does not infer them. That's by design — explicit declarations carry the agent's intent; inferred edges would introduce noise and ambiguity.

Convention 1 — Edge Declaration ​

CQ records relationships in cq_graph_edges. Agents declare semantic edges via the cq_edge_declare MCP tool. The system will not infer them.

When to call cq_edge_declare ​

Call immediately after any of these actions:

  • Submit an experiment varying one parameter from a prior one → edge_type: derives_from
  • Submit an experiment that scales/extends a prior one → edge_type: extends
  • Reproduce a published or prior result → edge_type: reproduces
  • Record a finding that opposes a prior conclusion → edge_type: contradicts
  • Compare experiments side-by-side in a knowledge doc → edge_type: compared_to (one call per baseline)

Endpoint types (only these 4) ​

  • session — ai_sessions.session_id
  • job — hub_jobs.id
  • doc — c4_documents.doc_id (e.g. exp-abc12345)
  • persona — user UUID

Examples ​

derives_from:

cq_edge_declare(
  from_type="job",  from_id="<new>",
  to_type="job",    to_id="<base>",
  edge_type="derives_from",
  metadata={"changed":["lr"], "from":"1e-3", "to":"1e-4"}
)

compared_to (one call per baseline):

cq_edge_declare(from_type="doc", from_id="doc-cmp-001",
                to_type="job",   to_id="job-a",
                edge_type="compared_to")

contradicts:

cq_edge_declare(
  from_type="doc", from_id="doc-finding-002",
  to_type="doc",   to_id="doc-finding-001",
  edge_type="contradicts",
  metadata={"reason":"opposite val_loss trend"}
)

Rules ​

  1. One call per relationship — do not batch.
  2. metadata is optional, ≤ 2 KB serialized; use it for the reason.
  3. Edges are immutable. Wrong edge → delete + re-declare (do not retry).
  4. Do not declare structural edges (produced_in, authored_by). Hooks own those — the tool returns structural_edge_forbidden if you try.
  5. Edge declaration is best-effort. If it fails, continue your primary task.

Failure modes ​

ErrorMeaning
structural_edge_forbiddenYou tried to declare produced_in or authored_by.
invalid_edge_typeUnknown semantic edge; see hint for valid options.
invalid_from_type / invalid_to_typeUnknown endpoint kind.
missing_endpoint_idfrom_id or to_id was empty.
metadata_too_largeSerialized metadata > 2 KB.
not_authenticated / session_expiredUser must run cq auth login.
supabase_errorServer-side failure; inspect hint.

Convention 2 — Evidence URL Attachment ​

When your answer references a specific experiment, knowledge doc, or conversation thread, you must attach the canonical Evidence Web URL so the researcher can verify the evidence in one click.

When to call cq_evidence_url ​

Call it immediately before you finalize your answer, for each referenced resource:

  • Mentioning an experiment by id (exp-abc123, job-xyz) → type="exp"
  • Referencing a knowledge doc → type="doc"
  • Pointing at a past conversation → type="thread"
  • Comparing multiple experiments → type="compare" with ids=[...]

Returned URL format ​

exp     → {base}/exp/{id}
doc     → {base}/doc/{id}
thread  → {base}/thread/{id}
compare → {base}/compare?ids=a,b,c

Attach them as Markdown links in your answer:

"The previous run [exp-abc123]({url}) used lr=1e-4 ..."

Rules ​

  1. One URL per mentioned resource. Don't batch references into a single link — each experiment/doc/thread gets its own mint.
  2. compare accepts 2-10 ids. If you need to compare > 10, summarize first or split the comparison.
  3. URL minting is offline-safe (pure construction). Errors return a structured {error, message, hint} — surface the hint, do not retry.
  4. If the user is working without Evidence Web access (no browser, headless), still mint URLs — the researcher may paste them into their notes for later verification.
  5. share=true returns POST instructions for the user to mint a shareable token themselves (the MCP server doesn't carry the user's Supabase JWT, so the agent hands the user the recipe instead).

Example ​

User: "어제 돌린 MNIST 실험이랑 오늘 꺼 비교해줘"

Agent (internal):
  [queries hub_jobs, finds exp-aaa (yesterday) and exp-bbb (today)]
  [cq_evidence_url(type="compare", ids=["aaa","bbb"])
   → {url: "https://cq.pilab.kr/compare?ids=aaa,bbb"}]
  [cq_evidence_url(type="exp", id="aaa")  → exp URL]
  [cq_evidence_url(type="exp", id="bbb")  → exp URL]

Agent (output):
  "[exp-aaa](https://cq.pilab.kr/exp/aaa) got val_loss 0.42,
   [exp-bbb](https://cq.pilab.kr/exp/bbb) got 0.31 — -26% improvement.
   [side-by-side](https://cq.pilab.kr/compare?ids=aaa,bbb)"

How they compose ​

Edge declaration and URL attachment work together — the agent both records the relationship in the graph and surfaces it to the user:

1. cq_edge_declare(derives_from)  → Phase 0 graph
2. cq_evidence_url(exp)           → Phase 2 link
3. Answer with Markdown link      → Researcher verifies

Future sessions reading this exchange will see both:

  • The graph shows the lineage.
  • The evidence link shows the actual artifacts.

Forward-looking (Phase 3+) ​

The conventions list is intentionally short. Planned additions:

  • Reproduction reporting — when an agent re-runs reproduces and the result diverges, declare a follow-up contradicts edge.
  • Decision logging — durable record of "why we chose X over Y" with trade-off metadata.
  • Confidence calibration — agent self-reports confidence on inferences; future sessions weight by that signal.

These are speculative; current code only enforces the two above.

Appendix A — One-line summary (marketing) ​

CQ agents declare every research relationship in a queryable graph and attach a verification URL to every claim — so future sessions know why the decision was made and the researcher can check that it was right in one click.

Appendix B — Cheat-sheet ​

You did this                         → Call this
─────────────────────────────────────  ─────────────────────────────
Submitted a tweaked variant         → cq_edge_declare(derives_from)
Scaled it up                        → cq_edge_declare(extends)
Reran a paper's claim               → cq_edge_declare(reproduces)
Found an opposing result            → cq_edge_declare(contradicts)
Compared 2-10 runs in a doc         → cq_edge_declare(compared_to) × N
Mentioned exp-X in your answer      → cq_evidence_url(exp, X) + link
Mentioned doc-Y in your answer      → cq_evidence_url(doc, Y) + link
Mentioned a past thread             → cq_evidence_url(thread, Z) + link
Showed N runs in your answer        → cq_evidence_url(compare, ids=…)
