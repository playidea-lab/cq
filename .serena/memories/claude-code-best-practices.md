## 1. 핵심 설계 패턴

### 계층화된 위임 구조

```
사용자 요청
    ↓
Commands (/plan, /tdd, /code-review)  ← 진입점
    ↓
Agents (planner, architect, reviewer)  ← 전문가
    ↓
Skills (패턴, 표준, 워크플로우)         ← 지식
    ↓
Rules (보안, 스타일, 테스트)           ← 제약
```

**핵심 원칙**:
- 각 레이어는 명확한 책임을 가짐
- Commands는 사용자 인터페이스
- Agents는 특정 도메인의 전문가
- Skills는 재사용 가능한 지식/워크플로우
- Rules는 항상 적용되는 제약 조건

### 우선순위 체계

```
CRITICAL (차단) > HIGH (경고) > MEDIUM (제안)
```

- **CRITICAL**: 즉시 수정 필요, 커밋/배포 차단
- **HIGH**: 경고 표시, 검토 후 진행 가능
- **MEDIUM**: 제안 사항, 선택적 적용

## 2. Agents 패턴

### 에이전트 목록 및 역할

| 에이전트 | 역할 | 자동 활성화 조건 |
|---------|------|-----------------|
| **Planner** | 복잡한 작업 계획 수립 | 복잡한 기능 요청 시 |
| **Architect** | 시스템 설계, 기술 의사결정 | 새 기능/아키텍처 변경 시 |
| **Code-Reviewer** | 코드 품질/보안 검사 | 모든 코드 변경 시 |
| **TDD-Guide** | 테스트 주도 개발 가이드 | 구현 작업 시 |
| **Security-Reviewer** | 보안 취약점 검사 | 보안 관련 코드 변경 시 |

### 에이전트 설계 원칙

1. **명확한 책임 범위**: 각 에이전트는 하나의 도메인에 집중
2. **자동 활성화**: 작업 유형에 따라 자동으로 적절한 에이전트 선택
3. **병렬 작업 지원**: 독립적인 에이전트는 동시 실행 가능
4. **결과 통합**: 여러 에이전트의 결과를 종합하여 최종 판단

## 3. Commands 패턴

### /plan - 계획 수립

```
단계:
1. 요구사항 재정의 (명확화)
2. 위험/복잡도 식별
3. 단계별 계획 수립
4. 사용자 승인 요청

⚠️ 중요: 승인 전 코드 작성 금지
```

### /tdd - 테스트 주도 개발

```
사이클:
🔴 RED: 실패하는 테스트 작성
    ↓
🟢 GREEN: 테스트 통과하는 최소 구현
    ↓
🔵 REFACTOR: 코드 개선 (테스트 유지)
    ↓
반복
```

### /code-review - 코드 리뷰

```
심각도별 처리:
- CRITICAL/HIGH 있음 → 커밋 차단
- MEDIUM만 있음 → 경고 후 병합 가능
- 없음 → 자동 승인

검사 항목:
- 보안 취약점
- 코드 품질
- 테스트 커버리지
- 성능 문제
```

## 4. Rules 패턴

### 보안 체크리스트 (security.md)

```
필수 검사 항목:
□ 하드코딩된 시크릿 없음
□ SQL 인젝션 방지 (파라미터화 쿼리)
□ XSS 방지 (입력 이스케이프)
□ CSRF 토큰 사용
□ 인증/인가 검증
□ 민감 데이터 암호화
□ 에러 메시지에 내부 정보 노출 금지
□ 의존성 취약점 검사
```

### 코딩 스타일 (coding-style.md)

```
파일 크기: 200-400줄 권장
함수 크기: 50줄 미만
불변성: 가능한 const, readonly 사용
명명 규칙: 명확하고 일관된 네이밍
```

### 테스트 (testing.md)

```
커버리지: 80% 이상 필수
테스트 우선: 구현 전 테스트 작성
격리: 각 테스트는 독립적으로 실행
```

## 5. Hooks 자동화 패턴

### PreToolUse (실행 전)

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "개발 서버는 tmux 세션에서만 실행"
      },
      {
        "matcher": "git push",
        "command": "검토 단계 추가"
      }
    ]
  }
}
```

### PostToolUse (실행 후)

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*.ts",
        "command": "자동 타입 검사 (tsc --noEmit)"
      },
      {
        "matcher": "*.{ts,tsx,js,jsx}",
        "command": "자동 포맷팅 (prettier --write)"
      },
      {
        "matcher": "console.log",
        "command": "경고 표시"
      }
    ]
  }
}
```

### 활용 사례

- **코드 품질**: 수정 후 자동 린트/포맷팅
- **타입 안전성**: TypeScript 수정 후 자동 타입 검사
- **보안**: 민감 정보 커밋 전 검사
- **일관성**: PR 생성 시 자동 템플릿 적용

## 6. 컨텍스트 윈도우 관리

### MCP 서버 제한

```
권장 사항:
- 프로젝트당 MCP 서버 10개 이하
- 총 도구 80개 이하 유지

이유:
- 모든 MCP 활성화 시 200k → 70k 토큰으로 감소
- 컨텍스트 부족으로 성능 저하 발생
```

### 최적화 전략

1. **필수 MCP만 활성화**: 프로젝트에 실제 필요한 것만
2. **로컬 우선**: 가능하면 로컬 MCP 사용 (NPX)
3. **용도별 분리**: 개발/테스트/프로덕션 MCP 설정 분리

## 8. 핵심 설계 철학

1. **"계획 → 검토 → 구현 → 검증" 사이클 기계화**
   - 각 단계를 명확히 분리
   - 승인 없이 다음 단계 진행 금지

2. **자동 활성화로 수동 호출 최소화**
   - 사용자가 에이전트를 일일이 호출하지 않아도 됨
   - 작업 컨텍스트에 따라 적절한 에이전트 자동 선택

3. **명확한 우선순위 체계**
   - CRITICAL: 무조건 수정
   - HIGH: 강력 권장
   - MEDIUM: 선택적

4. **사용자 승인 원칙**
   - 계획 승인 전 코드 작성 금지
   - 중요한 결정은 항상 사용자에게 확인

5. **컨텍스트 윈도우 관리**
   - MCP 10개 이하
   - 도구 80개 이하
   - 필요한 것만 활성화
