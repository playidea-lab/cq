1. Relay 서버 코드 위치 및 JWT 검증 로직 ​

위치: infra/relay/main.go

검증 흐름 (infra/relay/main.go:166-189) ​

go
// validateToken verifies a Supabase Auth JWT by calling the Supabase Auth API.
// When supabaseURL is empty (dev mode), validation is skipped.
func (s *server) validateToken(tokenStr string) error {
    if s.supabaseURL == "" {
        return nil // dev mode
    }
    req, _ := http.NewRequest("GET", s.supabaseURL+"/auth/v1/user", nil)
    req.Header.Set("Authorization", "Bearer "+tokenStr)
    req.Header.Set("apikey", s.supabaseKey)
    resp, err := s.httpClient.Do(req)
    // ...
    if resp.StatusCode != http.StatusOK {
        return fmt.Errorf("invalid token (status %d)", resp.StatusCode)
    }
    return nil
}
  • Relay는 자체 JWT 파싱 없음. Supabase Auth /auth/v1/user API에 위임.
  • 환경변수: SUPABASE_URL, SUPABASE_ANON_KEY — fly.io secrets에 bake.
  • 401 발생 조건: validateToken 에러 → http.Error(w, "unauthorized: ...", 401).

두 엔드포인트에서 발생 가능 (main.go:243, main.go:337):

  1. GET /connect?token=... — 워커가 WebSocket 업그레이드 시
  2. POST /w/{id}/mcp — MCP 콜러가 relay 통과 시

서버 검증 기준 ​

Relay 서버는 Supabase /auth/v1/user에 토큰을 그대로 전달. Supabase는 다음을 검증:

  • 서명 유효성
  • 만료(exp claim) 미경과
  • Supabase 프로젝트 audience(aud) 일치

scope 개념 없음: Supabase JWT에 relay 전용 scope 없음. claims 불일치 가설(C) 해소 — 토큰이 유효하면 통과, 만료/무효면 401.

4. 4가지 가설 평가 ​

A: Relay 독립 인증 (API key 등) ​

해소됨. infra/relay/main.go의 유일한 인증 메커니즘은 Supabase Auth API 호출. API key별도 관리 없음.

B: Supabase-Relay sync 부재 (DB 라운드트립) ​

해소됨. Relay는 Supabase DB를 전혀 조회하지 않는다. /auth/v1/user REST 호출만. Sync 문제 없음.

C: 토큰 scope 불일치 ​

해소됨. Supabase JWT에 relay 전용 scope 없음. claims 비교 불필요.

D: client-side only (메모리 reload 문제) ​

확정 근본 원인 — 2가지 세부 패턴:

D-1: 워치독 토큰 stale 문제 (가장 유력)

serve_watchdog.go:75-84: 워치독 tokenFunc가 ~/.c4/session.json을 매번 읽지만 갱신하지 않음. 토큰 만료 시 갱신 없이 stale 토큰을 relay에 전달 → Supabase /auth/v1/user 401 → relay /connect 401.

메모리 relay_multiuser_fix.md에도 동일 패턴 기록: "watchdog tokenFunc: session.json에서 토큰 읽기 (빈 토큰 401 수정)".

D-2: cq serve stop && start 무효 원인

재기동 시 ~/.c4/session.json의 토큰을 읽어 TokenProvider를 초기화. 만약 session.json의 expires_at이 과거면 TokenProvider가 즉시 refresh를 시도하지만, 워치독은 TokenProvider 없이 직접 session.json 읽기만 한다. cq auth refresh로 session.json을 업데이트해도 이미 실행 중인 serve 프로세스의 워치독 goroutine에는 반영되지 않음 — 재기동 후에야 적용.

단, 재기동(stop && start) 후에도 401 지속되는 경우는:

  • session.json 자체가 만료된 상태
  • cq auth refresh 없이 재기동 → 같은 stale 토큰 재사용

5. 실제 원인 결정 + 증거 ​

결정: D-1 (워치독 stale 토큰) + D-2 (session.json 갱신 없는 재기동)

증거:

  1. serve_watchdog.go:75-84 — tokenFunc가 session.json을 읽기만 하고 refresh 호출 없음. TokenProvider.Token() (token.go:49) 처럼 5분 전 자동 갱신 없음.
  2. relay reconnectLoop (client.go:419-422) — 재연결 시마다 tokenFunc()를 호출하지만, 워치독의 tokenFunc는 session.json 파일 내용만 반환. 파일이 stale이면 재연결마다 401.
  3. 정상 경로 (mcp_init_relay.go:61) — ctx.Cloud.TP.Token은 TokenProvider 기반으로 5분 전 자동 갱신. 워치독은 이 TokenProvider 없음.
  4. cq serve stop && start 후 401 지속 = session.json이 만료 상태이고 cq auth refresh 없이 재기동.

6. T-RAR-1~4 scope 조정 권고 ​

태스크현재 예상 scope조정 권고
T-RAR-1Relay 서버 재시작/재배포불필요 — 서버 코드 정상. 클라이언트 수정 필요

T-RAR-3cq auth refresh → session.json 갱신 후 serve 재기동운영 가이드 — session.json stale 시 조치 순서 문서화
T-RAR-4watcher/reconnect 로직저우선 — reconnectLoop의 tokenFunc 주기 갱신은 T-RAR-2 수정 후 자동 해결

최우선 수정: serve_watchdog.go의 buildWatchdogRelayClient에 TokenProvider(또는 AuthClient 기반 refresher) 주입. 현재 static session.json 읽기 방식은 토큰 만료 시 자동 복구 불가.
