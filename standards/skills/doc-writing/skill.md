# Doc Writing

기술 문서, ADR, 스펙 작성 가이드.

## 트리거

"문서 작성", "doc writing", "ADR 작성", "스펙 작성", "기술 문서", "README 작성"

## Steps

### 1. 문서 유형 판단

| 유형 | 목적 | 독자 |
|------|------|------|
| **ADR** | 설계 결정 기록 | 미래의 개발자 |
| **스펙** | 기능 요구사항 | 구현자 |
| **README** | 프로젝트 소개/사용법 | 새 팀원, 외부 |
| **Runbook** | 운영 절차 | 온콜 엔지니어 |
| **API 문서** | 엔드포인트 사용법 | API 소비자 |
| **가이드** | How-to 튜토리얼 | 학습자 |

### 2. ADR (Architecture Decision Record)

```markdown
## ADR-XXX: [제목]

**상태**: 제안 / 승인 / 폐기
**일시**: YYYY-MM-DD

### 맥락
왜 이 결정이 필요한가? 어떤 문제를 해결하는가?

### 결정
무엇을 선택했는가?

### 근거
왜 이것을 선택했는가? 어떤 기준으로 비교했는가?

### 대안
고려했지만 선택하지 않은 것과 그 이유.

### 결과
이 결정으로 인해 변하는 것. 트레이드오프.
```

### 3. 기술 스펙

```markdown
## 기능: [제목]

### 목표
[1-2문장]

### 비범위
[하지 않는 것]

### 설계
[기술적 접근 방식, 다이어그램]

### API
[엔드포인트, 요청/응답 예시]

### 데이터 모델
[스키마 변경]

### 테스트 전략
[어떻게 검증할 것인가]

### 롤아웃
[배포 전략, feature flag]

### 리스크
[알려진 리스크와 대응]
```

### 4. README

```markdown
# 프로젝트명

[1-2문장 설명]

## 시작하기

### 사전 요구사항
[필요한 도구/환경]

### 설치
[단계별 설치 가이드]

### 실행
[실행 명령]

## 사용법
[핵심 사용법 + 예시]

## 아키텍처
[구조 개요, 선택적]

## 기여
[기여 가이드, 선택적]
```

### 5. 작성 원칙

- **독자를 정의**하고 그 수준에 맞춰 쓴다
- **"왜"를 설명**한다 ("무엇"은 코드가 말한다)
- **예시를 포함**한다 (추상적 설명 < 구체적 예시)
- **최신 상태를 유지**한다 (오래된 문서 > 없는 문서는 거짓)
- **짧게 쓴다** (긴 문서는 안 읽힌다)

## 안티패턴

- 코드를 그대로 설명하는 문서 (코드 읽으면 되는 내용)
- 이유 없는 결정 기록 ("X를 선택했다" — 왜?)
- 설치 가이드 없는 README
- 업데이트 안 되는 문서를 유지하는 것
