# Document Review

사내 문서 리뷰 가이드. 제안서, 보고서, 스펙 등 다양한 문서에 적용 가능한 체계적 검토 워크플로우.

## 트리거

"문서 리뷰", "document review", "보고서 검토", "제안서 검토"

## Steps

### 1. 문서 파싱 및 전체 파악

- `c4_parse_document`로 문서 파싱
- 문서 유형 확인 (제안서, 보고서, 스펙, 가이드 등)
- 목적과 대상 독자 파악
- 전체 분량과 목차 구조 확인

### 2. 구조 검증

- 목차가 논리적 흐름을 따르는가 (도입 → 본문 → 결론)
- 섹션 간 분량 균형이 적절한가
- 필수 섹션 누락 여부 (문서 유형별 체크리스트 적용)
- 목차와 본문 제목의 일치 여부

### 3. 내용 정확성 검토

- 인용된 수치/통계의 출처가 명시되어 있는가
- 기술적 서술이 사실에 부합하는가
- 날짜, 버전, 이름 등 고유명사의 정확성
- 약속/기한이 현실적인가 (일정, 리소스 대비)

### 4. 일관성 검토

- 용어 통일 (같은 개념을 다른 이름으로 부르지 않는가)
- 수치 일관성 (요약과 본문의 숫자가 일치하는가)
- 문체 통일 (경어체/평어체, 능동/수동)
- 시제 일관성 (과거/현재/미래 혼용 없는가)

### 5. 가독성 검토

- 한 문장이 50자를 넘으면 분리 고려
- 한 단락이 5줄을 넘으면 분리 고려
- 표/차트로 대체 가능한 텍스트 나열이 있는가
- 전문 용어에 설명이 있는가 (대상 독자 기준)
- 강조가 과하지 않은가 (볼드, 색상 남용)

### 6. 피드백 작성

**피드백 작성 원칙:**
- 문제 위치(페이지/섹션)를 구체적으로 명시
- "수정 필요" 대신 구체적 수정안 제시
- 심각도 분류: Critical(사실 오류) / Major(구조/논리) / Minor(표현/스타일)
- 잘된 부분도 언급 (균형 잡힌 피드백)

**피드백 형식:**
```
[심각도] 위치: 피드백 내용
  - 현재: "..."
  - 제안: "..."
  - 사유: ...
```

- `c4_knowledge_record`로 반복 피드백 패턴 기록

## 안티패턴

- 오타/맞춤법만 찾고 내용 검토를 건너뜀
- 전체를 다시 쓰라는 식의 피드백 ("다시 쓰세요")
- 구체적 수정안 없이 "어색합니다", "이상합니다"만 반복
- 본인 선호 문체를 강요 (문서 스타일 가이드 기준으로 판단)
- 마감 직전에 구조 변경을 요구
- 리뷰 없이 승인 (형식적 리뷰)
