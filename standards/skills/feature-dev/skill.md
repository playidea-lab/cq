# Feature Development

체계적인 기능 개발 워크플로우: 탐색 → 질문 → 설계 → 구현 → 리뷰.

## 트리거

"기능 개발", "feature dev", "새 기능", "기능 구현", "feature 구현"

## Steps

### 1. 코드베이스 탐색

구현 전 기존 코드를 이해한다:

- 관련 파일/모듈 구조 파악
- 유사 기능이 이미 있는지 검색
- 사용 중인 패턴/컨벤션 확인 (네이밍, 에러 처리, 테스트 방식)
- 의존성 그래프 파악 (이 모듈을 누가 호출하는가)

### 2. 요구사항 명확화

모호한 부분을 질문으로 해소:

- 엣지 케이스: "X가 없으면?", "동시에 Y가 일어나면?"
- 범위: "이것도 포함인가?", "이건 비범위인가?"
- 우선순위: "MVP에 필수인가, nice-to-have인가?"
- 비기능: "성능 요구사항?", "동시 사용자?"

### 3. 설계

코드 작성 전 구조를 결정:

- 변경할 파일 목록 + 각 파일의 변경 내용 요약
- 새로 만들 파일이 있다면 위치와 이유
- 데이터 흐름 (입력 → 처리 → 출력)
- API가 포함되면: 엔드포인트, 요청/응답 스키마
- DB가 포함되면: 스키마 변경, 마이그레이션

### 4. 테스트 먼저

TDD 또는 test-first:

- 핵심 시나리오의 테스트를 먼저 작성 (실패하는 상태)
- 에러 경로 테스트도 포함
- 테스트가 명세 역할을 함 — "이 테스트가 통과하면 기능 완성"

### 5. 구현

최소한으로 구현:

- 테스트를 통과시키는 최소한의 코드
- 기존 패턴과 컨벤션을 따른다
- 한 번에 하나의 관심사만 변경
- 중간 중간 테스트 실행 (깨지면 즉시 수정)

### 6. 자기 리뷰

PR 전에 스스로 점검:

- [ ] 불필요한 변경이 섞이지 않았는가?
- [ ] 디버깅 코드(print, console.log)가 남아있지 않은가?
- [ ] 에러 메시지가 사용자 친화적인가?
- [ ] 하드코딩된 값이 없는가?
- [ ] 테스트가 충분한가? (happy + error path)

### 7. PR 작성

```markdown
## 무엇을
[기능 설명]

## 왜
[비즈니스 맥락 / 문제]

## 어떻게
[기술적 접근 방식]

## 테스트
[테스트 방법 / 확인 사항]

## 스크린샷
[UI 변경 시]
```

## 안티패턴

- 코드 읽기 전에 구현 시작
- 질문 없이 가정으로 진행
- 기능 + 리팩토링을 한 PR에
- "나중에 테스트 추가"
