# Performance Audit

성능 감사 및 최적화 워크플로우.

## 트리거

"성능 감사", "perf audit", "느려졌다", "성능 최적화", "병목", "latency"

## Steps

### 1. 문제 정의

최적화 전 반드시 측정:

- 현재 성능: P50, P95, P99 응답 시간
- 목표 성능: 구체적 수치 (예: P95 < 200ms)
- 언제부터 느려졌는가? (배포, 트래픽 증가, 데이터 증가)
- 어디가 느린가? (특정 엔드포인트, 페이지, 쿼리)

### 2. 프로파일링

**백엔드:**
- Go: `pprof` (CPU, memory, goroutine)
- Python: `py-spy`, `cProfile`, `line_profiler`
- DB: `EXPLAIN ANALYZE`, slow query log

**프론트엔드:**
- Chrome DevTools Performance 탭
- Lighthouse 점수 (LCP, FID, CLS)
- Bundle analyzer (webpack-bundle-analyzer)

**인프라:**
- CPU/메모리/디스크 I/O 사용률
- 네트워크 지연 (서비스 간)
- DB 커넥션 풀 사용률

### 3. 병목 식별

흔한 병목 패턴:

| 패턴 | 증상 | 해결 |
|------|------|------|
| N+1 쿼리 | 목록 조회 시 느림 | JOIN 또는 batch loading |
| 인덱스 누락 | 특정 쿼리 느림 | EXPLAIN → 인덱스 추가 |
| 직렬 I/O | 독립 API 호출이 순차 실행 | 병렬화 (goroutine, asyncio) |
| 큰 페이로드 | 응답 크기 과대 | 페이지네이션, 필드 필터링 |
| 메모리 누수 | 시간이 지나면 느려짐 | 프로파일러로 leak 추적 |
| Lock contention | 동시 요청 시 느림 | Lock 범위 축소, lock-free |

### 4. 최적화 적용

- 한 번에 하나만 변경 (복합 변경 시 효과 측정 불가)
- 변경 전후 벤치마크 비교
- 캐시 추가 시: 만료 정책, 무효화 전략 함께 설계
- 쿼리 최적화 시: EXPLAIN 결과 기록

### 5. 검증

- 동일 조건에서 벤치마크 재실행
- 개선 폭 수치로 기록 (P95: 800ms → 150ms, -81%)
- 부작용 확인 (메모리 증가, 정합성, 기존 테스트)
- 부하 테스트: 예상 트래픽 2배에서도 목표 충족 확인

### 6. 기록

```markdown
## 성능 개선: [대상]
- **문제**: [증상 + 수치]
- **원인**: [병목 지점]
- **해결**: [적용한 최적화]
- **결과**: [개선 수치]
- **트레이드오프**: [있다면]
```

## 안티패턴

- 측정 없이 "느린 것 같아서" 최적화
- 프로파일링 없이 캐시부터 추가
- 마이크로 최적화 (전체 지연의 1% 미만인 부분 최적화)
- 최적화 후 벤치마크 안 하고 "빨라진 것 같다"
