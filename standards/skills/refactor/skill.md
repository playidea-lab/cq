# Refactor

안전한 리팩토링 워크플로우.

## 트리거

"리팩토링", "refactor", "코드 정리", "기술 부채", "구조 개선", "tech debt"

## Steps

### 1. 범위 설정

리팩토링 전 반드시 정의:

- **목표**: 무엇을 개선하는가? (가독성 / 성능 / 확장성 / 중복 제거)
- **범위**: 어떤 파일/모듈을 건드리는가?
- **비범위**: 건드리지 않는 것은?
- **완료 조건**: 언제 끝난 건가?

리팩토링과 기능 변경을 **절대 같은 PR에 섞지 않는다**.

### 2. 테스트 확보

리팩토링 전에 기존 동작을 검증하는 테스트가 있어야 한다.

- 테스트가 있으면: 전부 통과 확인
- 테스트가 없으면: **먼저 테스트를 추가** (별도 PR)
- 테스트 추가 PR → merge → 리팩토링 PR (2단계)

### 3. 단계적 변경

한 번에 다 바꾸지 않는다. 각 단계가 독립적으로 커밋/리뷰 가능해야 한다.

**패턴 예시:**
- 함수 추출: 새 함수 생성 → 기존 코드에서 호출 → 기존 인라인 코드 삭제
- 모듈 분리: 새 모듈 생성 → import 변경 → 기존 위치에서 삭제
- 인터페이스 변경: 새 인터페이스 추가 → 구현 마이그레이션 → 이전 인터페이스 삭제

각 단계 사이에 **테스트 통과 확인**.

### 4. 회귀 검증

- 전체 테스트 스위트 실행
- 리팩토링 전후 동작이 동일한지 확인
- 성능 리팩토링이면: 벤치마크 전후 비교
- API 변경이면: 하위 호환성 확인

### 5. 리뷰

리팩토링 PR 설명에 포함:

```markdown
## Refactoring: [제목]
- **목표**: [왜 리팩토링하는가]
- **변경 요약**: [구조적으로 무엇이 바뀌는가]
- **동작 변경**: 없음 (리팩토링만)
- **테스트**: [기존 테스트 통과 / 추가된 테스트]
```

## 리팩토링 vs 재작성

| | 리팩토링 | 재작성 |
|--|---------|--------|
| 범위 | 점진적, 부분적 | 전체 교체 |
| 리스크 | 낮음 | 높음 |
| 테스트 | 기존 테스트 유지 | 새 테스트 필요 |
| 권장 | 대부분의 경우 | 기존 코드가 완전히 이해 불가능할 때만 |

## 안티패턴

- "리팩토링"이라며 기능을 변경하는 것
- 테스트 없이 리팩토링
- 한 PR에 리팩토링 + 기능 추가
- "나중에 정리하겠다"며 TODO 남기고 넘어가기
