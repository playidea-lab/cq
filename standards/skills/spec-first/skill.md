# Spec-First Development

코드 전에 스펙. 스펙이 SSOT.

## 트리거

"스펙 먼저", "spec first", "spec driven", "명세 작성", "요구사항 정의", "설계 문서"

## 철학

> "코드를 작성하기 전에 무엇을 만들지 합의한다."
> "스펙이 변하면 코드가 따라간다. 코드가 변하면 스펙을 업데이트한다."

Spec-driven development는 3가지 문서를 먼저 만든다:
1. **요구사항** (What) — 무엇을 만드는가?
2. **설계** (How) — 어떻게 만드는가?
3. **태스크** (Who/When) — 누가 언제 만드는가?

이 3개가 합의되면 구현은 기계적이 된다.

## Steps

### 1. 요구사항 정의 (EARS 패턴)

```markdown
## 요구사항: [기능명]

### Ubiquitous (항상)
- 시스템은 ~해야 한다

### Event-Driven (이벤트 발생 시)
- WHEN 사용자가 ~하면, THEN 시스템은 ~한다

### State-Driven (상태 조건)
- WHILE ~상태일 때, 시스템은 ~해야 한다

### Optional (선택적)
- IF ~기능이 활성화되면, 시스템은 ~한다

### Unwanted (에러/예외)
- IF ~에러가 발생하면, 시스템은 ~한다

### 비범위
- ~는 이번 범위에 포함하지 않는다

### 성공 지표
- [측정 가능한 기준]
```

**EARS가 좋은 이유**: 모호함을 제거한다. "잘 동작해야 한다" → "WHEN 사용자가 로그인하면, THEN 3초 이내에 대시보드가 표시된다."

### 2. 설계 문서

```markdown
## 설계: [기능명]

### 아키텍처 옵션
| 옵션 | 복잡도 | 장점 | 단점 |
|------|--------|------|------|
| A: ... | 낮음 | ... | ... |
| B: ... | 중간 | ... | ... |

### 선택: [옵션 X]
이유: ...

### 컴포넌트 설계
- [컴포넌트 1]: 역할, 인터페이스, 의존성
- [컴포넌트 2]: ...

### 데이터 흐름
[입력] → [처리 1] → [처리 2] → [출력]

### API (해당 시)
- `POST /api/xxx` — 요청/응답 스키마

### 설계 결정 (ADR)
- DEC-001: [질문] → [결정] (이유: ...)
```

### 3. 태스크 분해

```markdown
## 태스크

### T-001: [제목]
- scope: [영향 파일]
- dod: [완료 조건 — 검증 가능]
- deps: []

### T-002: [제목]
- scope: [영향 파일]
- dod: [완료 조건]
- deps: [T-001]
```

**DoD 규칙:**
- "구현한다" ✗ → "테스트가 통과한다" ✓
- "개선한다" ✗ → "P95 < 200ms" ✓
- 모든 DoD는 기계적으로 검증 가능해야

### 4. 리뷰 → 합의

3개 문서를 리뷰:
- 요구사항: 빠진 케이스, 모호한 표현 없는지
- 설계: 대안 검토, 트레이드오프 인지
- 태스크: 의존성 맞는지, 크기 적절한지

**합의 후 구현 시작.** 합의 전 코드 작성 금지.

### 5. 구현 (스펙 기반)

- 각 태스크는 DoD를 테스트로 먼저 작성 (TDD)
- 구현 중 스펙 변경 필요 시 → 스펙 업데이트 먼저, 코드는 그 다음
- 스펙에 없는 것을 구현하지 않음 (scope creep 방지)

### 6. 검증

- [ ] 모든 EARS 요구사항이 테스트로 커버되는가?
- [ ] 설계 결정이 코드에 반영되었는가?
- [ ] 비범위 항목을 구현하지 않았는가?
- [ ] 성공 지표를 달성했는가?

## Spec vs 코드 동기화

```
스펙 변경 → 태스크 DoD 갱신 → 코드 수정 → 테스트 통과
코드에서 발견 → 스펙 업데이트 → 팀 합의 → 계속
```

**단방향 원칙**: 스펙이 코드를 이끈다. 코드가 스펙을 바꾸지 않는다 (발견 시 스펙 업데이트 프로세스를 거침).

## 안티패턴

- "코드 먼저 짜고 스펙은 나중에" (스펙 = 사후 문서화 ≠ spec-first)
- 스펙 없이 "빨리 만들어" (단기 빠르지만 장기 느림)
- 스펙을 너무 상세하게 (구현 수준 상세 = 코드를 두 번 쓰는 것)
- 스펙과 코드 동기화 안 함 (오래된 스펙 = 거짓말)
- 스펙 합의 전 구현 시작 (rework 확률 높음)
