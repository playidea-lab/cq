# Migration

DB 스키마, API 버전, 데이터 마이그레이션 가이드.

## 트리거

"마이그레이션", "migration", "스키마 변경", "DB 변경", "API 버전업", "데이터 이전"

## Steps

### 1. 영향 분석

마이그레이션 전 반드시 확인:

- 변경 대상: 테이블/컬럼/인덱스/API 엔드포인트
- 영향 받는 서비스 목록
- 데이터 규모 (레코드 수, 테이블 크기)
- 예상 다운타임 (zero-downtime 가능 여부)
- 롤백 가능 여부

### 2. Backward Compatibility 체크

**DB 마이그레이션:**
- 컬럼 추가: OK (기존 코드 영향 없음)
- 컬럼 삭제: 2단계 — 코드에서 참조 제거 → 다음 릴리스에서 컬럼 삭제
- 컬럼 이름 변경: 2단계 — 새 컬럼 추가 + 데이터 복사 → 이전 컬럼 삭제
- 타입 변경: 새 컬럼으로 처리 (직접 ALTER 금지)

**API 마이그레이션:**
- 새 필드 추가: OK
- 필드 삭제/이름 변경: 버전업 (v1 → v2), 이전 버전 deprecated 기간 설정
- 응답 구조 변경: 새 버전 필수

### 3. 마이그레이션 스크립트

```
migrations/
  YYYYMMDD_HHMMSS_<설명>.up.sql    -- 적용
  YYYYMMDD_HHMMSS_<설명>.down.sql  -- 롤백
```

- 모든 마이그레이션은 idempotent (IF NOT EXISTS, IF EXISTS)
- up과 down은 반드시 쌍으로 존재
- 대용량 테이블: 배치 처리 (1000~10000건 단위)
- 인덱스 생성: `CONCURRENTLY` (PostgreSQL) 또는 온라인 DDL

### 4. 테스트

- 스테이징에서 실제 데이터 규모로 테스트
- 마이그레이션 시간 측정 + 기록
- 롤백 테스트 (down 스크립트 실행 후 서비스 정상 동작 확인)
- 마이그레이션 중 서비스 동작 확인 (zero-downtime인 경우)

### 5. 실행

- 실행 전: DB 백업 확인
- 실행 중: 모니터링 (쿼리 지연, 락 대기, 에러율)
- 실행 후: 데이터 무결성 확인 (카운트, 샘플 검증)
- 문제 발생: 즉시 down 스크립트로 롤백

### 6. 정리

- 이전 버전 코드/스키마 제거 (deprecated 기간 종료 후)
- 마이그레이션 기록 업데이트
- 관련 문서 갱신

## 안티패턴

- 프로덕션에서 직접 ALTER TABLE 실행
- 롤백 스크립트 없는 마이그레이션
- 대용량 테이블을 한 번에 UPDATE
- 마이그레이션 중 스키마와 비즈니스 로직을 동시에 변경
