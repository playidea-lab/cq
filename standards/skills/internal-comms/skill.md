# Internal Comms

사내 커뮤니케이션 작성 가이드 (상태 리포트, 공지, FAQ 등).

## 트리거

"사내 공지", "internal comms", "상태 리포트", "주간 보고", "팀 공지", "FAQ 작성"

## Steps

### 1. 문서 유형 판단

| 유형 | 목적 | 독자 | 빈도 |
|------|------|------|------|
| **주간 리포트** | 진행 상황 공유 | 팀/리더십 | 매주 |
| **프로젝트 업데이트** | 마일스톤, 리스크 공유 | 스테이크홀더 | 격주/월 |
| **장애 공지** | 장애 상황 + 복구 알림 | 전사 | 발생 시 |
| **기술 공지** | 인프라/도구 변경 알림 | 개발팀 | 수시 |
| **의사결정 공지** | 주요 결정 + 근거 | 관련자 | 수시 |
| **FAQ** | 반복 질문 정리 | 팀/전사 | 수시 갱신 |

### 2. 주간 리포트

```markdown
# 주간 리포트: [팀명] (MM/DD ~ MM/DD)

## 이번 주 완료
- [완료 항목 1]
- [완료 항목 2]

## 다음 주 계획
- [계획 1]
- [계획 2]

## 블로커 / 리스크
- [있다면 기술, 없으면 "없음"]

## 수치
- 배포: N회
- 버그 수정: N건
- PR 평균 리뷰 시간: N시간
```

### 3. 프로젝트 업데이트

```markdown
# [프로젝트] 상태 업데이트 (MM/DD)

## 요약
[1-2문장: 전체 상태 + 핵심 변화]

## 진행률
- 전체: N% (이전: N%)
- 마일스톤 1: ✅ 완료
- 마일스톤 2: 🔄 진행 중 (예정: MM/DD)
- 마일스톤 3: ⏳ 대기

## 리스크
| 리스크 | 영향 | 대응 |
|--------|------|------|
| [리스크 1] | [영향] | [대응 계획] |

## 다음 단계
- [구체적 액션 + 담당자 + 기한]
```

### 4. 장애 공지

```markdown
# 🔴 [서비스명] 장애 발생 (MM/DD HH:MM)

**상태**: 조사 중 / 복구 중 / 복구 완료
**영향**: [어떤 기능이 어떻게 영향받는지]
**시작 시각**: HH:MM KST

## 현재 상황
[무슨 일이 일어나고 있는지]

## 사용자 영향
[사용자가 겪는 증상]

## 조치
[현재 진행 중인 조치]

## 다음 업데이트
[예상 시각]
```

복구 후:

```markdown
# ✅ [서비스명] 장애 복구 완료 (MM/DD HH:MM)

**지속 시간**: N분
**근본 원인**: [간단 설명]
**영향**: [사용자/데이터 영향]
**포스트모템**: [링크, 48시간 내 공유]
```

### 5. 작성 원칙

- **결론 먼저**: 가장 중요한 정보를 첫 줄에
- **구체적으로**: "잘 진행 중" → "4/7 마일스톤 완료, 예정대로"
- **행동 가능하게**: "문제 있음" → "X가 블로커, Y팀 지원 필요"
- **짧게**: 5분 안에 읽을 수 있는 분량
- **정기적으로**: 같은 시간, 같은 형식으로

## 안티패턴

- 나쁜 소식 숨기기 (리스크는 빨리 공유할수록 좋다)
- 기술 용어 남발 (독자 수준에 맞추기)
- 업데이트 없이 방치 ("소식 없음 = 잘 되고 있음"이 아님)
- 형식 없는 자유 텍스트 (일관된 형식 = 빠른 파악)
