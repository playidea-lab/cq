# Estimation

작업량 추정 가이드: 분해 → 복잡도 → 리스크 → 버퍼.

## 트리거

"작업량 추정", "estimation", "얼마나 걸릴까", "일정 산정", "공수 산정"

## Steps

### 1. 작업 분해

큰 작업을 추정 가능한 단위로 분해:

- 1개 작업 = 1일 이내 완료 가능한 단위
- 분해할 수 없으면 아직 충분히 이해하지 못한 것
- "기타" 항목 금지 — 모든 항목이 구체적이어야

### 2. 복잡도 평가

각 항목에 복잡도 부여:

| 복잡도 | 기준 | 시간 |
|--------|------|------|
| S | 명확한 변경, 검색하면 바로 나옴 | 1-2시간 |
| M | 설계 필요, 여러 파일 변경 | 반나절-1일 |
| L | 조사 필요, 의존성 변경, 테스트 대량 | 1-3일 |
| XL | 아키텍처 변경, 마이그레이션 | 3일+ (더 분해) |

XL이 있으면 분해가 부족하다는 신호.

### 3. 불확실성 식별

각 항목의 불확실성을 표시:

- **확실**: 해본 적 있는 작업, 명확한 경로
- **보통**: 방법은 알지만 세부 사항 미확인
- **높음**: 조사 필요, 외부 의존성, 처음 하는 것

### 4. 추정 공식

```
총 추정 = Σ(각 항목 시간) × 불확실성 계수

불확실성 계수:
- 확실: ×1.2
- 보통: ×1.5
- 높음: ×2.0 (또는 "스파이크 먼저" 권고)
```

### 5. 추정 테이블

```markdown
| 항목 | 복잡도 | 불확실성 | 추정 | 비고 |
|------|--------|---------|------|------|
| API 엔드포인트 | M | 확실 | 1일 | 기존 패턴 |
| DB 마이그레이션 | M | 보통 | 1.5일 | 대용량 테이블 |
| 프론트 UI | L | 높음 | 4일 | 새 컴포넌트 |
| 테스트 | M | 확실 | 1일 | |
| 버퍼 (20%) | | | 1.5일 | |
| **총합** | | | **9일** | |
```

### 6. 주의사항

- 추정은 약속이 아니라 현재 이해도 기반 예측
- 불확실성이 높으면 스파이크(조사)를 먼저 수행하고 재추정
- 과거 유사 작업의 실제 소요 시간과 비교
- 추정 정확도는 시간이 지나면 개선됨 — 추정 vs 실제 기록

## 안티패턴

- 분해 없이 "2주" 같은 직감 추정
- 버퍼 없는 추정 (항상 무언가 예상 밖의 일이 생긴다)
- "최선의 경우"로 추정 (평균 또는 80% 확률 기준)
- 추정을 공약으로 취급 (조건이 바뀌면 재추정)
