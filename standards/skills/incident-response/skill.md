# Incident Response

프로덕션 장애 대응 워크플로우.

## 트리거

"장애 대응", "incident", "서비스 장애", "SEV1", "서버 다운", "장애 발생"

## Steps

### 1. 심각도 판단

현재 상황을 파악하고 심각도를 분류합니다:

| 등급 | 기준 | 대응 |
|------|------|------|
| SEV1 | 서비스 전체 불가, 데이터 손실 위험 | 즉시 대응, 전원 소집 |
| SEV2 | 주요 기능 장애, 우회 불가 | 30분 내 대응 |
| SEV3 | 부분 장애, 우회 가능 | 업무 시간 내 대응 |

### 2. 상황 파악

- 언제부터 발생했는가? (알림 시각, 사용자 리포트)
- 영향 범위는? (전체 / 특정 사용자 / 특정 기능)
- 최근 변경사항은? (`git log`, 배포 이력, 설정 변경)
- 관련 로그/메트릭 확인 (에러율, 지연, CPU/메모리)

### 3. 즉시 조치

우선순위: **복구 > 원인 분석**

- 최근 배포가 원인이면 → 즉시 롤백
- 리소스 고갈이면 → 스케일 업/아웃
- 외부 의존성 장애면 → circuit breaker / fallback 활성화
- DB 문제면 → 쿼리 킬, 커넥션 풀 확인, 복제 지연 확인

### 4. 원인 분석

복구 후 근본 원인 분석:

- 타임라인 재구성 (분 단위)
- 로그/트레이스에서 최초 에러 지점 식별
- 코드 변경, 설정 변경, 인프라 변경 중 어디서 시작됐는지
- "왜?"를 5번 반복 (5 Whys)

### 5. 핫픽스

- 최소한의 변경으로 수정 (리팩토링 금지)
- 테스트 작성 → 통과 확인
- 핫픽스 PR → 리뷰 → 배포 (긴급 시 리뷰 후 배포 가능)

### 6. 포스트모템

48시간 내 작성. 필수 항목:

```markdown
## 포스트모템: [제목]
- **일시**: YYYY-MM-DD HH:MM ~ HH:MM (KST)
- **심각도**: SEV1/2/3
- **영향**: [사용자 수, 기능, 기간]
- **타임라인**: [분 단위 이벤트 목록]
- **근본 원인**: [기술적 원인]
- **해결**: [어떻게 복구했는지]
- **재발 방지**:
  - [ ] 액션 1 (담당자, 기한)
  - [ ] 액션 2 (담당자, 기한)
- **교훈**: [이번 장애에서 배운 것]
```

비난 금지 (blameless). 시스템과 프로세스에 집중.

## 안티패턴

- 원인 분석 중 복구를 미루는 것
- "다시는 이런 일이 없도록" 같은 비구체적 액션
- 포스트모템 없이 "해결됨"으로 종료
