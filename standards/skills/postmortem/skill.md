# Postmortem

장애/사고 후 포스트모템 작성 가이드.

## 트리거

"포스트모템", "postmortem", "장애 분석", "사후 분석", "RCA", "근본 원인 분석"

## Steps

### 1. 타임라인 재구성

분 단위로 사건을 정리:

```markdown
| 시각 (KST) | 이벤트 |
|------------|--------|
| 14:03 | 모니터링 알림 발생 (에러율 5% 초과) |
| 14:05 | 온콜 엔지니어 확인 시작 |
| 14:08 | 원인 추정: 14:00 배포의 DB 마이그레이션 |
| 14:12 | 롤백 시작 |
| 14:15 | 롤백 완료, 서비스 정상화 |
| 14:30 | 모니터링 정상 확인 |
```

소스: 알림 로그, 채팅 이력, 배포 이력, git log, 메트릭

### 2. 영향 분석

- 영향 시간: 시작 ~ 복구 (총 N분)
- 영향 사용자 수 / 요청 수
- 데이터 손실 여부
- SLA 위반 여부
- 비즈니스 영향 (매출, 평판)

### 3. 근본 원인 분석 (5 Whys)

```
Q: 왜 서비스가 다운되었는가?
A: DB 마이그레이션이 테이블 락을 잡았다.

Q: 왜 테이블 락이 발생했는가?
A: ALTER TABLE이 LOCK=EXCLUSIVE로 실행되었다.

Q: 왜 LOCK=EXCLUSIVE를 사용했는가?
A: 마이그레이션 스크립트에 락 모드를 명시하지 않았다.

Q: 왜 명시하지 않았는가?
A: 마이그레이션 리뷰 체크리스트에 락 영향 항목이 없었다.

Q: 왜 체크리스트에 없었는가?
A: 이전에 이 문제를 경험한 적이 없어서 추가되지 않았다.

→ 근본 원인: 마이그레이션 리뷰 프로세스에 락 영향 검증 단계 부재
```

### 4. 재발 방지 액션

각 액션에 반드시 **담당자 + 기한**:

```markdown
### 재발 방지
- [ ] 마이그레이션 체크리스트에 "락 영향 분석" 항목 추가 (@홍길동, 03/25)
- [ ] 스테이징에서 마이그레이션 실행 시간 측정 자동화 (@김철수, 03/28)
- [ ] 배포 후 5분 자동 모니터링 + 이상 시 자동 롤백 (@이영희, 04/01)
```

**좋은 액션 vs 나쁜 액션:**

| 나쁨 | 좋음 |
|------|------|
| "더 주의한다" | "체크리스트에 항목 추가" |
| "코드 리뷰를 잘 한다" | "CI에 자동 검증 추가" |
| "다시는 이런 일이 없도록" | "알림 임계값을 X로 조정" |

### 5. 교훈

- 이번 장애에서 잘 된 점 (빠른 대응, 효과적 모니터링 등)
- 개선할 점 (감지 지연, 커뮤니케이션 등)
- 시스템/프로세스 수준의 교훈

### 6. 리뷰

- 포스트모템은 관련자 전원이 리뷰
- **비난 금지 (blameless)** — 사람이 아닌 시스템에 집중
- 액션 아이템 팔로업: 다음 스프린트에 포함

## 템플릿

```markdown
# 포스트모템: [장애 제목]

**일시**: YYYY-MM-DD HH:MM ~ HH:MM (KST)
**심각도**: SEV1 / SEV2 / SEV3
**작성자**: @이름
**리뷰어**: @이름, @이름

## 요약
[1-2문장 요약]

## 영향
- 시간: N분
- 사용자: ~N명
- 데이터 손실: 없음 / 있음 (상세)

## 타임라인
| 시각 | 이벤트 |
|------|--------|

## 근본 원인
[5 Whys 분석 결과]

## 해결
[어떻게 복구했는가]

## 재발 방지
- [ ] 액션 (담당자, 기한)

## 교훈
- 잘 된 점:
- 개선할 점:
```

## 안티패턴

- "누가 실수했는가"에 집중 (blame culture)
- 구체적 수치 없는 영향 분석
- 액션 없는 포스트모템
- 작성 후 팔로업 없이 방치
