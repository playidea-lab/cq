# PM / 기획 팀 규칙

> 요구사항 정의, 스펙 작성, 프로젝트 관리 기준.

## 요구사항 문서

- 모든 기능은 PRD(Product Requirements Document)로 시작한다.
- PRD 필수 항목: 목표(1문장), 비범위, 사용자 시나리오, 성공 지표, 리스크.
- EARS 패턴 사용: Ubiquitous / Event-Driven / State-Driven / Optional / Unwanted.
- "모든", "항상", "절대"는 검증 불가 — 구체적 조건으로 대체.

## 스펙 작성

- 기능 스펙은 구현자가 질문 없이 시작할 수 있을 정도로 구체적이어야 한다.
- 엣지 케이스와 에러 시나리오를 명시한다 (happy path만 쓰지 않는다).
- UI가 포함되면 와이어프레임 또는 목업 첨부. 텍스트 설명만으로는 불충분.
- API가 포함되면 요청/응답 예시를 포함한다.

## DoD (Definition of Done)

- 모든 태스크에 DoD가 있어야 한다. DoD 없는 작업은 '작업'이 아니다.
- DoD는 체크박스 형태, 검증 가능한 문장으로 작성.
- "잘 동작함", "깔끔함" 같은 주관적 기준 금지 — 테스트 통과, 리뷰 승인 등 객관적 기준.

## 우선순위 관리

- MoSCoW (Must/Should/Could/Won't) 또는 RICE 프레임워크 사용.
- 우선순위 변경 시 이유와 영향 범위를 기록한다.
- 기한이 있는 항목은 절대 날짜로 기록 (상대 날짜 금지: "다음 주" → "2026-03-28").

## 커뮤니케이션

- 결정사항은 문서로 남긴다. 구두 합의 후 반드시 기록.
- 스펙 변경은 변경 이유 + 영향 범위 + 관련자 태그.
- 회의록 필수 항목: 참석자, 결정사항, 액션 아이템(담당자 + 기한).

## 금지 사항

- 구현 중 스펙 변경 없이 "원래 이런 의도였다" 주장
- 기한 없는 "ASAP" 우선순위
- 성공 지표 없는 기능 요청
