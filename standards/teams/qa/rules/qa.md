# QA / 테스트 팀 규칙

> 품질 보증, 테스트 전략, 버그 관리 기준.

## 테스트 전략

- 모든 프로젝트에 테스트 피라미드 적용: Unit(70%) > Integration(20%) > E2E(10%).
- 새 기능에는 최소 happy path + 1개 이상 에러 경로 테스트.
- 버그 수정에는 재현 테스트가 선행되어야 한다 (테스트 → 실패 → 수정 → 통과).

## 테스트 작성 기준

- 테스트 이름: `Test_<시나리오>_<기대결과>` (한글 가능).
- 하나의 테스트에 하나의 검증 (assert). 여러 관심사 혼합 금지.
- 테스트 간 순서 의존성 금지. 공유 상태 금지.
- 외부 의존성(DB, API)은 테스트 시작 시 셋업, 종료 시 정리.
- sleep 기반 동기화 금지 — 폴링, 채널, 이벤트 대기 사용.

## 버그 관리

- 버그 리포트 필수 항목: 재현 순서, 기대 결과, 실제 결과, 환경(OS/브라우저/버전).
- 심각도 분류:
  | 심각도 | 정의 | 대응 |
  |--------|------|------|
  | Critical | 서비스 불가, 데이터 손실 | 즉시 핫픽스 |
  | High | 주요 기능 장애, 우회 불가 | 다음 릴리스 전 수정 |
  | Medium | 부분 장애, 우회 가능 | 백로그 우선 |
  | Low | 미관, 오타, 불편함 | 백로그 |
- 중복 버그는 원본에 링크하고 닫는다. 매주 stale 버그 정리.

## 회귀 테스트

- 릴리스 전 회귀 테스트 스위트 실행 필수.
- CI에서 자동 실행되는 테스트만 회귀 테스트로 인정.
- 수동 테스트가 필요하면 체크리스트로 문서화하고 실행 결과 기록.

## 성능 테스트

- API 응답 시간 기준: P95 < 500ms, P99 < 2s (프로젝트별 조정 가능).
- 부하 테스트: 예상 트래픽 2배로 실행, 결과 기록.
- 성능 회귀: 이전 릴리스 대비 10% 이상 저하 시 블로커.

## 금지 사항

- "잘 돌아가는 것 같다"로 테스트 완료 선언
- 실패하는 테스트를 skip/disable로 우회
- 프로덕션 데이터를 테스트에 사용 (PII 위험)
