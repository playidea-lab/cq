# 운영 / SRE 팀 규칙

> 서비스 운영, 장애 대응, 모니터링, 안정성 기준.

## 서비스 안정성

- SLO(Service Level Objective) 정의 필수: 가용성, 지연, 에러율.
- Error Budget 기반 운영: 예산 소진 시 기능 배포 동결, 안정화 우선.
- 모든 서비스에 health check 엔드포인트 (`/healthz`, `/readyz`).
- Graceful shutdown 필수: 진행 중 요청 완료 후 종료.

## 장애 대응 (Incident Response)

- 장애 감지 → 5분 내 Incident Commander 지정.
- 심각도 기준:
  | 등급 | 정의 | 대응 시간 |
  |------|------|----------|
  | SEV1 | 서비스 전체 불가 | 즉시 (24/7) |
  | SEV2 | 주요 기능 장애 | 30분 내 |
  | SEV3 | 부분 장애, 우회 가능 | 업무 시간 내 |
  | SEV4 | 경미, 모니터링 | 다음 스프린트 |
- 장애 중 커뮤니케이션: 상태 페이지 업데이트, 10분 간격 상황 공유.
- 장애 후 48시간 내 포스트모템 작성 필수.

## 모니터링

- 4대 골든 시그널: Latency, Traffic, Errors, Saturation.
- 알림은 액션 가능해야 한다. 무시되는 알림 = 삭제 대상.
- 알림 기준: SLO 기반 (P99 지연 > 임계값, 에러율 > 임계값).
- 대시보드: 서비스별 개요 + 상세 + 의존성 맵.
- 로그: 구조화(JSON), 상관 ID(trace_id) 필수, 30일 이상 보존.

## 변경 관리

- 프로덕션 변경은 MR/PR로만. 수동 변경 금지 (IaC 원칙).
- 배포 시간: 업무 시간 내 (금요일 오후, 공휴일 전날 배포 금지).
- 롤백 계획 없는 배포 금지. 1-command 롤백 가능해야 한다.
- 배포 후 30분 모니터링 필수. 이상 감지 시 즉시 롤백.

## 용량 관리

- 월 1회 용량 리뷰: CPU, 메모리, 디스크, 네트워크.
- 리소스 80% 도달 시 알림, 90% 도달 시 즉시 확장.
- 자동 스케일링 설정 시 상한/하한 명시. 비용 알림 설정.

## Runbook

- 모든 알림에 대응 runbook 링크.
- Runbook 필수 항목: 증상, 원인, 진단 명령, 해결 절차, 에스컬레이션 경로.
- 분기 1회 runbook 리뷰 + 갱신.

## 금지 사항

- SSH로 프로덕션 서버 직접 수정
- 포스트모템 없이 장애 종료 선언
- 알림 임계값 무한 완화 ("울리니까 올려")

## CQ 연동 (CQ 프로젝트인 경우)

| 작업 | CQ 도구/스킬 |
|------|-------------|
| 장애/배포 알림 전송 | `c4_notify` |
| 서비스 상태 조회 | `/c4-status` |
| 장애 이력/패턴 검색 | `c4_knowledge_search` |
