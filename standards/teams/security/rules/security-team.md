# Security Team Rules

> 보안 감사, 취약점 관리, 보안 정책 규칙.
> 적용: `cq init --team security`

---

## 취약점 관리

- 발견 즉시: 심각도 분류 (Critical/High/Medium/Low).
- Critical/High: 24시간 내 패치 또는 완화 조치.
- 추적: 이슈 트래커에 등록. 해결까지 추적.
- 공개: 패치 완료 전 외부 공개 금지.

## 코드 리뷰 (보안 관점)

- 인증/인가: 모든 엔드포인트에 권한 체크 확인.
- 입력 검증: 화이트리스트 방식. 경계값 테스트.
- 암호화: 저장 데이터(at rest) + 전송 데이터(in transit) 암호화 확인.
- 의존성: 알려진 CVE 없는지 확인. `trivy`, `osv-scanner` 활용.

## 감사 로깅

- 기록 대상: 인증 시도, 권한 변경, 데이터 접근, 관리 작업.
- 형식: 구조화된 로그 (JSON). timestamp, actor, action, resource, result.
- 보존: 최소 90일. 규정에 따라 연장.
- 변조 방지: 감사 로그는 별도 저장소. 어플리케이션에서 삭제 불가.

## 시크릿 관리

- 교체 주기: API 키 90일, 인증서 만료 전 30일 교체.
- 유출 대응: 즉시 교체 → 영향 범위 파악 → 로그 분석.
- 스캔: pre-commit hook으로 시크릿 패턴 탐지 (gitleaks 등).

## 침투 테스트

- 범위: 테스트 전 scope 문서화 + 승인.
- 환경: 프로덕션 직접 테스트 금지. staging 또는 별도 환경.
- 보고: 발견 사항 → 재현 단계 → 심각도 → 권장 조치.
- 도구: 승인된 도구만 사용. 결과는 암호화 저장.

## 컴플라이언스

- 개인정보: 수집 최소화. 목적 달성 후 삭제.
- 데이터 분류: 공개/내부/기밀/극비 등급 구분.
- 접근 기록: 기밀 이상 데이터 접근 시 로그 필수.

## CQ 연동 (CQ 프로젝트인 경우)

| 작업 | CQ 도구/스킬 |
|------|-------------|

| 취약점 발견 기록 | `c4_knowledge_record` |
