# Testing Standards

> 테스트 작성 기준.

## 원칙

- 새 기능/버그 수정에는 테스트가 동반된다.
- 테스트는 독립적이고, 순서에 의존하지 않으며, 반복 실행 가능하다.
- 외부 서비스(DB, API)에 의존하는 테스트는 명시적으로 분리한다.

## 테스트 유형

| 유형 | 범위 | 실행 시점 |
|------|------|----------|
| 단위 (Unit) | 함수/메서드 | 커밋 전 |
| 통합 (Integration) | 모듈 간 상호작용 | MR 시 |
| E2E | 전체 플로우 | 배포 전 |

## 언어별 도구

| 언어 | 단위 테스트 | 린트 |
|------|-----------|------|
| Go | `go test ./...` | `go vet ./...` |
| Python | `uv run pytest` | `uv run ruff check .` |
| TypeScript | `pnpm test` | `pnpm lint` |

## 테스트 작성 규칙

- 테스트 함수명은 `Test_<시나리오>_<기대결과>` 또는 동등 패턴.
- 테스트 데이터는 테스트 안에서 생성. 공유 상태 금지.
- 환경 의존(포트, 파일 경로) → 테스트 시작 시 셋업, 종료 시 정리.
- sleep 기반 동기화 금지 → 채널, 폴링, 이벤트 대기 사용.

## 커버리지

- 엄격한 수치 목표는 두지 않는다. 대신:
- 핵심 비즈니스 로직은 반드시 테스트.
- 에러 경로(실패 케이스)도 반드시 테스트.
- "테스트하기 어렵다"면 설계를 의심.
