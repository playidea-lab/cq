# Doc Review

문서의 품질을 체계적으로 검토합니다.

## 실행 순서

### Step 1: 문서 목록 파악

```bash
find . -name "*.md" -not -path "*/node_modules/*" -not -path "*/.git/*"
```

리뷰 대상 문서 목록을 확인하고 Read로 읽는다.

### Step 2: 정확성 (Accuracy)

- [ ] 코드 예제가 실제로 동작하는가?
- [ ] 커맨드 출력/결과가 현재 버전과 일치하는가?
- [ ] API/함수 시그니처가 코드와 동일한가?
- [ ] 설정 값이 최신인가?

```bash
# 코드 예제의 함수/변수명이 실제로 존재하는지 확인
grep -rn "<예제에서_쓰인_함수명>" --include="*.go" --include="*.py" .
```

### Step 3: 완성도 (Completeness)

- [ ] 설치 방법이 명확한가?
- [ ] 필수 사전 조건(Prerequisites)이 나열됐는가?
- [ ] 주요 기능이 모두 문서화됐는가?
- [ ] 에러 케이스/트러블슈팅 섹션이 있는가?
- [ ] 한국어/영어 혼용이 일관적인가?

### Step 4: 예제 품질 (Examples)

- [ ] 시작하기(Getting Started) 예제가 있는가?
- [ ] 예제가 복사-붙여넣기로 바로 실행되는가?
- [ ] 복잡한 기능은 전체 워크플로우 예제가 있는가?
- [ ] 예제 출력/결과가 명시됐는가?

### Step 5: 접근성 (Accessibility)

- [ ] 초보자도 이해할 수 있는 설명인가?
- [ ] 전문 용어에 설명/링크가 달렸는가?
- [ ] 목차(Table of Contents)가 있는가? (긴 문서)
- [ ] 섹션 구조가 논리적인가?
- [ ] 이미지에 alt 텍스트가 있는가?

### Step 6: 최신성 (Up-to-date)

- [ ] 마지막 수정 날짜가 합리적인가?
- [ ] 더 이상 지원하지 않는 기능/버전이 제거됐는가?
- [ ] CHANGELOG나 버전 이력이 문서에 반영됐는가?

### Step 7: 구조 및 형식

- [ ] Markdown 문법이 올바른가? (헤딩 계층, 코드 블록 언어 지정)
- [ ] 링크가 유효한가?
- [ ] 이미지가 로드되는가?
- [ ] 문서 간 상호 참조가 올바른가?

### Step 8: 리뷰 결과 출력

```
## 문서 리뷰 결과

**대상**: <문서 경로>

### 필수 수정
- 정확성: ...
- 링크 오류: ...

### 개선 제안
- 예제 추가 권장: ...
- 트러블슈팅 섹션 추가: ...

### 잘된 점
- ...
```
