## Go Hybrid 아키텍처 (v0.6.12+)

v0.6.12부터 성능 크리티컬 컴포넌트는 Go로 구현되며, Python 사이드카와 gRPC로 통신합니다.

```
Claude Code → MCP Server (Go c4-core)
                    ├→ State Machine (Go)
                    ├→ TaskStore (Go, SQLite 공유)
                    ├→ Validation Runner (Go)
                    └→ gRPC Bridge → Python Sidecar
                                      ├→ LSP (multilspy, Jedi)
                                      ├→ Knowledge Store (FTS5 + Vector)
                                      ├→ GPU Scheduler
                                      └→ Code Analyzer (tree-sitter)
```

### Go vs Python 책임 구분

| 기능 | 구현 | 비고 |
|------|------|------|
| State Machine | **Go** | Python과 동일 전이 규칙 |
| TaskStore (SQLite) | **Go** | Python과 동일 스키마 공유 |
| MCP Server (10개 도구) | **Go** | status, start, clear, get_task, submit, add_todo, mark_blocked, claim, report, checkpoint |
| CLI | **Go** | cobra 기반 (run, status, stop, add-task, mcp) |
| Auth / Token | **Go** | OAuth + token 관리 |
| Realtime / SSE | **Go** | SSE 스트리밍 |
| gRPC Bridge | **Go ↔ Python** | Protobuf 통신 |
| Code Analysis (LSP) | **Python** | multilspy → Jedi → tree-sitter fallback |
| Knowledge Store v2 | **Python** | FTS5 + sqlite-vec hybrid search |
| GPU/ML Scheduler | **Python** | CUDA/MPS 감지, DAG→Task |
| Experiment Tracker | **Python** | @c4_track 데코레이터 |
| Hook Registry | **Python** | 생명주기 훅 |
| Agent Routing | **Python** | 도메인 → 에이전트 매핑 |
| Supervisor | **Python** | Claude CLI backend |

## 핵심 컴포넌트

### C4Daemon

프로젝트 상태와 태스크 오케스트레이션을 관리하는 중앙 컨트롤러.

- MCP 도구 핸들러의 진입점
- StateMachine, WorkerManager, EventBus 통합
- Direct Mode (claim/report) + Worker Mode (get_task/submit) 이중 지원

### StateMachine

8개 상태와 전이 규칙 관리: INIT → DISCOVERY → DESIGN → PLAN → EXECUTE ⇄ CHECKPOINT → COMPLETE (+ HALTED)

### Code Analysis Engine

3단계 fallback 체인으로 코드 인텔리전스 제공:
1. **Multilspy** — LSP 프로토콜 (가장 정확)
2. **Jedi** — Python 전용 정적 분석
3. **Tree-sitter** — 구문 기반 (모든 언어 지원)

MCP 심볼 도구(find_symbol, replace_symbol_body, rename_symbol 등)와 LSP 서버 양쪽에서 사용.

### Knowledge Store v2

Obsidian-style Markdown 문서 = SSOT:
- **DocumentStore**: `.c4/knowledge/docs/*.md` (YAML frontmatter + body)
- **KnowledgeSearcher**: FTS5(키워드) + sqlite-vec(벡터) → RRF merge
- **4가지 문서 유형**: experiment(`exp-`), pattern(`pat-`), insight(`ins-`), hypothesis(`hyp-`)
- **Backlink**: `[[doc-id]]` 참조로 지식 그래프 구성

### WorkerManager

워커 생명주기 관리:
- Heartbeat 기반 stale 감지 (30분 타임아웃)
- TTL 기반 자동 정리
- 좀비 워커 6가지 원인 대응

### Agent Routing

도메인 기반 에이전트 자동 선택:

| Domain | Primary Agent | Chain |
|--------|--------------|-------|
| `web-frontend` | `frontend-developer` | frontend → test → reviewer |
| `web-backend` | `backend-architect` | architect → python → test → reviewer |
| `ml-dl` | `ml-engineer` | ml → python → test |
| `infra` | `cloud-architect` | cloud → deployment |

Impact 기반 스킬 매칭: `score = impact_weight × (1 + domain_boost) + rule_bonus`

## 확장 포인트 (ABC 인터페이스)

| 인터페이스 | 현재 구현 | Cloud 확장 |
|-----------|----------|-----------|
| `StateStore` | SQLiteStateStore | SupabaseStateStore |
| `KnowledgeStore` | DocumentStore | SharedKnowledgeStore |
| `ArtifactStore` | LocalArtifactStore | CloudArtifactStore |
| `GpuScheduler` | LocalGpuScheduler | CloudGpuScheduler |
| `SupervisorBackend` | ClaudeCliBackend | 다른 LLM 연동 |

## 설계 원칙

1. **Protocol 기반 추상화** — 구현이 아닌 인터페이스에 의존
2. **의존성 주입** — Store, Backend 교체 가능
3. **상태 머신 패턴** — 명시적 전이, 불변 조건 강제
4. **LSP Fallback 체인** — 정확도 순서로 3단계 시도
5. **Markdown SSOT** — 지식은 사람이 읽을 수 있는 형태로 저장
