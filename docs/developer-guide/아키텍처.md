# C4 아키텍처

> 최종 업데이트: 2026-02-09

이 문서는 C4의 내부 구조와 컴포넌트 간 관계를 설명합니다.

## 전체 구조 개요

C4는 Go MCP 서버(Primary)와 Python 사이드카로 구성된 AI 오케스트레이션 시스템입니다. ~153K LOC 대규모 정리를 거쳐 현재 총 **~21K LOC**로 경량화되었습니다.

```
c4/           — Python sidecar (~10.5K LOC) - LSP, Knowledge, GPU만 담당
c4-core/      — Go MCP server (~10.5K LOC) - Primary, 51개 도구
c1/           — Tauri 2.x 데스크톱 앱 (React ~3.2K + Rust ~2.4K LOC)
docs/         — 문서
tests/        — 테스트 (~5.6K LOC)
```

## Go MCP 서버 (Primary) — c4-core/

Go로 구현된 MCP 서버가 Claude Code와 직접 통신하는 Primary 서버입니다. stdio 프로토콜로 51개 도구를 Registry 기반으로 제공합니다.

### c4-core/ 패키지 구조

```
c4-core/
├── cmd/c4/               # CLI 진입점 (cobra), MCP 서버 엔트리
├── internal/
│   ├── mcp/              # MCP Registry + 핸들러 (51개 도구)
│   │   └── handlers/     # 카테고리별 핸들러
│   │       ├── sqlite_store.go  # Spec, Design, Checkpoint (SQLite)
│   │       ├── files.go         # 파일 읽기/검색/패턴 검색
│   │       ├── git.go           # Git 분석, 커밋 검색
│   │       ├── discovery.go     # 요구사항 수집, 스펙 관리
│   │       ├── artifacts.go     # 아티팩트 저장/조회
│   │       ├── proxy.go         # Python 사이드카 JSON-RPC 프록시
```

## 핵심 컴포넌트

### StateMachine

8개 상태와 전이 규칙 관리:

```
INIT → DISCOVERY → DESIGN → PLAN → EXECUTE ⇄ CHECKPOINT → COMPLETE (+ HALTED)
```

Go `internal/state/` 패키지에서 구현. 명시적 전이 규칙과 불변 조건을 강제합니다.

### Code Analysis Engine

3단계 fallback 체인으로 코드 인텔리전스 제공 (`c4/daemon/code_ops.py`):

1. **Multilspy** — LSP 프로토콜 (가장 정확)
2. **Jedi** — Python 전용 정적 분석
3. **Tree-sitter** — 구문 기반 (모든 언어 지원)

MCP 심볼 도구(find_symbol, replace_symbol_body, rename_symbol, insert_before/after_symbol, find_referencing_symbols)가 이 체인을 사용합니다. `_SymbolProxy` 어댑터로 multilspy(0-indexed)와 Jedi(1-indexed) 양쪽을 통합 지원합니다.

### Knowledge Store v2

Obsidian-style Markdown 문서 = SSOT (`c4/knowledge/`):

- **DocumentStore**: `.c4/knowledge/docs/*.md` (YAML frontmatter + body)
- **KnowledgeSearcher**: FTS5(키워드) + sqlite-vec(벡터) → RRF merge
- **4가지 문서 유형**: experiment(`exp-`), pattern(`pat-`), insight(`ins-`), hypothesis(`hyp-`)
- **Backlink**: `[[doc-id]]` 참조로 지식 그래프 구성
- **Vector auto-indexing**: MCP record + hook 양쪽에서 best-effort 인덱싱

### WorkerManager

워커 생명주기 관리 (Go `internal/worker/`):

- Heartbeat 기반 stale 감지 (30분 타임아웃)
- TTL 기반 자동 정리
- 좀비 워커 6가지 원인 대응
- Direct Mode (claim/report) + Worker Mode (get_task/submit) 이중 지원

### JSON-RPC Bridge

Go와 Python 사이드카 간 통신 (`c4-core/internal/bridge/`):

- TCP 기반 JSON-RPC 프로토콜
- 자동 재시작 (사이드카 크래시 시)
- Ping health check
- 16개 도구를 Python 사이드카로 프록시

## 확장 포인트 (ABC 인터페이스)

`c4/interfaces.py`에 정의된 ABC 인터페이스로 백엔드 교체가 가능합니다.

| 인터페이스 | 현재 구현 | Cloud 확장 |
|-----------|----------|-----------|
| `KnowledgeStore` | DocumentStore | SharedKnowledgeStore |
| `ArtifactStore` | LocalArtifactStore | CloudArtifactStore |
| `GpuScheduler` | LocalGpuScheduler | CloudGpuScheduler |

Go 측 TaskStore도 SQLite, Memory, Supabase 3가지 백엔드를 지원합니다.

## 설계 원칙

1. **Protocol 기반 추상화** — 구현이 아닌 인터페이스에 의존 (ABC)
2. **의존성 주입** — Store, Backend 교체 가능
3. **상태 머신 패턴** — 명시적 전이, 불변 조건 강제 (8개 상태)
4. **LSP Fallback 체인** — 정확도 순서로 3단계 시도 (Multilspy → Jedi → Tree-sitter)
5. **Markdown SSOT** — 지식은 사람이 읽을 수 있는 형태로 저장
6. **Go Primary + Python Sidecar** — 성능 크리티컬은 Go, 전문 기능은 Python
