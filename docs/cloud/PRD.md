# C4 Cloud PRD (Product Requirements Document)

## 1. Overview

### 1.1 Product Name
**C4 Cloud** - C4의 호스팅 SaaS 버전

### 1.2 One-liner
C4 Cloud는 **웹 브라우저에서 AI 프로젝트를 생성, 실행, 모니터링하고 결과를 GitHub에 자동 배포하는 서비스**다.

### 1.3 Relationship to C4 Local

| 구분 | C4 Local (v0-v1) | C4 Cloud (v2+) |
|------|-----------------|----------------|
| 대상 | 개발자, 파워유저 | 일반 사용자, 팀 |
| 설치 | `pip install c4d` | 회원가입만 |
| 실행 | 로컬 터미널 | 웹 대시보드 |
| API 키 | 사용자 직접 관리 | C4가 관리 |
| 비용 | API 비용만 (직접) | 구독 + 사용량 |
| 수익 | 없음 | 있음 |

---

## 2. Problem Statement

### 2.1 C4 Local의 한계

1. **진입 장벽**: CLI 설치, API 키 발급, 환경 설정 필요
2. **접근성**: 터미널이 있어야만 확인 가능
3. **확장성**: 워커 추가가 수동 (터미널 여러 개)
4. **결과 관리**: 로컬 파일로만 존재, 공유 어려움
5. **수익화 불가**: 오픈소스 CLI는 과금 어려움

### 2.2 C4 Cloud가 해결하는 것

1. **원클릭 시작**: 회원가입 → 바로 프로젝트 시작
2. **어디서나 확인**: 웹/모바일에서 상태 모니터링
3. **자동 스케일링**: 슬라이더로 워커 수 조절
4. **GitHub 연동**: 결과물 자동 push/PR
5. **확실한 과금**: 사용량 기반 + 구독

## 3. Target Users

### 3.1 Primary Users

1. **솔로 개발자**
   - CLI 싫어함, 웹 선호
   - 사이드 프로젝트 빠르게 진행하고 싶음
   - "코드 짜줘" → GitHub에 결과 원함

2. **스타트업 팀**
   - 개발자 부족, AI로 보완
   - 여러 프로젝트 동시 진행
   - 팀원들과 진행상황 공유 필요

3. **비개발자 (PM, 디자이너)**
   - 터미널 모름
   - 간단한 프로토타입 만들고 싶음
   - 결과만 GitHub에 올라오면 됨

### 3.2 Personas

**Alex (솔로 개발자)**
> "주말에 사이드 프로젝트 하려는데, 보일러플레이트 만드는 데 시간 다 가.
> AI한테 시키고 월요일에 GitHub 보면 되게 해줘."

**팀 리드 Sarah**
> "주니어 개발자 채용이 안 돼. 반복적인 작업은 AI가 하고,
> 시니어들은 리뷰만 하게 만들고 싶어."

**PM Mike**
> "개발자 기다리다 스프린트 다 가. MVP 프로토타입이라도
> AI가 만들어주면 피드백 받고 요구사항 구체화할 수 있는데."

## 4. User Journey

### 4.1 First-time User

```
1. c4.dev 방문
2. "Get Started" 클릭
3. GitHub 로그인 (OAuth)
4. 플랜 선택 (Free로 시작)
5. "New Project" 클릭
6. 프로젝트 설명 입력
   └─ "Next.js 블로그 만들어줘. MDX 지원, 다크모드, Vercel 배포"
7. GitHub 리포 선택 (또는 새로 생성)
8. "Start" 클릭
9. 실시간 진행상황 모니터링
10. 완료 → GitHub PR 알림
11. PR 리뷰 → Merge
```

### 4.2 Returning User

```
1. 대시보드 접속
2. 기존 프로젝트 상태 확인
3. 완료된 프로젝트 → GitHub에서 결과 확인
4. 진행 중인 프로젝트 → 로그 확인, 워커 조절
5. 새 프로젝트 시작
```

### 4.3 Team User

```
1. Team 플랜 구독
2. 팀원 초대 (이메일)
3. 팀 프로젝트 생성
4. 팀원들과 프로젝트 공유
5. 공동 모니터링
6. 감사 로그 확인
```

## 6. Non-Functional Requirements

### NFR-1: Performance

- 대시보드 로딩 < 2초
- 로그 스트림 지연 < 500ms
- 워커 스케일링 < 30초

### NFR-2: Availability

- 99.9% uptime (월 43분 다운타임 허용)
- 자동 장애 복구
- 데이터 백업 (매일)

### NFR-3: Security

- SOC 2 Type II (목표)
- 데이터 암호화 (at rest, in transit)
- 워크스페이스 격리

### NFR-4: Scalability

- 동시 워커 1000개 지원
- 사용자 10만 명 지원

## 8. Business Model

### 8.1 Pricing Tiers

| Tier | Price | Credits | Workers | Key Features |
|------|-------|---------|---------|--------------|
| Free | $0 | $10/월 | 1 | 체험용, 공개 리포만 |
| Pro | $30/월 | $50/월 | 5 | 모든 모델, 비공개 리포 |
| Team | $100/월 | $200/월 | 20 | 팀 협업, 감사 로그 |
| Enterprise | 문의 | 협의 | 무제한 | SSO, SLA, 온프레미스 |

### 8.2 Revenue Model

```
수익 = 구독료 + (사용량 - 포함 크레딧) × 마크업

예시 (Pro 사용자):
- 구독료: $30
- 포함 크레딧: $50
- 실제 사용: $70
- 추가 과금: ($70 - $50) × 1.2 = $24
- 총 수익: $30 + $24 = $54
- API 비용: $70
- 순이익: $54 - $70 = -$16 (이 경우 손해)

→ 마크업 또는 포함 크레딧 조정 필요
```

### 8.3 Unit Economics Target

- CAC (고객획득비용): < $50
- LTV (고객생애가치): > $300
- LTV/CAC: > 6
- Gross Margin: > 50%

## 9. Success Metrics (KPIs)

### 9.1 Acquisition

- 월간 신규 가입자 수
- 무료 → 유료 전환율 (목표: 5%)
- CAC

### 9.2 Engagement

- 월간 활성 사용자 (MAU)
- 프로젝트 생성 수 / 사용자
- 평균 세션 시간

### 9.3 Revenue

- MRR (월간 반복 수익)
- ARPU (사용자당 평균 매출)
- Churn rate (목표: < 5%/월)

### 9.4 Product

- 프로젝트 완료율
- GitHub push 성공률
- 평균 프로젝트 완료 시간

## 10. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| API 비용 > 수익 | High | 마크업 조정, 사용량 제한 |
| GitHub API 제한 | Medium | 캐싱, rate limit 관리 |
| LLM 품질 저하 | High | 다중 모델, fallback |
| 보안 사고 | Critical | 워크스페이스 격리, 감사 |
| 경쟁사 출현 | Medium | 빠른 기능 개발, 커뮤니티 |
