# C4/C5 워커 가이드

Claude Code 세션을 C5 Hub의 워커로 전환하여, Hub에서 잡을 받아 자동 실행하는 시스템입니다.

## 아키텍처 요약

```
                    ┌─────────────┐
                    │   C5 Hub    │  (Go 분산 작업 큐)
                    │  :8585/v1   │
                    └──────┬──────┘
                           │ REST API
              ┌────────────┼────────────┐
              v            v            v
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Worker A │ │ Worker B │ │ Worker C │  (Claude Code 세션)
        │ standby  │ │ standby  │ │ standby  │
        └──────────┘ └──────────┘ └──────────┘
              │
              └─→ C1 Messenger (worker- 채널, 상태 표시)
```

## 사전 요구사항

### 1. C5 Hub 실행

```bash
cd c5 && go build -o bin/c5 ./cmd/c5/
./bin/c5 serve --port 8585 --db ./c5.db
```

기본 포트는 8585. health 확인: `curl http://localhost:8585/v1/health`

### 2. C4 Hub 설정 (.c4/config.yaml)

```yaml
hub:
  enabled: true
  url: "http://localhost:8585"
  api_prefix: "/v1"          # C5 Hub은 /v1 prefix 필수
  team_id: "my-team"
```

**주의사항:**
- `api_prefix: "/v1"`이 빠지면 404 에러 발생
- config 변경 후 Claude Code **세션 재시작** 필요 (MCP 서버가 기동 시 config을 로드)
- API key 인증이 필요한 경우: C5 Hub에 `--api-key` 설정 후 `.env`에 키 추가

### 3. 바이너리 재빌드

config이나 핸들러 코드 변경 후:
```bash
cd c4-core && go build -o ~/.local/bin/cq ./cmd/c4/
```
그 다음 Claude Code 세션 재시작.

## MCP 도구 3개

### c4_worker_standby

Hub에 워커로 등록하고, 잡이 올 때까지 **블로킹 대기**합니다.

```
c4_worker_standby(
  worker_id: "worker-1",           # 워커 식별자 (필수)
  capabilities: {                   # 선택
    "tags": ["claude", "mcp"]
  }
)
```

**동작:**
1. Hub에 워커 등록 (hostname 자동 주입)
2. C1 Messenger에 `worker-{id}` 채널 생성 + presence 설정
3. **5초 간격 폴링** — 잡 큐에서 claim 시도
4. **30초 간격 heartbeat** — Hub에 생존 신호
5. 잡 발견 시 → job_id, command, lease_id 반환
6. shutdown signal 감지 시 → `{shutdown: true, reason: "..."}` 반환

**반환 (잡 할당):**
```json
{
  "job_id": "j-abc123",
  "command": "task_id=T-001-0",
  "lease_id": "l-xyz789",
  "name": "untitled",
  "workdir": ".",
  "env": null,
  "tags": null
}
```

**반환 (shutdown):**
```json
{
  "shutdown": true,
  "reason": "사용자 요청"
}
```

### c4_worker_complete

잡 완료를 Hub에 보고합니다.

```
c4_worker_complete(
  job_id: "j-abc123",             # 필수
  worker_id: "worker-1",          # 필수
  status: "SUCCEEDED",            # 필수: SUCCEEDED | FAILED
  lease_id: "l-xyz789",           # 선택 (현재 미사용, 향후 검증용 예약)
  commit_sha: "abc1234",          # 선택: 코드 커밋 시
  summary: "구현 완료"             # 선택: 작업 요약
)
```

**동작:**
1. Hub에 CompleteJob 호출
2. C1 Messenger worker 채널에 결과 포스팅
3. presence를 idle로 변경

### c4_worker_shutdown

워커에게 정상 종료를 요청합니다. 다음 폴 사이클(~5초 내)에 종료됩니다.

```
c4_worker_shutdown(
  worker_id: "worker-1",          # 필수
  reason: "작업 완료"              # 선택 (기본: "shutdown requested")
)
```

**동작:** SQLite에 shutdown signal 저장 → standby가 다음 폴에서 감지 → 반환

## 워크플로우

### 기본 흐름: 잡 제출 → 워커 실행 → 완료

```
1. 잡 제출 (Hub API 또는 c4_hub_submit)
   curl -X POST http://localhost:8585/v1/jobs/submit \
     -d '{"command":"task_id=T-001-0","priority":5}'

2. 워커 대기
   c4_worker_standby(worker_id="w1")
   → 잡 반환: {job_id, command, lease_id}

3. 잡 실행
   command에서 task_id 파싱 → c4_get_task → 구현

4. 완료 보고
   c4_worker_complete(job_id, worker_id="w1", status="SUCCEEDED", commit_sha="...")

5. 다시 대기 (루프)
   c4_worker_standby(worker_id="w1")
```

### /c4-standby 스킬 (자동화)

Claude Code 세션을 상주 워커로 전환합니다:

```
/c4-standby
```

내부적으로 `standby → 잡 파싱 → 실행 → complete → 다시 standby` 루프를 실행합니다.

### 다중 워커 병렬 실행

```
# 터미널 1
claude> /c4-standby   (worker_id=worker-1)

# 터미널 2
claude> /c4-standby   (worker_id=worker-2)

# 잡 제출 (2개 동시 처리)
c4_hub_submit(command="task_id=T-001-0")
c4_hub_submit(command="task_id=T-002-0")
```

## 설정 참조

### Hub Config 필드

| 필드 | 필수 | 기본값 | 설명 |
|------|------|--------|------|
| `hub.enabled` | yes | false | Hub 연동 활성화 |
| `hub.url` | yes | — | C5 Hub 서버 URL |
| `hub.api_prefix` | yes | "" | API 경로 prefix (`/v1` for C5 Hub) |
| `hub.team_id` | no | — | 팀 식별자 |

### C5 Hub 서버 옵션

| 옵션 | 기본값 | 설명 |
|------|--------|------|
| `--port` | 8585 | HTTP 리스닝 포트 |
| `--db` | ./c5.db | SQLite 데이터베이스 경로 |
| `--api-key` | — | API 키 인증 (선택) |

### 워커 타이밍

| 항목 | 값 | 설명 |
|------|-----|------|
| 폴링 간격 | 5초 | 잡 큐 확인 주기 |
| Heartbeat | 30초 | Hub에 생존 신호 |
| Stale 판정 | 2분 | heartbeat 미수신 시 offline 처리 |
| Lease 만료 | 5분 | 갱신 없으면 잡 재큐 |

## 트러블슈팅

### "hostname is required" 에러
`c4_worker_standby`가 자동으로 `os.Hostname()`을 주입합니다. 이 에러가 나면 바이너리가 구 버전입니다:
```bash
cd c4-core && go build -o ~/.local/bin/cq ./cmd/c4/
# 세션 재시작
```

### "404 Not Found" on /workers/register
`api_prefix`가 비어있을 때 발생. C5 Hub은 `/v1` prefix가 필요합니다:
```yaml
hub:
  api_prefix: "/v1"    # 이것이 빠지면 404
```

### "connection refused"
C5 Hub이 실행 중이 아닙니다:
```bash
curl http://localhost:8585/v1/health
# 실패하면 Hub 시작: cd c5 && ./bin/c5 serve
```

### MCP 도구가 안 보임
Hub이 비활성화되어 있으면 워커 도구가 등록되지 않습니다:
1. `.c4/config.yaml`에서 `hub.enabled: true` 확인
2. Hub URL이 접근 가능한지 확인
3. 세션 재시작

### 잡은 큐에 있는데 워커가 못 가져감
- 워커가 offline 상태인지 확인: `curl http://localhost:8585/v1/workers`
- Lease가 만료되어 잡이 재큐되었는지 확인: `c4_hub_status(job_id="...")`
- heartbeat가 정상인지 MCP 서버 stderr 로그 확인

## C1 Messenger 연동

워커가 standby에 진입하면:
- `worker-{id}` 채널 자동 생성 (channel_type: `worker`)
- ChannelSidebar의 **WORKERS** 섹션에 표시
- 상태 dot: online(녹색) / idle(노란색) / offline(회색)
- 잡 claim/complete 시 자동 포스팅

Messenger가 비활성 상태(Supabase 미연결)이면 워커 기능은 정상이고 채널/포스팅만 생략됩니다.
