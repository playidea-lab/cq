# 명령어 레퍼런스

C4는 Claude Code에서 슬래시 명령어로 사용합니다.
이 문서는 모든 명령어의 사용법을 설명합니다.

## 명령어 요약

| 명령어 | 설명 | 사용 가능 상태 |
|--------|------|----------------|
| `/c4-init` | 프로젝트 초기화 | 초기화 전 |
| `/c4-plan` | 계획 등록 | INIT, HALTED |
| `/c4-run` | 실행 시작 | PLAN, HALTED |
| `/c4-stop` | 실행 중지 | EXECUTE |
| `/c4-status` | 상태 조회 | 모든 상태 |
| `/c4-worker` | Worker 작업 | EXECUTE |
| `/c4-validate` | 검증 실행 | EXECUTE |
| `/c4-submit` | 태스크 제출 | EXECUTE |
| `/c4-checkpoint` | 체크포인트 리뷰 | CHECKPOINT |
| `/c4-add-task` | 태스크 추가 | EXECUTE, CHECKPOINT |

---

## /c4-init

프로젝트를 C4로 초기화합니다.

### 사용법
```
/c4-init [project_name]
```

### 파라미터
- `project_name` (선택): 프로젝트 이름. 생략 시 디렉토리 이름 사용

### 생성되는 구조
```
.c4/
├── c4.db           # SQLite 데이터베이스 (상태, 잠금)
├── config.yaml     # 설정 파일
├── tasks.json      # 태스크 정의
├── events/         # 이벤트 로그
├── bundles/        # 체크포인트 번들
├── workers/        # Worker 정보
└── runs/           # 실행 결과
    ├── tests/
    └── logs/

docs/
└── PLAN.md         # 계획 템플릿
```

### 예시
```
/c4-init my-project
```

## /c4-plan

`docs/PLAN.md`를 파싱하여 태스크와 체크포인트를 등록합니다.

### 사용법
```
/c4-plan
```

### 동작
1. `docs/PLAN.md` 파일 파싱
2. 태스크 추출 및 큐에 등록
3. 체크포인트 정의 등록
4. 상태 저장

### PLAN.md 형식
```markdown
## 태스크

### T-001: 태스크 제목
- **설명**: 태스크 설명
- **범위**: `src/` (scope lock 범위)
- **의존성**: T-000 (선행 태스크)
- **DoD**:
  - [ ] 완료 조건 1
  - [ ] 완료 조건 2

## 체크포인트

### CP-001: 체크포인트 이름
- **조건**:
  - required_tasks: [T-001]
  - required_validations: [lint, unit]
```

## /c4-run

EXECUTE 상태로 전환하여 태스크 실행을 시작합니다.

### 사용법
```
/c4-run
```

### 동작
1. PLAN → EXECUTE 상태 전환
2. 이벤트 로그 기록
3. Worker 수락 준비

### 주의
- PLAN 또는 HALTED 상태에서만 실행 가능
- EXECUTE 상태에서는 `/c4-stop`으로 중지 가능

---

## /c4-stop

실행을 일시 중지하고 HALTED 상태로 전환합니다.

### 사용법
```
/c4-stop
```

### 동작
1. EXECUTE → HALTED 상태 전환
2. 진행 중인 태스크는 완료되지 않음
3. 이벤트 로그 기록

---

## /c4-worker

Worker로 태스크 루프를 실행합니다.

### 사용법
```
/c4-worker join [worker_id]
/c4-worker submit
```

### 서브커맨드

#### `join`
Worker Loop을 시작하여 태스크를 자동으로 처리합니다.

```
/c4-worker join
```

Worker Loop 흐름:
1. `c4_get_task` - 다음 태스크 요청
2. 태스크 구현 수행
3. `c4_run_validation` - 검증 실행
4. `c4_submit` - 결과 제출
5. 1번으로 반복

#### `submit`
현재 진행 중인 태스크를 제출합니다 (수동 모드).

```
/c4-worker submit
```

## /c4-validate

검증(lint, test)을 실행합니다.

### 사용법
```
/c4-validate [validation_names...]
```

### 파라미터
- `validation_names` (선택): 실행할 검증 이름. 생략 시 모두 실행

### 지원 검증
| 이름 | 설명 | 기본 명령 |
|------|------|-----------|
| `lint` | 코드 스타일 검사 | `uv run ruff check` |
| `unit` | 단위 테스트 | `uv run pytest tests/unit` |
| `integration` | 통합 테스트 | `uv run pytest tests/integration` |
| `e2e` | E2E 테스트 | `uv run pytest tests/e2e` |

### 예시
```
/c4-validate lint unit
```

### 설정
`.c4/config.yaml`에서 검증 명령을 커스터마이즈할 수 있습니다:

```yaml
validations:
  lint:
    command: "uv run ruff check --fix"
  unit:
    command: "uv run pytest tests/unit -v"
  custom:
    command: "./scripts/custom-check.sh"
```

## /c4-submit

완료된 태스크를 제출합니다.

### 사용법
```
/c4-submit <task_id> <commit_sha> [validation_results]
```

### 파라미터
- `task_id`: 완료된 태스크 ID
- `commit_sha`: Git 커밋 SHA
- `validation_results`: 검증 결과 JSON (선택)

### 예시
```
/c4-submit T-001 abc123 '[{"name":"lint","status":"pass"},{"name":"unit","status":"pass"}]'
```

### 동작
1. 태스크 상태를 DONE으로 변경
2. 검증 결과 기록
3. 체크포인트 조건 확인
4. 조건 만족 시 CHECKPOINT 상태 전환

---

## /c4-checkpoint

체크포인트 리뷰를 수행합니다.

### 사용법
```
/c4-checkpoint [decision] [notes]
```

### 파라미터
- `decision` (선택): APPROVE, REQUEST_CHANGES, REPLAN
- `notes` (선택): 리뷰 코멘트

### 자동 모드
파라미터 없이 실행하면 Supervisor Loop이 자동으로 리뷰합니다:

```
/c4-checkpoint
```

### 수동 모드
결정을 직접 지정할 수 있습니다:

```
/c4-checkpoint APPROVE "LGTM, 잘 구현되었습니다"
/c4-checkpoint REQUEST_CHANGES "테스트 커버리지 부족"
/c4-checkpoint REPLAN "아키텍처 재설계 필요"
```

### 결정 유형
| 결정 | 설명 | 다음 상태 |
|------|------|-----------|
| `APPROVE` | 승인, 다음 태스크 진행 | EXECUTE |
| `APPROVE_FINAL` | 최종 승인 | COMPLETE |
| `REQUEST_CHANGES` | 수정 요청, 새 태스크 추가 | EXECUTE |
| `REPLAN` | 계획 재수립 필요 | PLAN |

## /c4-add-task

새 태스크를 큐에 추가합니다.

### 사용법
```
/c4-add-task <task_id> <title> [options]
```

### 파라미터
- `task_id`: 고유 태스크 ID
- `title`: 태스크 제목
- `--scope`: 범위 (예: `src/`)
- `--dod`: 완료 정의
- `--deps`: 의존성 (쉼표 구분)

### 예시
```
/c4-add-task T-FIX-001 "버그 수정: 로그인 오류" --scope "src/auth/" --dod "로그인 테스트 통과"
```

### 사용 시나리오
- REQUEST_CHANGES 결정 후 수정 태스크 추가
- 긴급 버그 수정 태스크 추가
- 리팩토링 태스크 추가
