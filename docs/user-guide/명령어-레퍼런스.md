# 명령어 레퍼런스

C4는 클라이언트별 **Skill/Agent**를 통해 동일 MCP 도구를 호출합니다.
이 문서의 `/c4-*` 표기는 Claude Code 기준 별칭이며, Codex에서는 `.codex/agents/c4-*.md` 트리거로 대응됩니다.

---

## cq CLI 명령어

### 세션 관리

```bash
cq [-t <name>]         # 프로젝트 초기화 + Claude Code 실행 (named session)
cq claude -t <name>    # 동일 (명시적)
cq ls                  # 저장된 세션 목록 (tmux-style)
cq session rm <name>   # 세션 삭제
```

`-t <name>` 지정 시 세션이 저장되어 이후 동일 이름으로 재접속하면 대화가 이어집니다.
탭 자동완성 지원: `cq -t <Tab>` → 세션 목록.

### Shell 자동완성

```bash
cq completion zsh   # zsh completion 스크립트 출력
cq completion bash  # bash completion 스크립트 출력
cq completion fish  # fish completion 스크립트 출력
```

`cq init` 및 `install.sh` 실행 시 자동으로 `~/.zshrc` / `~/.bashrc`에 추가됩니다.

### cq serve

```bash
cq serve             # 장기 실행 서비스 시작 (StaleChecker, EventBus, HubPoller 등)
cq serve install     # OS 서비스로 등록 (macOS: LaunchAgent, Linux: systemd)
cq serve uninstall   # OS 서비스 해제
cq serve start       # OS 서비스 시작
cq serve stop        # OS 서비스 중지
cq serve status      # 서비스 상태 확인
```

### 기타

```bash
cq auth login            # GitHub OAuth 로그인 (브라우저 자동 열기)
cq auth login --device  # headless/SSH: user_code 표시 → 브라우저에서 입력 (RFC 8628)
cq auth login --link    # 인증 URL 직접 출력 → 수동으로 열기
cq auth status       # 인증 상태 확인
cq secret set <key>  # API 키 암호화 저장
cq doctor            # 환경 진단
cq version           # 버전 확인
cq mail ls           # 세션 간 메일 목록
```

## 명령어 요약

| 명령어 | 설명 | 사용 가능 상태 |
|--------|------|----------------|
| `/c4-init` | 프로젝트 초기화 | 초기화 전 |
| `/c4-plan` | 계획 등록 (Discovery → Design → Tasks) | INIT, HALTED |
| `/c4-run` | Worker Loop 실행 | PLAN, HALTED, EXECUTE |
| `/c4-quick` | 즉시 태스크 생성 + Worker 실행 | PLAN, HALTED, EXECUTE |
| `/c4-stop` | 실행 중지 | EXECUTE |
| `/c4-status` | 상태 조회 | 모든 상태 |
| `/c4-validate` | 검증 실행 | EXECUTE |
| `/c4-submit` | 태스크 제출 | EXECUTE |
| `/c4-checkpoint` | 체크포인트 리뷰 | CHECKPOINT |
| `/c4-add-task` | 태스크 추가 | EXECUTE, CHECKPOINT |
| `/c4-swarm` | 팀 협업 (병렬 구현/리뷰/조사) | EXECUTE |
| `/c4-polish` | 품질 수렴 루프 (CRITICAL+HIGH=0 → 수정사항=0) | 구현 완료 후 |
| `/c4-finish` | 완료 루틴 (빌드·테스트·설치·문서·커밋) | polish 후 |
| `/c4-release` | 체인지로그 생성 + 버전 태그 | finish 후 (또는 수동) |
| `/c4-attach` | 현재 세션에 이름 붙이기 | 모든 상태 |
| `/c4-reboot` | named 세션 재시작 | 모든 상태 |

## /c4-init

프로젝트를 C4로 초기화합니다.

### 사용법
```
/c4-init [project_name]
```

### 파라미터
- `project_name` (선택): 프로젝트 이름. 생략 시 디렉토리 이름 사용

### 생성되는 구조
```
.c4/
├── c4.db           # SQLite 데이터베이스 (상태, 잠금)
├── config.yaml     # 설정 파일
├── tasks.json      # 태스크 정의
├── events/         # 이벤트 로그
├── bundles/        # 체크포인트 번들
├── workers/        # Worker 정보
└── runs/           # 실행 결과
    ├── tests/
    └── logs/

docs/
└── PLAN.md         # 계획 템플릿
```

### 예시
```
/c4-init my-project
```

## /c4-plan

`docs/PLAN.md`를 파싱하여 태스크와 체크포인트를 등록합니다.

### 사용법
```
/c4-plan
```

### 동작
1. `docs/PLAN.md` 파일 파싱
2. 태스크 추출 및 큐에 등록
3. 체크포인트 정의 등록
4. 상태 저장

### PLAN.md 형식
```markdown
## 태스크

### T-001: 태스크 제목
- **설명**: 태스크 설명
- **범위**: `src/` (scope lock 범위)
- **의존성**: T-000 (선행 태스크)
- **DoD**:
  - [ ] 완료 조건 1
  - [ ] 완료 조건 2

## 체크포인트

### CP-001: 체크포인트 이름
- **조건**:
  - required_tasks: [T-001]
  - required_validations: [lint, unit]
```

## /c4-run

Worker Loop를 실행합니다. 상태에 따라 자동 처리:
- PLAN/HALTED → EXECUTE 전환 후 작업 시작
- EXECUTE → 바로 작업 참여 (멀티 워커 지원)

### 사용법
```
/c4-run
```

### 동작
1. 상태 확인 (PLAN/HALTED → EXECUTE 전환, EXECUTE → 바로 참여)
2. Worker Loop 시작:
   - `c4_get_task` - 다음 태스크 요청
   - 태스크 구현 수행
   - `c4_run_validation` - 검증 실행
   - `c4_submit` - 결과 제출
   - 반복

### 멀티 워커
여러 Claude Code 창에서 동시에 `/c4-run` 실행 가능:
- 첫 번째 창: PLAN → EXECUTE 전환 + 작업
- 추가 창들: 바로 작업 참여

SQLite WAL 모드로 race condition 없이 안전하게 동작합니다.

### 주의
- CHECKPOINT 상태에서는 리뷰 대기
- COMPLETE 상태에서는 실행 불가

## /c4-stop

실행을 일시 중지하고 HALTED 상태로 전환합니다.

### 사용법
```
/c4-stop
```

### 동작
1. EXECUTE → HALTED 상태 전환
2. 진행 중인 태스크는 완료되지 않음
3. 이벤트 로그 기록

## /c4-validate

검증(lint, test)을 실행합니다.

### 사용법
```
/c4-validate [validation_names...]
```

### 파라미터
- `validation_names` (선택): 실행할 검증 이름. 생략 시 모두 실행

### 지원 검증
| 이름 | 설명 | 기본 명령 |
|------|------|-----------|
| `lint` | 코드 스타일 검사 | `uv run ruff check` |
| `unit` | 단위 테스트 | `uv run pytest tests/unit` |
| `integration` | 통합 테스트 | `uv run pytest tests/integration` |
| `e2e` | E2E 테스트 | `uv run pytest tests/e2e` |

### 예시
```
/c4-validate lint unit
```

### 설정
`.c4/config.yaml`에서 검증 명령을 커스터마이즈할 수 있습니다:

```yaml
validations:
  lint:
    command: "uv run ruff check --fix"
  unit:
    command: "uv run pytest tests/unit -v"
  custom:
    command: "./scripts/custom-check.sh"
```

## /c4-submit

완료된 태스크를 제출합니다.

### 사용법
```
/c4-submit <task_id> <commit_sha> [validation_results]
```

### 파라미터
- `task_id`: 완료된 태스크 ID (필수)
- `commit_sha`: Git 커밋 SHA (필수)
- `validation_results`: 검증 결과 JSON 배열 (선택 — 생략 시 응답에 `validation_skipped: true` 포함)
  - 제공 시 반드시 1개 이상의 항목 포함 (빈 배열 `[]` 는 에러)
  - 각 항목의 `status`는 반드시 `"pass"` 또는 `"fail"`이어야 함 (그 외 값은 즉시 에러 반환)

### 예시
```
/c4-submit T-001 abc123 '[{"name":"lint","status":"pass"},{"name":"unit","status":"pass"}]'
```

검증 결과 없이 제출:
```
/c4-submit T-001 abc123
```

### 동작
1. `validation_results[*].status` 값을 `pass|fail` enum으로 서버측 검증 (그 외 값은 에러)
2. 태스크 상태를 DONE으로 변경
3. 검증 결과 기록 (`validation_results` 미제공 시 `validation_skipped: true` 반환)
4. 체크포인트 조건 확인
5. 조건 만족 시 CHECKPOINT 상태 전환

## /c4-checkpoint

체크포인트 리뷰를 수행합니다.

### 사용법
```
/c4-checkpoint [decision] [notes]
```

### 파라미터
- `decision` (선택): APPROVE, APPROVE_FINAL, REQUEST_CHANGES, REPLAN
- `notes` (선택): 리뷰 코멘트

### 자동 모드
파라미터 없이 실행하면 Supervisor Loop이 자동으로 리뷰합니다:

```
/c4-checkpoint
```

### 수동 모드
결정을 직접 지정할 수 있습니다:

```
/c4-checkpoint APPROVE "LGTM, 잘 구현되었습니다"
/c4-checkpoint APPROVE_FINAL "최종 승인, 프로젝트 완료"
/c4-checkpoint REQUEST_CHANGES "테스트 커버리지 부족"
/c4-checkpoint REPLAN "아키텍처 재설계 필요"
```

### 결정 유형
| 결정 | 설명 | 다음 상태 |
|------|------|-----------|
| `APPROVE` | 승인, 다음 태스크 진행 | EXECUTE |
| `APPROVE_FINAL` | 최종 승인 | COMPLETE |
| `REQUEST_CHANGES` | 수정 요청, 새 태스크 추가 | EXECUTE |
| `REPLAN` | 계획 재수립 필요 | PLAN |

## /c4-add-task

새 태스크를 큐에 추가합니다.

### 사용법
```
/c4-add-task <task_id> <title> [options]
```

### 파라미터
- `task_id`: 고유 태스크 ID
- `title`: 태스크 제목
- `--scope`: 범위 (예: `src/`)
- `--dod`: 완료 정의
- `--deps`: 의존성 (쉼표 구분)

### 예시
```
/c4-add-task T-FIX-001 "버그 수정: 로그인 오류" --scope "src/auth/" --dod "로그인 테스트 통과"
```

### 사용 시나리오
- REQUEST_CHANGES 결정 후 수정 태스크 추가
- 긴급 버그 수정 태스크 추가
- 리팩토링 태스크 추가

## 설정 레퍼런스

`.c4/config.yaml` 파일의 주요 설정 옵션입니다.

### 브랜치 전략

```yaml
# 기본 브랜치 (PR/머지 대상)
default_branch: main

# 작업 브랜치 (null이면 c4/{project_id} 사용)
work_branch: null

# 플랜 완료 시 동작
# - merge: main으로 squash-merge (기본값)
# - pr: GitHub PR 생성
# - manual: 아무것도 안 함
completion_action: merge
```

### Review-as-Task

```yaml
# 리뷰를 별도 태스크로 생성 (기본값: true)
review_as_task: true

# 최대 재작업 횟수: max_revision=N이면 N회 구현 시도 허용 (버전 0..N-1).
# 버전 N 생성 시점에 BLOCKED (nextVersion >= max_revision).
# 예: max_revision=3 → 버전 0,1,2 허용, 버전 3에서 BLOCKED.
max_revision: 3

# 리뷰 태스크 우선순위 감소량
review_priority_offset: 10
```

### 검증 설정

```yaml
validation:
  commands:
    lint: "uv run ruff check"
    unit: "uv run pytest tests/unit"
    e2e: "npm run e2e"
  required:
    - lint
    - unit
```

### 예산 제한

```yaml
budgets:
  max_iterations_per_task: 7      # 태스크당 최대 반복
  max_failures_same_signature: 3  # 동일 에러 최대 실패 횟수
```
