# 워크플로우 개요

C4는 상태 머신(State Machine) 기반으로 프로젝트 진행을 관리합니다.
이 문서는 각 상태와 전이 과정을 상세히 설명합니다.

## 전체 워크플로우

```
INIT → DISCOVERY → DESIGN → PLAN → EXECUTE ⇄ CHECKPOINT → REFINE → POLISH → COMPLETE
```

DISCOVERY와 DESIGN은 PLAN 단계 진입 전 `/c4-plan` 스킬 내부의 서브-페이즈입니다.
CHECKPOINT 이후에는 REFINE(재구현 루프)과 POLISH(품질 수렴 루프)를 거쳐 COMPLETE에 도달합니다.

## 상태 다이어그램

```
                    ┌─────────────────────────────────────────┐
                    │                                         │
                    v                                         │
┌──────┐  /c4-plan ┌──────────────────────────┐  /c4-run  ┌─────────┐         │
│ INIT │ ─────────▶│ DISCOVERY → DESIGN → PLAN │ ─────────▶│ EXECUTE │ ────────┤
└──────┘           └──────────────────────────┘            └─────────┘         │
                                                                 │              │
                                                                 │ gate_reached │
                                                                 v              │
                                                        ┌────────────┐         │
                                                        │ CHECKPOINT │ ────────┘
                                                        └────────────┘
                                                             │
                                               approve / approve_final
                                                             │
                                                    ┌────────┴────────┐
                                                    v                 v
                                             ┌──────────┐    ┌──────────────┐
                                             │  REFINE  │    │    POLISH    │
                                             └──────────┘    └──────────────┘
                                                  │                  │
                                               REQUEST_CHANGES=0     │ 수정사항=0
                                                  │                  v
                                                  └────────────▶ COMPLETE

                    c4_stop          ┌──────────┐
           ─────────────────────────▶│  HALTED  │
                                     └──────────┘   c4_run/c4_plan

                    fatal_error      ┌───────┐
           ─────────────────────────▶│ ERROR │
                                     └───────┘
```

## 상태 설명

### INIT
- **설명**: 프로젝트가 C4로 초기화되기 전 상태
- **진입 조건**: `/c4-init` 실행 전
- **허용 작업**: `init`
- **전이**: `/c4-plan` → DISCOVERY (서브-페이즈 시작)

### DISCOVERY (서브-페이즈)
- **설명**: PLAN 단계 내 첫 번째 서브-페이즈. 요구사항을 수집하고 스펙을 정의합니다.
- **진입 조건**: `/c4-plan` 스킬 실행 시 자동 진입
- **주요 활동**:
  - EARS 요구사항 작성 (`c4_save_spec`)
  - 기존 코드베이스 분석
  - 요구사항 범위/비범위 명확화
- **전이**: `c4_discovery_complete` → DESIGN

### DESIGN (서브-페이즈)
- **설명**: PLAN 단계 내 두 번째 서브-페이즈. 아키텍처와 구현 방향을 결정합니다.
- **진입 조건**: DISCOVERY 완료 후
- **주요 활동**:
  - 아키텍처 결정 문서 작성 (`c4_save_design`)
  - Lighthouse 계약 정의 (`c4_lighthouse register`)
  - 태스크 분해 및 DoD 작성 (`c4_add_todo`)
- **전이**: `c4_design_complete` → PLAN (태스크 실행 준비 완료)

### PLAN
- **설명**: DISCOVERY→DESIGN 서브-페이즈를 거쳐 태스크 계획이 수립된 상태
- **진입 조건**: DISCOVERY/DESIGN 완료 후 또는 CHECKPOINT REPLAN 결정 후
- **허용 작업**: `plan`, `run`, `stop`, `status`
- **주요 활동**:
  - 태스크 정의 및 의존성 설정
  - 체크포인트 정의
- **전이**:
  - `/c4-run` → EXECUTE
  - `c4_stop` → HALTED

### EXECUTE
- **설명**: Worker들이 태스크를 수행 중
- **진입 조건**: `/c4-run` 실행 후
- **허용 작업**: `status`, `worker join`, `worker submit`, `stop`
- **주요 활동**:
  - Worker Loop 실행 (태스크 수행)
  - 검증 실행 (lint, test)
  - 태스크 제출
- **전이**:
  - `gate_reached` → CHECKPOINT (체크포인트 조건 만족)
  - `all_done` → COMPLETE (모든 태스크 및 체크포인트 완료)
  - `c4_stop` → HALTED
  - `fatal_error` → ERROR

### CHECKPOINT
- **설명**: Supervisor가 변경사항을 리뷰 중
- **진입 조건**: 체크포인트 게이트 조건 만족
- **허용 작업**: `status`
- **주요 활동**:
  - 변경사항 번들 생성
  - Supervisor 리뷰
  - 결정 기록
- **전이**:
  - `approve` → EXECUTE (다음 태스크 진행)
  - `approve_final` → REFINE → POLISH (품질 수렴 루프로 진행)
  - `request_changes` → EXECUTE (수정 태스크 추가됨)
  - `replan` → PLAN (계획 재수립 필요)
  - `fatal_error` → ERROR

### REFINE
- **설명**: 리뷰 태스크(R-)에서 REQUEST_CHANGES가 발생했을 때의 재구현 루프
- **진입 조건**: CHECKPOINT 이후 리뷰 태스크(R-)가 REQUEST_CHANGES를 반환할 때
- **핵심 개념**:
  - 리뷰어가 코드를 리뷰하고 변경 요청(REQUEST_CHANGES) 발행
  - RPR- 태스크(수정 재작업)가 생성되어 Worker가 수정 구현
  - 수정 완료 후 새 리뷰 라운드 시작 (RF- 태스크)
  - REQUEST_CHANGES가 0이 될 때까지 반복
- **태스크 흐름**:
  ```
  R-001-0 (리뷰) → REQUEST_CHANGES →
  RPR-001-0 (수정 재작업) → R-001-1 (재리뷰) →
  APPROVED → POLISH
  ```
- **주요 활동**:
  - `c4_request_changes`: 리뷰 태스크 거절 + 다음 버전 생성
  - RPR- 태스크: 수정 재작업 (domain=refine)
  - RF- 태스크: 리파인 루프 내 반복 리뷰
- **전이**: REQUEST_CHANGES=0 달성 → POLISH

### POLISH
- **설명**: `/c4-polish` 스킬로 빌드→테스트→리뷰→수정을 수정사항이 0이 될 때까지 반복
- **진입 조건**: CHECKPOINT approve_final 또는 REFINE 완료 후
- **핵심 개념**:
  - 매 라운드마다 새 리뷰어를 스폰하여 confirmation bias 제거
  - Phase 1: CRITICAL+HIGH=0 품질 게이트 달성 (`refine` gate 기록)
  - Phase 2: 수정사항=0 완전 수렴 달성 (`polish` gate 기록)
- **사용법**:
  ```
  /c4-polish                # 기본 (최대 8라운드, threshold=medium)
  /c4-polish --max-rounds 5 # 최대 5회
  /c4-polish --threshold low # LOW 이슈도 0이 될 때까지
  ```
- **종료 조건**:
  - LOW 이슈만 남은 경우 → 기본 종료 (--threshold low 옵션 시 계속)
  - max-rounds 도달 → 경고 후 종료
- **Gate 기록**: `polish` gate가 기록되어야 `/c4-finish` 진행 가능
- **전이**: 수정사항=0 달성 → COMPLETE (또는 `/c4-finish` 실행)

### HALTED
- **설명**: 사용자가 실행을 일시 중지함
- **진입 조건**: `/c4-stop` 실행
- **허용 작업**: `run`, `plan`, `status`
- **전이**:
  - `c4_run` → EXECUTE
  - `c4_plan` → PLAN

### COMPLETE
- **설명**: 모든 작업 완료
- **진입 조건**: 모든 체크포인트 통과
- **허용 작업**: `status`
- **전이**: 없음 (최종 상태)

### ERROR
- **설명**: 복구 불가능한 오류 발생
- **진입 조건**: `fatal_error` 이벤트
- **허용 작업**: `status`
- **전이**: 없음 (수동 개입 필요)

## REFINE vs POLISH 구분

| 항목 | REFINE | POLISH |
|------|--------|--------|
| **트리거** | R- 태스크의 REQUEST_CHANGES | CHECKPOINT approve_final |
| **목적** | 리뷰어가 지적한 문제 수정 | 전체 품질 수렴 (수정사항=0) |
| **태스크** | RPR- (수정 재작업), RF- (재리뷰) | 내부 라운드 (외부 태스크 없음) |
| **종료 조건** | REQUEST_CHANGES=0 | 수정사항=0 |
| **스킬** | 자동 (c4_request_changes 호출 시) | `/c4-polish` |
| **Gate** | — | `refine` gate + `polish` gate |

## Worker Loop

EXECUTE 상태에서 Worker는 다음 루프를 실행합니다:

```

│                    Worker Loop                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌──────────────┐                                      │
│   │ c4_get_task  │ ◄───────────────────────┐           │
│   └──────┬───────┘                         │           │
│          │                                  │           │
│          v                                  │           │
│   ┌──────────────┐                          │           │
│   │  태스크 있음? │                          │           │

│          │                                  │           │

```

```
│              v                                          │

│   └──────────┬───────────┘                              │
│              │                                          │

│   ┌──────────────────────┐                              │

```

## 체크포인트 게이트 조건

체크포인트는 다음 조건이 모두 만족될 때 트리거됩니다:

1. **required_tasks**: 지정된 모든 태스크가 완료됨
2. **required_validations**: 지정된 모든 검증이 통과함

예시:
```yaml
checkpoints:
  - id: CP-001
    required_tasks:
      - T-001
      - T-002
    required_validations:
      - lint
      - unit
```

## 불변 조건 (Invariants)

C4는 다음 불변 조건을 항상 유지합니다:

1. **CHECKPOINT 중 Worker 실행 금지**: 리뷰 중에는 코드 변경 불가
2. **EXECUTE 중 PLAN 수정 금지**: 계획 수정은 PLAN 상태에서만
3. **상태 즉시 저장**: 모든 전이 후 상태 파일 즉시 저장
4. **POLISH gate 없이 finish 금지**: `polish` gate 기록 전 `/c4-finish` 실행 불가

## 브랜치 전략

C4는 자동으로 Git 브랜치를 관리합니다. 별도 설정 없이 Convention over Configuration 원칙을 따릅니다.

### 브랜치 흐름

```
main (또는 default_branch)
  │
  │  c4_start (PLAN → EXECUTE)
  ▼
c4/{project_id}                    ← 작업 브랜치 (자동 생성)
  │
  ├── c4/w-T-001-0                 ← 태스크 브랜치 (c4_get_task 시 생성)
  ├── c4/w-T-002-0
  └── c4/w-T-003-0
        │
        │  체크포인트 APPROVE
        ▼
c4/{project_id}                    ← 태스크 브랜치들이 작업 브랜치로 머지됨
  │
  │  플랜 완료 (approve_final)
  ▼
main                               ← 작업 브랜치가 main으로 머지됨
```

### 주요 시점

| 시점 | 동작 | 생성되는 브랜치 |
|------|------|----------------|
| `c4_start` | 작업 브랜치 생성 | `c4/{project_id}` |
| `c4_get_task` | 태스크 브랜치 생성 | `c4/w-T-XXX-N` |
| `checkpoint APPROVE` | 완료된 태스크 브랜치 → 작업 브랜치 머지 | - |
| `approve_final` | 작업 브랜치 → main 머지 (또는 PR) | - |

### 설정 옵션

`.c4/config.yaml`에서 브랜치 동작을 커스터마이즈할 수 있습니다:

```yaml
# 기본 브랜치 (기본값: main)
default_branch: main

# 작업 브랜치 (기본값: c4/{project_id})
work_branch: null  # null이면 c4/{project_id} 사용

# 플랜 완료 시 동작 (기본값: merge)
completion_action: merge  # merge | pr | manual
```

### completion_action 옵션

| 값 | 동작 |
|----|------|
| `merge` | 작업 브랜치를 main으로 squash-merge (기본값) |
| `pr` | GitHub PR 생성 (`gh pr create`) |
| `manual` | 아무것도 안 함 (사용자가 수동으로 처리) |

### 예시: PR 생성 모드

```yaml
# .c4/config.yaml
completion_action: pr
```

플랜 완료 시:
1. 작업 브랜치를 origin에 push
2. `gh pr create`로 PR 자동 생성
3. 사용자가 GitHub에서 리뷰 후 머지
