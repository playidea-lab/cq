# C4 사용 시나리오 - 다양한 유저 관점

> **버전**: v0.5.0
> **작성일**: 2026-01-10
> **목적**: 다양한 유저 타입별 C4 사용 시나리오 및 UX 분석

---

## 개요

C4는 **AI 기반 프로젝트 오케스트레이션 시스템**으로, 다양한 유저 타입에게 서로 다른 가치를 제공합니다.

---

## 1. 솔로 개발자 (Solo Developer)

### 페르소나
- 1인 개발자, 사이드 프로젝트 또는 소규모 스타트업
- Claude Code를 주로 사용
- 빠른 프로토타이핑과 배포가 목표

### 사용 시나리오

**시나리오 A: 신규 SaaS 프로젝트 시작**

```bash
# 1. 프로젝트 생성 및 C4 초기화
mkdir my-saas && cd my-saas
c4  # 또는 claude code 내에서 /c4-init

# 2. 요구사항 정의 (Discovery)
/plan
# → Claude가 "어떤 서비스를 만드시나요?" 질문
# → "사용자 인증 + 대시보드가 있는 구독 서비스"
# → EARS 패턴으로 요구사항 자동 수집

# 3. 설계 (Design)
# → 아키텍처 옵션 제시 (JWT vs Session, React vs Vue)
# → 사용자가 선택하면 ADR로 기록

# 4. 실행 (Execute)
/run
# → Claude가 자동으로 태스크 순차 실행
# → 커피 마시고 돌아오면 완료
```

**장점**:
- 빠른 시작: 한 줄 명령어로 전체 워크플로우
- 자동화: 코드 작성, 테스트, 리뷰까지 Claude가 처리
- 비동기: `/run` 후 다른 일 가능

**단점**:
- 대기 시간: 복잡한 프로젝트는 실행에 시간 소요
- 커스터마이징 어려움: 세밀한 제어 필요시 직접 개입

## 2. 팀 리드 / 테크 리드 (Tech Lead)

### 페르소나
- 3-10명 팀 관리
- 코드 리뷰 병목
- 일관된 코드 품질 유지가 목표

### 사용 시나리오

**시나리오 B: 스프린트 태스크 분배**

```
# 1. 스프린트 계획 (PLAN 상태에서)
사용자: /add-task
Claude: 어떤 작업을 추가하시겠습니까?
사용자: 로그인 페이지 만들어줘
Claude: DoD 제안: JWT 인증, 소셜 로그인... 괜찮을까요?
사용자: 응
Claude: ✅ T-001 추가 (web-frontend, src/auth/)

# 반복해서 태스크 추가
/add-task OAuth2 백엔드
/add-task Terraform EKS 설정

# 2. 다중 워커 실행
# 터미널 1: claude code → /run (frontend 담당)
# 터미널 2: claude code → /run (backend 담당)
# 터미널 3: claude code → /run (infra 담당)

# 3. 상태 모니터링
/status
# → 각 워커의 진행 상황, 완료된 태스크, 대기 중 태스크 확인
```

**시나리오 C: 자동 코드 리뷰 설정**

```yaml
# .c4/config.yaml
checkpoints:
  - id: "CP-PR"
    description: "PR 전 자동 리뷰"
    required_tasks: ["T-*"]  # 모든 태스크
    required_validations: [lint, unit, security]

verifications:
  items:
    - type: cli
      name: "보안 스캔"
      config:
        command: "bandit -r src/"
```

**장점**:
- 병렬 작업: 여러 Claude가 동시에 다른 스코프 작업
- 자동 리뷰: Checkpoint에서 AI가 코드 리뷰
- 가시성: 팀 전체 진행 상황 한눈에

**단점**:
- 초기 설정: 체크포인트, 검증 설정에 시간 투자 필요
- 머지 충돌: 같은 파일 수정 시 수동 해결 필요

## 3. ML/AI 엔지니어 (ML Engineer)

### 페르소나
- 모델 학습/배포 담당
- 실험 재현성이 중요
- Jupyter + 프로덕션 코드 혼재

### 사용 시나리오

**시나리오 D: 모델 학습 파이프라인 구축**

```bash
# 1. ML 도메인으로 초기화
c4
/plan
# → 도메인 자동 감지: "ml-dl" (PyTorch 파일 발견)
# → ml-engineer 에이전트 자동 선택

# 2. 검증 설정 (Metrics Verifier 활용)
verifications:
  items:
    - type: metrics
      name: "모델 성능"
      config:
        metrics_file: "experiments/latest/metrics.json"
        thresholds:
          accuracy: {"min": 0.90}
          latency_ms: {"max": 100}

# 3. 실행
/run
# → ml-engineer → python-pro → test-automator 체인 실행
# → 학습 완료 시 자동으로 성능 검증
```

**시나리오 E: A/B 테스트 배포**

```
# 태스크에 검증 포함 (대화형)
사용자: /add-task Model B 배포
Claude: ML 모델 배포군요. DoD를 제안해볼게요:
  - accuracy >= 0.92
  - latency < 50ms
  - A/B 테스트 완료
  검증 방식은 metrics + http로 할까요?
사용자: 응

# Checkpoint에서 성능 비교
# → Supervisor가 Model A vs B 비교
# → APPROVE 또는 REQUEST_CHANGES
```

**장점**:
- 자동 성능 검증: 배포 전 메트릭 자동 체크
- 재현성: 실험 설정이 코드로 관리
- 실험 관리: 여러 실험 병렬 실행

**단점**:
- GPU 리소스: 멀티워커 시 GPU 충돌 주의
- 대용량 데이터: 데이터 동기화 별도 관리

## 4. DevOps / SRE 엔지니어

### 페르소나
- 인프라 자동화 담당
- 무중단 배포, 장애 복구
- Terraform, Kubernetes 사용

### 사용 시나리오

**시나리오 F: 인프라 변경 자동화**

```bash
# 1. infra 도메인 설정
domain: "infra"

# 2. Dryrun Verifier로 안전 검증
verifications:
  items:
    - type: dryrun
      name: "Terraform Plan"
      config:
        command: "terraform plan -no-color"
        success_patterns: ["No changes", "Plan:"]
        failure_patterns: ["Error:"]
    - type: cli
      name: "Kubectl Dry-run"
      config:
        command: "kubectl apply --dry-run=client -f k8s/"

# 3. 체크포인트에서 human approval
checkpoints:
  - id: "CP-PROD"
    description: "프로덕션 배포 승인"
    require_human_approval: true  # AI 대신 사람이 승인
```

**시나리오 G: 장애 대응 자동화**

```
# 1. 장애 감지 시 태스크 생성
사용자: /add-task DB 연결 장애 복구
Claude: 긴급 장애 복구군요. 도메인은 infra로 설정할게요.
  DoD 제안:
  - DB 연결 정상화
  - 헬스체크 통과
  - 장애 원인 분석 로그 작성
사용자: 확인

# 2. incident-responder 에이전트 자동 선택

# 3. 복구 후 검증
verifications:
  - type: http
    name: "서비스 헬스체크"
    config:
      url: "https://api.example.com/health"
      expected_status: 200
      timeout: 10
```

**장점**:
- 안전 장치: Dry-run 필수, 프로덕션 변경 전 검증
- 승인 워크플로우: 중요 변경은 사람 승인
- 감사 로그: 모든 변경 이벤트 기록

**단점**:
- 권한 관리: Claude에 인프라 접근 권한 설정 필요
- 네트워크: 에어갭 환경에서 제한적

## 5. 풀스택 프리랜서 (Freelancer)

### 페르소나
- 클라이언트 프로젝트 수주
- 빠른 납품, 다양한 기술 스택
- 문서화 시간 부족

### 사용 시나리오

**시나리오 H: 클라이언트 프로젝트 빠른 시작**

```bash
# 1. 클라이언트 요구사항 → EARS 변환
/plan
# 클라이언트: "쇼핑몰인데 결제랑 재고 관리 필요해요"

# C4가 자동 변환:
# - Ubiquitous: "The system shall process payments via Stripe"
# - Event-driven: "When stock < 5, the system shall send alert"
# - Unwanted: "The system shall not store card numbers"

# 2. fullstack 도메인으로 자동 설정
# → backend-architect → frontend-developer → test 체인

# 3. 한 번에 실행
/run
# → 백엔드 API, 프론트엔드 UI, 테스트 코드 자동 생성
```

**시나리오 I: 레거시 프로젝트 인수**

```bash
# 1. 기존 코드베이스에서 시작
cd client-legacy-project
c4

# 2. Discovery에서 도메인 자동 감지
# → "Django + jQuery 발견, web-backend로 분류"

# 3. 점진적 마이그레이션 계획
/plan
# "React로 프론트엔드 마이그레이션"
# → 기존 코드 분석 → 마이그레이션 태스크 자동 생성
```

**장점**:
- 요구사항 관리: 클라이언트 말을 EARS로 정형화
- 멀티 스택: fullstack 도메인이 프론트/백 모두 처리
- 자동 문서화: ADR로 설계 결정 기록

**단점**:
- 커뮤니케이션: 클라이언트에게 C4 개념 설명 필요
- 정밀 제어: 클라이언트 피드백 반영 시 수동 개입

## 6. 엔터프라이즈 개발자 (Enterprise)

### 페르소나
- 대규모 조직, 엄격한 프로세스
- 규정 준수, 보안 감사
- 여러 팀 협업

### 사용 시나리오

**시나리오 J: 규정 준수 자동화**

```yaml
# .c4/config.yaml
checkpoints:
  - id: "CP-SECURITY"
    description: "보안 검토"
    required_validations:
      - lint
      - security-scan
      - dependency-audit
      - license-check

verifications:
  items:
    - type: cli
      name: "OWASP 스캔"
      config:
        command: "npm audit --production"
        expected_exit_code: 0
    - type: cli
      name: "라이센스 검사"
      config:
        command: "license-checker --onlyAllow 'MIT;Apache-2.0;BSD'"
```

**시나리오 K: 다중 팀 협업 (향후 Phase 6)**

```
# 현재 (v0.5.0)
팀 A (터미널 1): /run  → 같은 프로젝트, 다른 scope
팀 B (터미널 2): /run  → SQLite 기반 동기화

# 향후 (v0.6.0 Team Collaboration)
팀 A (서울):  → Supabase StateStore
팀 B (미국):  → 실시간 동기화
대시보드:     → 전체 진행 상황 시각화
```

**장점**:
- 규정 준수: 체크포인트에서 자동 검사
- 감사 추적: 모든 이벤트 로깅
- 권한 분리: scope lock으로 충돌 방지

**단점**:
- 온프레미스: 클라우드 없이 로컬 SQLite만 지원 (현재)
- SSO 통합: 엔터프라이즈 인증 미지원 (현재)

## 7. 사용 시나리오 요약

| 유저 타입 | 핵심 가치 | 주요 기능 | 현재 지원 |
|----------|----------|----------|----------|
| 솔로 개발자 | 빠른 시작, 자동화 | /run 한 줄 실행 | O |
| 팀 리드 | 병렬 작업, 품질 관리 | 멀티워커, 체크포인트 | O |
| ML 엔지니어 | 성능 검증, 재현성 | Metrics Verifier | O |
| DevOps/SRE | 안전 배포, 승인 워크플로우 | Dryrun Verifier | O |
| 프리랜서 | 요구사항 정형화, 빠른 납품 | EARS, ADR | O |
| 엔터프라이즈 | 규정 준수, 감사 | 검증 체인, 이벤트 로그 | O |

## 8. 현재 제약 사항

| 제약 | 설명 | 해결 예정 |
|------|------|----------|
| **로컬 전용** | SQLite 기반, 원격 동기화 없음 | Phase 6 (Supabase) |
| **단일 프로젝트** | 프로젝트 간 공유 없음 | Phase 7 (Cloud) |
| **Claude 의존** | Claude Code 필수 | 향후 다른 LLM 지원 |
| **수동 설치** | 원격 설치 스크립트 필요 | 개선 예정 |
