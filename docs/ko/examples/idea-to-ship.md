# 예시: 아이디어에서 배포까지

막연한 아이디어에서 커밋된 코드까지 — 중간에 명령어 하나 안 치고.

::: info solo 티어
이 예시는 `solo` 티어에서 동작합니다. 클라우드 설정이 필요 없습니다.
:::

## 시나리오

대충 아이디어가 있습니다: "헬스체크 엔드포인트 추가." `/c4-plan`, `/c4-run`, `/c4-finish`를 하나씩 실행하고 싶지 않습니다. 아이디어만 말하고 CQ가 알아서 하길 원합니다.

## Step 1 — 아이디어 설명

```sh
cq claude
```

> **You:** "헬스체크 엔드포인트 추가하고 싶어. DB 연결이랑 디스크 용량도 체크하면 좋겠는데"

CQ의 `/pi`가 활성화됩니다. 몇 가지 질문 후 수렴합니다:

```
💡 아이디어 모드

헬스체크 엔드포인트. 몇 가지 확인:
- 컴포넌트별 상세 상태? 아니면 OK/FAIL만?
- 공개 엔드포인트? 인증 필요?
- 응답 포맷? (JSON, plain text)

> JSON, 공개, 컴포넌트별 상세

📋 idea.md 생성: .c4/ideas/health-check-endpoint.md
```

## Step 2 — "자동" 선택

CQ가 다음 단계를 물어봅니다:

```
다음 단계:
  (1) 자동 구현 — plan → run → finish, 전체 자동
  (2) 계획만 — 태스크만 생성, 실행 전 검토

> 1
```

끝입니다. 더 이상 입력할 게 없습니다.

## 이후 진행 (사람 개입 없음)

```
🔗 파이프라인: plan → run → finish (자동)

━━ /c4-plan ━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ● Discovery: idea.md에서 EARS 요구사항 파싱
  ● Design: 단일 핸들러 + checker 인터페이스
  ● Tasks:
    T-001  헬스체크 핸들러 (/healthz)
    T-002  DB 연결 체커
    T-003  디스크 용량 체커

━━ /c4-run ━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ◆ T-001  [worker-a]  구현 중...  ✓ 완료
  ◆ T-002  [worker-b]  구현 중...  ✓ 완료
  ◆ T-003  [worker-c]  구현 중...  ✓ 완료
  ✓ R-001  리뷰 통과
  ✓ R-002  리뷰 통과
  ✓ R-003  리뷰 통과

━━ /c4-finish ━━━━━━━━━━━━━━━━━━━━━━━━━
  ● Polish: 1라운드 → 수정 0건 (수렴)
  ● Build:  go build ./... ✓
  ● Tests:  312개 통과, 0 실패
  ● Commit: feat(health): add /healthz with DB and disk checks
```

돌아오면 커밋된 기능이 레포에 있습니다. 중간에 "진행할까요?" 같은 질문 없이.

## 파이프라인 상세

```
"자동" 선택
  ↓
/pi가 .c4/pipeline-state.json 생성 (auto=true)
  ↓
/c4-plan이 액션 선택 skip (FROM_PI), 에디터 skip (AUTO_RUN)
  ↓
/c4-run이 격리된 worktree에서 워커 병렬 실행
  ↓
/c4-finish가 polish 루프 → 빌드 → 테스트 → 커밋
  ↓
pipeline-state.json 삭제 — 파이프라인 완료
```

체인은 간단한 상태 파일로 구동됩니다. 각 스킬이 `auto=true`를 확인하고 묻지 않고 진행합니다. 실패하면 에이전트가 수정합니다 — 파이프라인은 개입하지 않습니다.

## 이럴 때 사용

- 아이디어는 명확한데 과정을 감시하고 싶지 않을 때
- 소~중규모 기능 (1~5개 태스크)
- CQ의 계획을 신뢰하고, 과정이 아닌 결과를 검토하고 싶을 때

## 이럴 때는 사용하지 마세요

- 대규모 기능에서 계획을 먼저 검토하고 싶을 때
- 설계가 불명확한 탐색적 작업
- 중요 시스템 변경에서 각 단계를 승인하고 싶을 때

그런 경우에는 `/c4-plan`을 수동으로 실행하고 검토 후 진행하세요.
