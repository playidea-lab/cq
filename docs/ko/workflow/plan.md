# /c4-plan

트리거: `/c4-plan "기능 설명"` 또는 키워드: `계획`, `plan`, `설계`

## 계획 전 단계: /pi

아이디어를 탐색 중이라면 `/c4-plan` 전에 `/pi`(Play Idea)를 사용하세요:

```
/pi
```

`/pi`는 아이디어 탐색 모드입니다 — 온라인 리서치, 트레이드오프 토론, 개념 구체화. 준비되면 `idea.md`를 생성하고 자동으로 `/c4-plan`을 호출합니다.

---

## 동작 방식

구조화된 계획 프로세스를 실행합니다:

1. **Discovery** — EARS 형식으로 요구사항을 명확히 합니다. 범위가 명확해질 때까지 질문합니다.
2. **Design** — 트레이드오프가 있는 아키텍처와 핵심 결정 사항을 제안합니다.
3. **Lighthouse** — 도구 계약을 등록합니다 (태스크당 DoD 체크리스트). Lighthouse는 C4의 계약 우선 TDD 레이어입니다: 코드 작성 전 각 태스크에 검증 가능한 완료 조건을 부여합니다.
4. **Tasks** — 태스크당 완료 조건(DoD)이 있는 태스크 큐를 작성합니다.
5. **Plan Critique Loop** — 매 라운드마다 새로운 리뷰어를 스폰하여 스펙, DoD, 태스크 설계를 검증합니다. CRITICAL 또는 HIGH 이슈가 없을 때까지 계속합니다. 그래서 별도의 `/c4-refine` 단계가 필요 없습니다.
6. **Commit** — 검증된 태스크를 `.c4/tasks.db`에 저장합니다.

## 예시

```
/c4-plan "Google과 GitHub OAuth2 로그인 추가"
```

CQ가 명확화 질문을 한 뒤 다음과 같은 태스크를 생성합니다:
- T-001: OAuth2 provider 추상화 구현
- T-002: Google provider 추가
- T-003: GitHub provider 추가
- T-004: session store와 통합

## 태스크 완료 조건 (DoD)

각 태스크에는 DoD 체크리스트가 있습니다:

```
- [ ] 유닛 테스트 통과
- [ ] mock provider로 통합 테스트
- [ ] 토큰 만료 에러 처리
- [ ] handoff에 근거 문서화
```

## 계획 이후

`/c4-run`으로 워커를 시작합니다. `/c4-run`이 나머지를 처리합니다 — 구현, polish, finish.
