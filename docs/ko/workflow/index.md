# 워크플로우 개요

CQ는 구조화된 루프를 따릅니다:

```
아이디어 (/pi) → INIT → DISCOVERY → DESIGN → PLAN → EXECUTE → CHECKPOINT → COMPLETE
```

## 명령어

| 명령 | 단계 | 사용 시기 |
|------|------|---------|
| [`/pi`](/workflow/plan#pi) | Ideation | 계획 전 아이디어 발산·수렴. 자동으로 `/c4-plan` 전환. |
| [`/c4-plan`](/ko/workflow/plan) | Plan | 새 기능 또는 중요한 변경 |
| [`/c4-run`](/ko/workflow/run) | Execute | 워커를 스폰하고 end-to-end 실행 |
| [`/c4-finish`](/ko/workflow/finish) | Complete | 빌드, 테스트, 문서, 커밋 — `/c4-run`이 자동 호출, 추가 변경 후 수동 실행도 가능 |
| `/c4-status` | 언제든지 | 언제든지 진행 상황 확인 |
| `/c4-quick` | — | 소규모 단일 태스크, 계획 단계 생략 |

## 태스크 라이프사이클

모든 태스크가 거치는 과정:

```
pending → in_progress → (review) → completed
                      ↘ request_changes → revision → ...
```

- **T-001-0** — 구현 태스크 (버전 0)
- **R-001-0** — 각 구현 완료 후 자동 생성되는 리뷰 태스크
- **T-001-1** — 리뷰에서 변경 요청 시 생성되는 수정 태스크
- **CP-001** — 단계 완료 시 자동 생성되는 체크포인트

체크포인트는 자동으로 생성됩니다 — 수동으로 트리거하지 않습니다. 체크포인트가 나타나면 CQ가 일시 중지하고 다음 단계로 넘어가기 전에 단계 결과를 검토하도록 요청합니다.

## 워커 격리

각 워커는 자체 git 워크트리(`c4/w-T-XXX-N`)에서 실행되므로 병렬 워커가 서로 충돌하지 않습니다. 워크트리는 `c4-finish` 이후 자동으로 정리됩니다.

## 지식 루프

완료된 모든 태스크에서 발견 사항과 근거가 기록됩니다. 미래 태스크는 관련 과거 지식을 컨텍스트에 자동으로 주입받습니다.
