# /c4-run

트리거: `/c4-run` 또는 키워드: `실행`, `run`, `ㄱㄱ`

## 동작 방식

준비된 태스크당 하나의 워커를 스폰합니다. 워커는 병렬로 실행되며 각자 격리된 git 워크트리에서 작업합니다.

```
/c4-run
→ pending 태스크 탐색 (의존성 해결됨)
→ T-001용 Worker 1 스폰 (워크트리: c4/w-T-001-0)
→ T-002용 Worker 2 스폰 (워크트리: c4/w-T-002-0)
→ 각 워커: 태스크 읽기 → 구현 → 테스트 → 제출
→ 리뷰 태스크 (R-001, R-002) 자동 생성
→ 각 리뷰 워커 스폰
→ 큐가 빌 때까지 respawn 반복
→ /c4-finish: polish (변경 없을 때까지 수렴) → 빌드 · 테스트 · 문서 · 커밋
```

## 워커 라이프사이클

각 워커:
1. `c4_get_task` 호출 → DoD와 지식 컨텍스트가 있는 태스크 수신
2. 할당된 워크트리에서 구현
3. 검증 실행 (lint + 테스트)
4. commit SHA와 handoff 요약을 포함하여 `c4_submit` 호출
5. 리뷰 태스크 자동 생성

## 모니터링

```
/c4-status    → 의존성 그래프가 포함된 시각적 진행 상황
```

## 리뷰에서 변경 요청 시

수정 태스크 (T-001-1)가 자동으로 생성됩니다. `/c4-run`이 다음 사이클에서 이를 처리합니다.

## 연속 모드

`/c4-run`은 모든 태스크가 완료될 때까지 계속 respawn합니다 — 수동으로 다시 실행할 필요가 없습니다.

## 실행 이후

모든 태스크가 완료되면 `/c4-run`이 자동으로 `/c4-finish`를 호출합니다 — polish 루프(변경 없을 때까지 build-test-review-fix 수렴)가 `/c4-finish`에 내장되어 있습니다. 추가 단계 불필요.

추가 수동 변경을 했다면 `/c4-finish`로 마무리합니다.
