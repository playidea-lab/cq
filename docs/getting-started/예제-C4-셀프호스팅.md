# 예제: C4로 C4 개발하기 (셀프호스팅)

이 튜토리얼은 C4 프로젝트 자체를 C4 워크플로우로 개발하는 과정을 보여줍니다.
실제 프로젝트에서 C4를 어떻게 사용하는지 이해하는 데 도움이 됩니다.

## 시나리오

C4에 새로운 기능 "이벤트 익스포터"를 추가합니다:
- `.c4/events/` 디렉토리의 이벤트를 JSON으로 익스포트
- 테스트 작성
- 문서 업데이트

## Step 1: 프로젝트 상태 확인

C4 프로젝트 디렉토리에서 Claude Code를 열고 상태를 확인합니다:

```
/c4-status
```

이미 초기화되어 있다면:
```
## C4 Status
- **Project**: c4
- **Status**: PLAN
- **Queue**: 0 pending, 0 in_progress, 15 done
```

초기화가 안 되어 있다면 `/c4-init` 실행.

## Step 2: 계획 수립

Claude Code에서 `/c4-plan` 스킬을 실행합니다:

```
/c4-plan "이벤트 익스포터 추가 — .c4/events/를 JSON으로 익스포트, 테스트, 문서화"
```

C4가 대화형으로 Discovery → Design → Task 생성을 안내합니다:

```
[DISCOVERY] 도메인 감지: Go Backend
요구사항을 EARS 패턴으로 수집 중...
  - When user calls export-events, system shall write .c4/events/*.json
  - If export path already exists, system shall overwrite with confirmation
  ...

[DESIGN] 아키텍처 옵션 제안:
  Option A: Go native handler (c4-core) — 권장
  Option B: Python Sidecar 확장

[TASKS] 태스크 생성:
  T-EXP-01: EventExporter 핸들러 구현 (c4-core/)
  T-EXP-02: 단위 테스트 작성 (tests/)
  T-EXP-03: 문서 업데이트 (docs/)
  CP-EXP: 이벤트 익스포터 체크포인트 (requires T-EXP-01..03)
```

## Step 3: 태스크 확인

```
/c4-status
```

태스크가 큐에 등록된 것을 확인합니다:
```
State: PLAN
Queue: 3 pending | 0 in_progress | 0 done
Tasks:
  T-EXP-01 [pending] EventExporter 핸들러 구현
  T-EXP-02 [pending, blocked by T-EXP-01] 단위 테스트 작성
  T-EXP-03 [pending, blocked by T-EXP-01] 문서 업데이트
  CP-EXP   [checkpoint]
```

## Step 4: 실행 시작

```
/c4-run
```

상태가 PLAN → EXECUTE로 전환됩니다.

## Step 5: 워커 실행

```
/c4-run
```

C4가 준비된 태스크 수만큼 Worker를 백그라운드로 스폰합니다:

```
🔄 Worker Loop 시작 (worker-abc123)

📋 태스크 수신: T-EXP-01 - EventExporter 클래스 구현
   범위: c4/

🔧 구현 중...
   - c4/event_exporter.py 생성
   - EventExporter 클래스 구현
   - export_all(), export_by_type() 메서드 추가

✅ 검증 실행 중...
   - lint: pass
   - unit: pass (해당 없음, 아직 테스트 없음)

📤 제출: T-EXP-01 완료
```

## Step 6: 다음 태스크 자동 할당

Worker가 계속해서 다음 태스크를 처리합니다:

```
📋 태스크 수신: T-EXP-02 - EventExporter 테스트
   범위: tests/
   의존성: T-EXP-01 ✅

🔧 구현 중...
   - tests/unit/test_event_exporter.py 생성
   - test_export_all 작성
   - test_export_by_type 작성

✅ 검증 실행 중...
   - lint: pass
   - unit: pass (12 tests, 0.3s)

📤 제출: T-EXP-02 완료
```

```
📋 태스크 수신: T-EXP-03 - 문서 업데이트
   범위: docs/

🔧 구현 중...
   - README.md 업데이트
   - docs/user-guide/이벤트-익스포트.md 생성

✅ 검증 실행 중...
   - lint: pass
   - unit: pass

📤 제출: T-EXP-03 완료

🎯 체크포인트 도달: CP-EXP
   모든 태스크 완료, 검증 통과
```

## Step 7: Checkpoint 리뷰

자동으로 EXECUTE → CHECKPOINT 상태로 전환됩니다:

```
/c4-checkpoint
```

Supervisor Loop이 실행되어 변경사항을 리뷰합니다:

```
🔍 Checkpoint 리뷰: CP-EXP

📦 변경사항 요약:
   - c4/event_exporter.py (새 파일, 150줄)
   - tests/unit/test_event_exporter.py (새 파일, 80줄)
   - README.md (수정, +15줄)
   - docs/user-guide/이벤트-익스포트.md (새 파일, 50줄)

✅ 검증 결과:
   - lint: pass
   - unit: pass (222 tests, 4.5s)

🤔 리뷰 중...
```

### 리뷰 결과 예시

**Case 1: APPROVE**
```
✅ 승인됨

코멘트:
- EventExporter 구현이 깔끔합니다
- 테스트 커버리지 충분
- 문서화 잘 되어 있음

→ CHECKPOINT → EXECUTE (또는 COMPLETE)
```

**Case 2: REQUEST_CHANGES**
```
🔄 수정 요청

필요한 변경:
1. export_by_date_range() 메서드 추가 필요
2. 대용량 이벤트 처리 시 메모리 최적화 필요

새 태스크 추가됨:
- T-EXP-04: export_by_date_range 구현
- T-EXP-05: 스트리밍 익스포트 구현

→ CHECKPOINT → EXECUTE
```

## Step 8: 완료

모든 체크포인트를 통과하면 COMPLETE 상태가 됩니다:

```
/c4-status

## C4 Status
- **Status**: COMPLETE
- **Queue**: 0 pending, 0 in_progress, 18 done
- **Passed Checkpoints**: CP-EXP
```

## 핵심 포인트

### Worker Loop의 자동화
- 태스크 할당 → 구현 → 검증 → 제출이 자동으로 반복
- 의존성이 있는 태스크는 선행 태스크 완료 후 할당

### Scope 기반 잠금
- 같은 범위의 태스크는 동시에 하나만 실행
- 예: `c4/` 범위 태스크 실행 중에는 다른 Worker가 같은 범위 태스크 불가

### Checkpoint에서의 Supervisor 리뷰
- 모든 조건 만족 시 자동으로 CHECKPOINT 진입
- Supervisor가 전체 변경사항을 검토
- APPROVE/REQUEST_CHANGES/REPLAN 결정
