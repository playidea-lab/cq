# C4 팀 온보딩 가이드

C4 프로젝트에 참여하는 개발자, PM, QA 팀원들이 빠르게 시작할 수 있도록 하는 가이드입니다.

---

## 파트 1: 개발자 첫 설정 (30분)

### 1단계: 사전 요구사항 확인

터미널에서 다음 명령어를 실행하여 필수 도구를 확인합니다:

```bash
# Go 버전 확인 (1.22+ 필요)
go version

# Python 버전 확인 (3.11+ 필요)
python3 --version

# uv 설치 확인 (없으면 설치)
uv --version || curl -LsSf https://astral.sh/uv/install.sh | sh
```

- [ ] Go 1.22 이상 설치
- [ ] Python 3.11 이상 설치
- [ ] uv 설치 완료

### 2단계: 저장소 클론

```bash
git clone https://git.pilab.co.kr/pi/c4.git
cd c4
```

- [ ] 저장소 클론 완료

### 3단계: 자동 설치 실행

```bash
./install.sh
```

설치 스크립트가 자동으로 처리하는 항목:
- Go 바이너리 빌드 (`c4-core/bin/c4`)
- Python 의존성 설치 (`uv sync`)
- `.mcp.json` 설정 (Claude Code용)
- `.c4/` 초기화

- [ ] install.sh 실행 완료
- [ ] 에러 메시지 확인 (있으면 처리)

### 4단계: 환경 변수 설정

```bash
# 템플릿 복사
cp .env.example .env

# 에디터로 열기
vi .env
```

**최소 필수 설정** (로컬 개발):
```bash
C4_STORE_BACKEND=sqlite
# Supabase는 나중에 팀 서버에 연결할 때만 필요
```

- [ ] `.env` 파일 생성 및 설정

### 5단계: Claude Code 재시작

```bash
# 설치 확인 (112개 도구가 로드되어야 함)
c4-status
```

Claude Code 확장에서 `.mcp.json`을 다시 로드합니다:
1. Claude Code 사이드바 열기
2. "MCP Servers" 섹션 확인
3. `C4 MCP Server` 연결 상태 확인 (초록색)

- [ ] C4 MCP 서버 연결 확인
- [ ] `/c4-status` 명령어 실행 성공

## 파트 2: 프로젝트에 참여하기

### 프로젝트 초기화

새 프로젝트를 시작할 때:

```bash
c4 init my-project
```

이 명령어가 생성하는 구조:
```
my-project/
├── .c4/                    # C4 상태 저장
│   ├── tasks.db           # 로컬 태스크 스토어
│   ├── state.yaml         # 현재 상태 (INIT → DISCOVERY → ... → COMPLETE)
│   └── config.yaml        # 프로젝트별 설정 (LLM 모델, 경제 프리셋 등)
├── .claude/               # Claude Code 프로필 (공유)
│   └── c4.yaml           # 워크플로우 선호도
└── README.md              # 프로젝트 설명
```

- [ ] 프로젝트 `.c4/` 구조 이해

### .c4 디렉토리의 역할

| 파일 | 용도 |
|------|------|
| `tasks.db` | SQLite 데이터베이스 — 모든 작업(T-001, R-001 등) 저장 |
| `state.yaml` | 프로젝트 상태 머신 (INIT/DISCOVERY/DESIGN/PLAN/EXECUTE/CHECKPOINT/COMPLETE) |
| `config.yaml` | 프로젝트 설정 (LLM 선택, economic preset, 기본 reviewer 등) |

## 파트 3: 역할별 워크플로우

### CEO / PM 워크플로우

**목표**: 기능을 계획하고 진행 상황을 모니터링

```bash
# 1. 기능 기획
/c4-plan "사용자 인증 기능 추가 — JWT 사용"

# C4가 자동으로:
# - 요구사항 추출 및 문서화 (spec)
# - 설계 결정 기록 (design)
# - 구현 태스크 생성 (T-001, T-002, ...)
# - 검증 전략 수립

# 2. 진행 상황 모니터링
/c4-status

# 출력 예시:
# ▶ State: PLAN
# ▶ Tasks: 5/5 (완료: 3, 진행중: 1, 대기: 1)
# ▶ Next: T-004 (authentication model) 진행 중
# ▶ Checkpoint: CP-001 (설계 리뷰) 대기 중

# 3. 결정점에서 승인/거절
/c4-checkpoint-approve CP-001
# 또는
/c4-checkpoint-reject CP-001 "DB 스키마 변경 필요"
```

**체크리스트**:
- [ ] 기능을 명확한 한 문장으로 표현
- [ ] Definition of Done(DoD) 이해
- [ ] 주간 진행 상황 체크 습관화

### Developer 워크플로우

**목표**: 태스크를 구현하고 자동 검증을 통과하기

#### Worker 모드 (병렬 작업, 권장)

```bash
# 1. 태스크 할당받기
/c4-run

# C4가:
# - 대기 중인 태스크(T-001, T-002, ...)를 병렬로 할당
# - 각각 별도 worktree에서 실행
# - 자동으로 검증(lint, test) 수행

# 2. 진행 상황 확인
/c4-status

# 3. 리뷰 피드백 처리
/c4-request-changes T-001-0 "에러 처리 추가 필요"
# → 자동으로 T-001-1 버전 생성, 새로운 리뷰 R-001-1 생성

# 4. 완료 시 병합
# C4가 자동으로 처리 (모든 검증 통과 시)
```

#### Direct 모드 (순차 작업, 파일 의존성 높을 때)

```bash
# 설정 예시 (config.yaml)
c4_add_todo \
  --task_id T-001 \
  --title "데이터베이스 마이그레이션" \
  --execution_mode direct \
  --dependencies "T-000"  # 선행 작업 있을 때

# 워커 시작
c4_get_task my-worker-1
# → T-001 할당 (T-000이 완료되어야 함)

# 직접 작업
c4_claim T-001
# ... 코드 작성/수정 ...
c4_report T-001 "마이그레이션 완료" --files_changed "db/migrations/001_*.sql"
```

**체크리스트**:
- [ ] Worker 모드로 병렬 작업 경험해보기
- [ ] 검증 실패 시 로그 읽기 능력 개발
- [ ] 리뷰 피드백에 신속하게 대응

### QA / 검증자 워크플로우

**목표**: 태스크 완료 시 검증 및 리뷰

```bash
# 1. 검증할 태스크 확인
/c4-status

# 2. 자동 검증 결과 확인
# - 단위 테스트 (pytest/go test)
# - Linting (ruff/go vet)
# - 타입 검사 (mypy)
# - Integration 테스트 (선택)

# 3. 수동 검증 항목
# - 요구사항 충족 여부 (DoD 체크)
# - 사용자 스토리 검증
# - 엣지 케이스 테스트

# 4. 리뷰 의견 제출
/c4-checkpoint-approve CP-001  # 모든 기준 만족
# 또는
/c4-request-changes R-001 "다음 항목을 수정해주세요: 1. 에러 메시지 추가 2. 로깅 개선"

# 5. 리뷰 기록 확인
/c4-status
```

**체크리스트**:
- [ ] DoD(Definition of Done) 숙지
- [ ] 자동 검증 결과 해석 능력
- [ ] 명확한 피드백 작성 연습

## 파트 5: 자주 묻는 질문 (FAQ)

### Q1: "c4 command not found" 에러가 나요

**해결책**:
```bash
# 1. 재설치
cd c4 && ./install.sh

# 2. PATH 확인
echo $PATH | grep ".local/bin"

# 3. 수동 설정 (필요 시)
export PATH="$HOME/.local/bin:$PATH"
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
```

### Q2: 검증(pytest/go test)이 실패했어요

**디버깅 단계**:
```bash
# 1. 로컬에서 테스트 실행
cd c4-core && go test ./...
cd c4 && uv run pytest tests/

# 2. 실패한 테스트만 실행
go test -run TestName ./...
uv run pytest tests/unit/test_module.py::test_name -v

# 3. C4 로그 확인
c4 logs T-001-0  # 특정 태스크 로그
```

### Q3: Supabase에 연결하고 싶어요

**단계**:
1. 팀 리더에게 Supabase 프로젝트 URL과 API 키 요청
2. `.env` 파일에 추가:
   ```bash
   C4_SUPABASE_URL=https://your-project.supabase.co
   C4_SUPABASE_KEY=eyJ...
   ```
3. 재시작:
   ```bash
   source ~/.zshrc  # 또는 ~/.bashrc
   c4 status
   ```

### Q4: C5 Hub 서버와 연결하려면?

**단계**:
1. 팀 인프라 팀에게 Hub 서버 URL과 API 키 요청
2. `.env` 파일:
   ```bash
   C5_HUB_URL=http://hub-server.example.com:8585
   C5_API_KEY=your-api-key
   ```
3. 테스트:
   ```bash
   c4_hub_stats
   ```

### Q5: Claude Code 재시작 후에도 도구가 안 보여요

**해결책**:
1. Claude Code 개발자 도구 열기 (F12)
2. "MCP Servers" 로그 확인
3. 에러 메시지 캡처 후 팀 리더에게 보고
4. 다시 설치:
   ```bash
   cd c4 && ./install.sh --force
   ```
