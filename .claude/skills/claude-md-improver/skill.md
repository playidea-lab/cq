# CLAUDE.md Improver

프로젝트의 CLAUDE.md를 분석하고 개선하는 가이드.

## 트리거

"CLAUDE.md 개선", "claude md improver", "에이전트 지침 개선", "CLAUDE.md 리뷰"

## Steps

### 1. 현재 CLAUDE.md 분석

다음 항목을 점검:

- [ ] **프로젝트 컨텍스트**: 프로젝트가 무엇인지, 기술 스택이 명시되어 있는가?
- [ ] **빌드/실행 명령**: 개발자가 바로 시작할 수 있는 명령이 있는가?
- [ ] **테스트 명령**: 테스트 실행 방법이 명시되어 있는가?
- [ ] **코드 컨벤션**: 프로젝트 고유의 패턴/규칙이 있는가?
- [ ] **금지 사항**: 하지 말아야 할 것이 명시되어 있는가?
- [ ] **디렉토리 구조**: 핵심 디렉토리의 역할이 설명되어 있는가?

### 2. 좋은 CLAUDE.md 구조

```markdown
# 프로젝트명

[1문장 설명]

## 기술 스택
- 언어: Go 1.22+
- 프레임워크: ...
- DB: ...

## 시작하기

### 빌드
[빌드 명령]

### 테스트
[테스트 명령]

### 실행
[실행 명령]

## 코드 컨벤션
- [프로젝트 고유 규칙]
- [네이밍 규칙]
- [에러 처리 패턴]

## 금지 사항
- [하지 말아야 할 것]

## 디렉토리 구조
- `cmd/` — 엔트리포인트
- `internal/` — 비즈니스 로직
- `pkg/` — 공개 라이브러리
```

### 3. 개선 원칙

- **구체적으로**: "코드를 깨끗하게" → "함수는 30줄 이내"
- **실행 가능하게**: 복사 붙여넣기로 실행 가능한 명령
- **최신으로**: 오래된 명령/경로는 즉시 수정
- **짧게**: 에이전트가 매번 읽으므로 불필요한 내용 제거
- **프로젝트 고유하게**: 일반적인 Go/Python 규칙은 piki 표준에 위임

### 4. 흔한 개선 포인트

| 문제 | 개선 |
|------|------|
| 빌드 명령 없음 | `## 빌드` 섹션 추가 |
| 너무 김 (200줄+) | 핵심만 남기고 상세는 docs/ 링크 |
| 일반 규칙 반복 | piki 표준(rules/)에 위임, 프로젝트 고유만 유지 |
| 오래된 정보 | 실제 실행해서 검증 후 업데이트 |
| 금지사항 없음 | 프로젝트에서 실수하기 쉬운 것 추가 |

### 5. 검증

개선 후 확인:

- [ ] 새 팀원이 CLAUDE.md만 보고 빌드/테스트/실행 가능한가?
- [ ] 에이전트가 이 지침을 따르면 프로젝트 컨벤션에 맞는 코드를 생성하는가?
- [ ] 200줄 이내인가?

## 안티패턴

- CLAUDE.md에 모든 규칙 넣기 (piki 표준 활용)
- 실행 불가능한 명령 (검증 안 한 코드 블록)
- "좋은 코드를 작성하세요" 같은 추상적 지침
