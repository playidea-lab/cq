# Debugging

체계적 디버깅 워크플로우: 재현 → 가설 → 고립 → 수정 → 검증.

## 트리거

"디버깅", "debugging", "버그 추적", "원인 분석", "왜 안 되지", "에러 원인"

## 철학

> "디버깅은 추측이 아니라 과학이다. 가설을 세우고 실험으로 검증한다."

## Steps

### 1. 재현 (Reproduce)

수정하기 전에 반드시 재현:

- **무엇이** 잘못되는가? (기대 vs 실제)
- **언제** 발생하는가? (항상? 간헐적? 특정 조건?)
- **어디서** 발생하는가? (어떤 환경, 어떤 입력?)
- **재현 가능한가?** (재현 못 하면 수정 못 한다)

```bash
# 재현 스크립트 또는 테스트로 작성
# TDD: 실패하는 테스트 = 버그의 재현
```

**간헐적 버그**: race condition, 타이밍, 메모리 상태 의심.

### 2. 가설 수립 (Hypothesize)

가능한 원인을 나열 (최대 3개):

```markdown
가설 1: [가장 가능성 높은 원인]
  검증 방법: [어떻게 확인?]

가설 2: [두 번째 가능성]
  검증 방법: [어떻게 확인?]

가설 3: [세 번째 가능성]
  검증 방법: [어떻게 확인?]
```

**가설 수립 단서:**
- 에러 메시지 / 스택 트레이스
- 최근 변경사항 (`git log`, `git diff`)
- 동작하던 시점과의 차이
- 로그 / 메트릭의 이상 패턴

### 3. 고립 (Isolate)

범위를 좁힌다:

| 기법 | 방법 |
|------|------|
| **이분법** | 동작하는 커밋 ↔ 깨진 커밋 사이에서 `git bisect` |
| **제거법** | 코드를 하나씩 제거하며 문제가 사라지는 지점 탐색 |
| **최소 재현** | 문제를 재현하는 가장 작은 코드/입력 |
| **로깅** | 의심 지점에 로그 추가 후 흐름 추적 |

```bash
# git bisect 예시
git bisect start
git bisect bad HEAD
git bisect good v1.2.0
# → git이 이분 탐색으로 문제 커밋을 찾아줌
```

**핵심**: 한 번에 하나만 변경하며 테스트.

### 4. 수정 (Fix)

**최소한의 변경**으로 수정:

- 근본 원인을 수정 (증상이 아님)
- 수정 범위를 최소화
- 수정과 함께 재현 테스트 추가 (회귀 방지)

```
수정 전: 재현 테스트 실패 ✗
수정 후: 재현 테스트 통과 ✓ + 기존 테스트 통과 ✓
```

### 5. 검증 (Verify)

- [ ] 재현 테스트가 통과하는가?
- [ ] 기존 테스트가 모두 통과하는가?
- [ ] 원래 환경에서도 수정이 동작하는가?
- [ ] 유사한 곳에 같은 버그가 없는가? (패턴 검색)

### 6. 기록

```markdown
## Bug: [제목]
- **증상**: [사용자가 겪는 문제]
- **근본 원인**: [왜 발생했는가]
- **수정**: [무엇을 바꿨는가]
- **재현 테스트**: [추가한 테스트]
- **교훈**: [앞으로 방지할 방법]
```

## 흔한 버그 패턴

| 패턴 | 증상 | 확인 방법 |
|------|------|----------|
| Off-by-one | 마지막 항목 누락/초과 | 경계값 테스트 |
| Null/nil | 예상치 못한 crash | null check 추가 |
| Race condition | 간헐적 실패 | `-race` 플래그, 동시성 테스트 |
| State mutation | 예측 불가 동작 | 불변성, deep copy |
| Type coercion | 잘못된 비교/연산 | strict equality, 타입 확인 |
| Resource leak | 점진적 성능 저하 | defer/close, 프로파일러 |

## 안티패턴

- 재현 없이 "아마 이거겠지" 수정 (증상만 숨김)
- print 디버깅만 (체계적 가설 검증 없이)
- 여러 곳 동시 수정 (뭐가 효과인지 모름)
- 재현 테스트 없이 수정 (같은 버그 재발)
- "내 환경에서는 되는데" (환경 차이 무시)
