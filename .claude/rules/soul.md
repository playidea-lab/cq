# PI Lab — Soul (판단 기준)

> 에이전트가 "무엇을 하는가"는 agents.md가, "어떤 기준으로 판단하는가"는 이 파일이 정의한다.
> 리뷰, 설계 결정, PR 승인 등 판단이 필요한 순간에 참조.

---

## 비타협 규칙 (Critical)

1. **명세(DoD) 없는 작업은 '작업'이 아니다.**

2. **테스트 없는 병합은 원칙적으로 금지.**
   - 예외 허용 시: "왜/리스크/상환일"을 남겨야 함.

3. **큰 PR은 비용을 숨긴다.**
   - "작게 쪼개기"가 기본. 설계는 문서/ADR로 분리.

4. **시크릿은 코드에 없다.**
   - 하드코딩된 API 키, 비밀번호, 토큰 → 즉시 차단.

---

## 리뷰 우선순위 (높은 것부터)

1. **데이터 무결성 / 보안 / 권한** — 사용자 데이터 손상, 인증 우회, 권한 상승
2. **장애 복구 가능성** — rollback, migration idempotency, graceful degradation
3. **관측 가능성** — logging, metrics, tracing (장애 시 원인 파악 가능한가)
4. **테스트 커버리지 / 회귀 위험** — 변경이 기존 기능을 깨뜨릴 가능성
5. **가독성 / 스타일** — 그 다음

---

## 품질 철학

### "동작하는 최소한"
- 첫 구현은 가장 단순한 방법으로.
- 추상화는 3번 반복될 때.
- 미래 요구사항을 위한 설계 금지 (YAGNI).

### "실패를 가정한 설계"
- 네트워크는 끊긴다. 디스크는 찬다. 프로세스는 죽는다.
- 모든 외부 호출에 timeout. 모든 쓰기에 rollback 경로.
- "happy path만 테스트"는 테스트가 아니다.

### "코드는 읽히는 시간이 쓰는 시간보다 길다"
- 변수명은 의도를 드러내라.
- 주석은 "왜"를 설명하라 ("무엇"이 아니라).
- 매직 넘버 금지 — 상수로 추출.

---

## 설계 결정 기록

중요한 설계 결정은 ADR(Architecture Decision Record) 형식으로 기록:

```markdown
## ADR-XXX: [제목]
- **상태**: 제안 / 승인 / 폐기
- **맥락**: 왜 이 결정이 필요한가
- **결정**: 무엇을 선택했는가
- **근거**: 왜 이것을 선택했는가
- **대안**: 고려했지만 선택하지 않은 것
- **결과**: 이 결정으로 인해 변하는 것
```
