CQ Supabase PostgreSQL 스키마 레퍼런스 ​

목적: CQ 프로젝트의 Supabase PostgreSQL 스키마를 백엔드(A)/프론트엔드(B) 영역이 공유하는 계약 문서. 모든 테이블 정의의 SSOT는 supabase/migrations/ 마이그레이션 파일이며, 이 문서는 그 요약본이다.

개요 ​

CQ 스키마는 크게 네 영역으로 나뉜다:

접두어영역설명
c4_C4 코어프로젝트, 태스크, 지식 문서, 드라이브
hub_Hub 실행 엔진GPU 작업 큐, 워커, 메트릭, DAG
cf_CF Guard승인 큐, 감사 로그, 예산 (Streamable MCP 방어선)
cq_CQ v2Timeline 그래프 엣지, 텔레그램 채팅

Hub 테이블(hub_*)은 API 키 기반 멀티테넌시를 사용한다. project_id가 TEXT(UUID 문자열)이고 auth.users FK가 없다. Hub 서버는 service_role로 직접 접근하며, 클라이언트 JWT에 project_id claim을 포함하는 방식으로 RLS를 통과한다.

C4 테이블(c4_*)은 Supabase Auth + c4_project_members 기반 격리를 사용한다. project_id가 UUID이고 c4_projects(id) FK가 있다.

주요 테이블 ​

RLS 정책 요약 ​

C4 테이블 (c4_*) — Supabase Auth + project_members 격리 ​

모든 c4_* 테이블은 c4_is_project_member(project_id) 헬퍼 함수로 접근을 제한한다.

sql
-- 헬퍼 함수 (00001_c4_projects.sql)
CREATE OR REPLACE FUNCTION c4_is_project_member(p_project_id UUID)
RETURNS BOOLEAN AS $$
    SELECT EXISTS (
        SELECT 1 FROM c4_project_members
        WHERE project_id = p_project_id AND user_id = auth.uid()
    );
$$;
  • c4_tasks, c4_documents, c4_lighthouses 등: 프로젝트 멤버는 SELECT/INSERT/UPDATE/DELETE 모두 가능.
  • c4_project_members: owner/admin만 멤버 추가/삭제 가능.
  • c4_projects: 소유자만 수정/삭제.

Hub 테이블 (hub_*) — API 키 기반 격리 ​

sql
-- JWT claim에서 project_id 추출 (00038_hub_rls.sql)
CREATE OR REPLACE FUNCTION hub_jwt_project_id()
RETURNS TEXT AS $$
    SELECT current_setting('request.jwt.claims', TRUE)::jsonb ->> 'project_id';
$$;
  • service_role: 전체 접근 (Hub 서버가 service_role 사용).
  • authenticated: 자기 project의 행만 SELECT (project_id = hub_jwt_project_id()).
  • anon: 접근 없음.

hub_workers 등록/업데이트는 register_worker() RPC를 통해서만 허용 (직접 INSERT 불가).

CF 테이블 (cf_*) — 사용자 격리 ​

  • cf_pending_actions: 본인 행만 SELECT/INSERT. 본인만 cancelled/rejected 상태 전환 가능. 다른 상태 전환은 service_role만.
  • cf_audit_log: Append-only. authenticated 사용자는 본인 행 SELECT만. UPDATE/DELETE는 USING (false) 정책으로 명시적 차단.
  • cf_budget: service_role이 카운터 갱신. 사용자는 SELECT만.
  • cf_budget_limits: 사용자가 본인 행 READ/WRITE 가능. Streamable 채널에서의 한도 상향은 앱 레이어 차단.

cq_graph_edges — 개인 격리 ​

  • created_by = auth.uid() 기반. 본인 엣지만 SELECT/INSERT/DELETE.
  • UPDATE 정책 없음 (엣지는 불변).
  • service_role: BYPASSRLS (구조적 엣지는 CompletionHook 등이 service_role로 생성).

Telegram 알림 (DB Webhook — SQL 아님) ​

마이그레이션: 00049_telegram_notify_webhook.sql

Supabase Dashboard에서 DB Webhook으로 구성 (SQL로 관리되지 않음):

  • hub_jobs UPDATE → telegram-notify Edge Function
  • c4_tasks UPDATE → telegram-notify Edge Function

Edge Function이 터미널 상태(DONE/FAILED/done/blocked)만 필터링하여 Telegram 알림 발송.

주요 RPC 함수 ​

함수명설명
register_worker(...)워커 등록/갱신 + 구독 티어별 수 제한 검사
c4_is_project_member(UUID)프로젝트 멤버 여부 확인 (RLS 헬퍼)
hub_jwt_project_id()JWT claim에서 project_id 추출 (RLS 헬퍼)
cf_budget_default_limits()채널별 기본 예산 한도 반환

Cron ​

  • 00050_worker_offline_cron.sql: 일정 시간 heartbeat 없는 워커를 offline으로 전환.
  • 00066_cf_pending_ttl_cron.sql: 만료된 cf_pending_actions를 expired로 전환.

변경 시 주의사항 ​

마이그레이션 규칙 ​

  1. DROP COLUMN/TABLE 금지 — 대신 새 컬럼 추가 후 앱 레이어에서 마이그레이션. 라이브 서비스에서 DROP은 장애 원인.
  2. RLS ENABLE 필수 — 새 테이블 생성 시 반드시 ALTER TABLE ... ENABLE ROW LEVEL SECURITY; 추가. 누락 시 데이터 전체 노출.
  3. migration-lint.sh 실행 — DROP/RLS 검증. CI에서 자동 실행.
  4. A↔B 공유 테이블 변경 — c4_documents, c4_tasks, hub_jobs 등 양 영역이 쓰는 테이블 변경 시 백엔드 + 프론트엔드 담당자 동시 리뷰 필수.

Hub 테이블 특이사항 ​

  • hub_* 테이블의 project_id는 TEXT 타입. UUID처럼 생겼지만 FK 제약 없음. Supabase auth와 직접 연결되지 않는다.
  • submitted_by도 TEXT(UUID 문자열). auth.users FK 없음.
  • Hub 서버는 반드시 service_role 키로 접근. anon 키로는 RLS를 통과하지 못한다.

CF 테이블 특이사항 ​

  • cf_audit_log는 절대 UPDATE/DELETE 마이그레이션 작성 금지. append-only 보장이 깨지면 감사 가치 소멸.
  • cf_pending_actions의 상태 전환은 앱/서비스 레이어에서만. DB 직접 UPDATE 금지.

스키마 drift 방지 ​

  • c4_documents에 컬럼 추가 시 hub_job_completion_hook 함수의 INSERT 문도 함께 갱신 필요 (NOT NULL 컬럼 추가 시 트리거 INSERT가 실패함).
  • cf_pending_actions.status ENUM 확장 시 cf_budget_limits 기반의 CF Worker 검증 로직도 동기화 필요.
