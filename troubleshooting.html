CQ Troubleshooting ​

"왜 안 되지?" — CQ 사용 중 자주 마주치는 문제와 빠른 해결. 각 항목: 증상 → 원인 → 해결.

진단 우선 ​

뭐가 잘못됐는지 모르겠으면 먼저:

bash
cq audit              # 4표면 헬스 체크 (3초 안)
cq doctor             # 환경 진단
cq status             # 서비스 + 프로젝트 상태
cq intro --check      # 5 capability + auth/serve 진단

이 4개로 80%의 문제가 잡힙니다.

1. 인증 / Cloud Sync ​

cq auth login 브라우저 안 열림 (headless 서버) ​

증상: Failed to open browser 또는 무한 대기. 원인: SSH 세션 / X11 없음. 해결: --link 자동 사용. install.sh + ensureCloudAuth가 이미 SSH 감지하고 분기.

bash
cq auth login --link    # 명시적으로 강제

"cloud not configured" 에러 ​

원인: 릴리즈 빌드는 Supabase URL/anon_key를 ldflags로 bake함. cq update로 받았는지 확인. 해결:

bash
cq update           # 최신 릴리즈 바이너리
cq config get cloud # URL이 보여야 함

금지: ~/.c4/config.yaml에 anon_key를 직접 넣지 말 것 (메모리 cq-ops.md 참조).

"session expired" 반복 ​

원인: refresh token 갱신 실패. 해결:

bash
cq auth refresh <refresh_token>   # 또는
cq auth login --otp                # 이메일 OTP (브라우저 불필요)

2. MCP / 에이전트 ​

cq_xxx 도구가 에이전트에서 안 보임 ​

원인: MCP 서버는 세션 시작 시 바이너리 버전으로 고정됨. cq update 후 새 도구는 세션 재시작 필요. 해결:

bash
/reboot       # cq -t <name> 세션이면
# 또는 새 터미널에서 claude/cursor 다시 시작

에이전트가 cq_edge_declare 호출 안 함 ​

원인: MCP initialize Instructions에 컨벤션이 주입됐는지 확인. 해결:

bash
echo '{"jsonrpc":"2.0","id":1,"method":"initialize"}' | cq mcp 2>/dev/null \
  | jq -r '.result.instructions' | head -20

"Agent Edge Declaration"이 보여야 함. 안 보이면 빌드 누락 — cd apps/backend && make install.

cq_evidence_url share=true → null ​

의도된 동작. MCP는 사용자 JWT가 없어 토큰을 직접 발급 못 함. 응답의 share_instructions로 사용자에게 POST 레시피 제공. (T-EVW-6 결정)

"encrypted dispatch required — sender_pub missing" ​

원인: MCP 서버에 relay 키가 없음. 해결: /reboot. (cq serve 재시작 아님 — 세션 자체를 재시작)

3. 워커 / Hub ​

워커가 새 태스크를 안 잡음 ​

원인 후보:

  1. 우선순위 밀림 — 새 태스크 priority 0 vs 레거시 100
  2. target_worker 누락 — GPU 잡인데 아무 워커나 가져갈 수 있음
  3. dependency 미해소

해결:

bash
# DB로 직접 확인:
sqlite3 .c4/c4.db "SELECT task_id, status, priority, dependencies FROM c4_tasks WHERE task_id='T-XXX-N';"

# priority 올리기:
sqlite3 .c4/c4.db "UPDATE c4_tasks SET priority=200 WHERE task_id='T-XXX-N';"

# GPU 잡은 반드시 target_worker:
cq_job_submit(target_worker="pi-System-Product-Name", required_tags=["gpu"], ...)

"in_progress" 상태로 멈춤 (stale worker) ​

증상: 8시간 이상 진행 없음, 워커 ID는 있지만 연결 끊김. 해결:

bash
sqlite3 .c4/c4.db "UPDATE c4_tasks SET status='pending', worker_id=NULL WHERE task_id='T-XXX-N';"

worker_id는 fully cleared 해야 새 워커가 픽업.

워커 브랜치 fast-forward 불가 ​

원인: 워커가 c4_submit 후 main이 다른 commit을 받음. 해결:

bash
git checkout main
git stash push -u                              # working dir 정리
git merge --no-ff c4/w-T-XXX-N                # 명시적 merge commit
git push origin main

또는 cherry-pick 후 워커 브랜치 폐기.

c4/w-* 브랜치 누적 ​

증상: git branch | wc -l 100+ 해결: done 태스크의 워커 브랜치 일괄 삭제.

bash
git worktree prune
git branch --merged main "c4/w-*" | grep -v "^[\*\+]" | xargs git branch -d

worktree에 부착된 것은 먼저 git worktree remove.

4. 빌드 / 실행 ​

go build 후 변경 반영 안 됨 ​

원인: cq PATH 바이너리는 ~/.local/bin 또는 GitHub Releases 버전. 로컬 빌드는 다른 위치. 해결:

bash
cd apps/backend && make install   # ldflags 포함 + ~/.local/bin 설치

금지: go build ./cmd/c4/ 직접 사용 시 version=dev 문제 + ldflags 누락.

cq serve start 후에도 작동 안 함 ​

증상: 서비스 시작했는데 cq status가 비어 있음. 원인 후보:

  1. 백그라운드 startup 실패 — 로그 확인
  2. 포트 충돌 (4143, 4144)
  3. systemd/launchd 권한

해결:

bash
cq serve              # 포그라운드로 실행 → 로그 직접 확인
lsof -i :4143         # 포트 점유 확인
launchctl list | grep cq    # macOS 서비스 상태
systemctl --user status cq-serve   # Linux

make install 후 MCP 도구가 그대로 ​

원인: 세션이 이전 바이너리 사용 중. 위 §2 참조.

5. Evidence Web ​

/exp/{id} 또는 /doc/{id} 404 ​

원인: RLS가 row를 안 보여줌. 의도된 보안 동작 (private leak prevention). 진단:

bash
# 본인 소유인지 확인
sqlite3 .c4/c4.db "SELECT created_by_user_id FROM hub_jobs WHERE id='<id>';"
# auth.uid()와 일치해야 함

해결:

  • 다른 사람 실험이면 owner에게 share token 발급 요청 (POST /api/share)
  • 본인인데 안 보이면 cookie 만료 — 재로그인

Share token이 작동 안 함 ​

원인 후보:

  1. 토큰 만료 (7일 기본)
  2. resource_type 또는 resource_id 불일치
  3. revoked (cf_revoked_share_tokens 확인)

진단:

bash
# 토큰 디코드 (jwt.io)
# 또는 만료 확인
echo "<token>" | cut -d. -f2 | base64 -d | jq .exp

Metric chart 비어 있음 ​

원인: hub_metrics 테이블에 step/value 데이터 없음. 확인: 학습 스크립트가 @key=value 형식으로 print하는지.

python
print(f"@epoch={epoch} @loss={loss:.4f}")   # cq MetricWriter가 자동 파싱

6. 데이터 / DB ​

"PostgREST 1000 row limit" ​

원인: PostgREST 기본 응답 페이지 크기 1000. 해결: 이미 v1.64.0에서 fix — 10k 요청. 더 필요하면 페이지네이션.

c4.db lock 에러 ​

원인: 두 프로세스가 동시 쓰기 (cq serve + 직접 실행). 해결:

bash
cq serve stop          # 백그라운드 종료
# ... 작업 ...
cq serve start

DB 파일은 .c4/c4.db, WAL 모드라 일반적으로 락 안 잡힘. 잡혔으면 process 충돌.

cf_audit_log 못 읽음 ​

원인: RLS — user_id = auth.uid() 본인 행만. 해결: 본인 토큰으로 쿼리 또는 service_role 사용 (워커/CF Worker 한정).

7. CI / 배포 ​

GitLab tag 후 GitHub Releases 미반영 ​

원인: GitLab CI 빌드 5-10분 소요. 해결: https://github.com/PlayIdea-Lab/cq/releases 5분 후 확인. 매번 수동 push 금지 (memory github_release_only.md).

wrangler pages deploy 실패 — secret 누락 ​

원인: production env vars가 git에 없어 빌드 시 누락. 해결:

bash
wrangler pages secret put NEXT_PUBLIC_SUPABASE_URL --project-name cq-evidence
wrangler pages secret put NEXT_PUBLIC_SUPABASE_ANON_KEY --project-name cq-evidence
wrangler pages secret put EVIDENCE_SHARE_SECRET --project-name cq-evidence

태그 충돌 (v1.X.Y already exists) ​

원인: 이전에 부분 push된 태그. 해결: patch 증가 (예: v1.63.0 충돌 → v1.63.1). 메모리 cq_v2_phase0_edges.md 참조.

8. 흔한 실수 (메모리 "cq-ops.md"에서) ​

실수올바른 방법
config.yaml에 anon_key 넣기넣지 마. 바이너리에 bake됨
cq serve stop && start로 relay 해결세션 재시작 (/reboot)
target_worker 없이 GPU job 제출반드시 target_worker 지정
hub.team_id에 문자열UUID만 (c4_project_members 매칭)
cq serve install로 백그라운드 시작cq serve start 사용
base64로 파일 stdout 전송Drive 또는 cq_relay_call + cq_read_file

그래도 안 되면 ​

  1. cq audit --json 출력 + cq doctor 결과를 캡처
  2. git log --oneline -5 로 최근 변경 확인
  3. memory 파일에서 키워드 검색: grep -r "키워드" ~/.claude/projects/-Users-changmin-git-cq/memory/
  4. docs/specs/ 에서 관련 스펙 확인
  5. 새로운 패턴이면 memory/feedback_*.md 에 기록
