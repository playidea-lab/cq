Feature Freeze Checklist — Compound Pilot ​

During the 4-week pilot window, the following changes are banned. Any PR touching these files or patterns must be blocked until the pilot concludes.

Automated enforcement: scripts/check-pilot-freeze.sh (CI check, DEC-003).

Banned Keywords / Patterns ​

The CI check (scripts/check-pilot-freeze.sh) rejects any diff that adds or modifies lines matching these patterns in Go, TypeScript, or Python source files:

PatternReason
recommend_nextProactive suggestion to run another experiment
proactive_suggestAny unsolicited next-step recommendation
nudgeAny nudge toward an action
next_experimentAutomated "try this next" feature
compound_suggestCompound loop suggestion trigger
auto_linkAutomatic edge declaration (must be manual during pilot)

Banned File Changes ​

The following files must not be modified during the pilot window:

File / DirectoryReason
c4-core/internal/onboarding/Onboarding copy changes invalidate natural measurement
c4-core/cmd/c4/mcp_handlers_research_suggest.goResearch suggestion handlers
Any file adding new email or Telegram broadcast to pilot usersInvalidates no-nudge principle
c4-core/internal/context_injection/New automatic context injection triggers
supabase/migrations/*_nudge*New nudge-related DB changes

Banned Change Categories ​

Even if the specific file is not listed, the following types of changes are banned:

  1. New nudge or recommendation features — any code that detects a user state and proactively sends a message or suggests an action.

  2. New automatic context injection triggers — the pilot measures natural usage; injecting context automatically biases the signal.

  3. New onboarding copy changes — the cohort was onboarded under the current copy; mid-pilot copy changes create an uneven cohort.

  4. New email or Telegram broadcast to pilot users — even informational messages can influence behavior and invalidate the natural-usage measurement.

  5. Changes to pilot_cohort_metrics, pilot_chain_depth, or pilot_verdict — changing the measurement instrument during the pilot invalidates comparability.

Permitted Emergency Exceptions ​

Only the following changes are permitted without lifting the freeze:

Change TypeCondition
P0 security fixConfirmed exploit in production; no alternative
Authentication / session bugUsers cannot log in or data is at risk
Data loss fixPilot user data at risk of corruption
Infrastructure (DNS, TLS, relay)Service completely down

All exceptions require a comment in the PR: PILOT-FREEZE-EXCEPTION: <reason>. The exception must be reviewed by at least one team member before merge.

Lifting the Freeze ​

The feature freeze is lifted after:

  1. cq pilot report has been generated and the verdict recorded.
  2. The verdict and raw report have been archived in the project knowledge base.
  3. The pilot window end date has passed.

To lift: remove the PILOT_ACTIVE=true environment variable from the CI configuration (or merge the freeze-lift PR).

자동 강제 (T-CP-3) ​

이 체크리스트는 scripts/check-pilot-freeze.sh 로 CI에서 자동 검사됩니다.

CI 플랫폼Job 이름트리거
GitHub ActionsPilot Freeze Check모든 PR to main, 모든 push
GitLab CIpilot-freeze-check모든 브랜치

파일럿 라이프사이클 ​

bash
# 파일럿 시작 — .c4/pilot-active 센티널 생성, CI 가드 활성화
cq pilot start --cohort <이름> --weeks 4

# 파일럿 종료 — 센티널 삭제, CI 가드 비활성화
cq pilot stop

# 파일럿 결과 리포트 생성
cq pilot report --out pilot-report.md

작동 방식 ​

  1. .c4/pilot-active 파일이 없으면 → "pilot not active; skipping freeze check" 후 exit 0 (차단 없음).
  2. 파일이 있으면 → git diff main..HEAD에서 추가된 코드 행을 스캔.
  3. 금지 키워드 발견 시 → 파일명 + 키워드 출력 후 exit 1 (머지 차단).
  4. 주석 행 (//, #, *) 및 docs/pilot/, .c4/ideas/, .c4/designs/, docs/specs/ 경로는 스캔에서 제외.

긴급 우회 ​

bash
bash scripts/check-pilot-freeze.sh --override

PR 본문에 반드시 추가: PILOT-FREEZE-EXCEPTION: <사유>
