Compound Pilot — 4-Week Operator Playbook ​

Purpose: Validate that the CQ Compound loop triggers naturally in the wild. Pass threshold γ: at least one pilot user declares a derives_from chain ≥ 3 deep within the 4-week window — without any nudges or artificial prompts.

Purpose ​

The 4-week pilot measures whether BYO-GPU researchers, left to work naturally with CQ, spontaneously build experiment chains that form a Compound loop. This is the critical early-market signal before broader rollout (Stage 4/5).

The single hypothesis: Compound loop will emerge without encouragement.

γ threshold: at least 1 pilot user with derives_from chain depth ≥ 3.

All measurement is passive. No nudges, no "try this next" suggestions, no proactive outreach during the pilot window.

Pre-Flight Checklist ​

Before inviting the first participant, confirm all of the following:

  • [ ] T-SLO-3 merged — /me/compound personal dashboard live (R11)
  • [ ] T-OBD-1 – T-OBD-8 merged — invite code + worker register flow live (R2, R4, R5)
  • [ ] T-CP-1 merged — pilot_cohort_metrics, pilot_chain_depth, pilot_verdict in production
  • [ ] T-CP-2 merged — cq pilot report CLI available in release binary
  • [ ] RLS verified — each pilot user's data is isolated (no cross-user leakage)
  • [ ] Feature freeze active — nudge / recommend_next / proactive_suggest code paths disabled
  • [ ] Invite codes generated — 5 codes, one per participant
  • [ ] Pilot start date recorded — used as --since in cq pilot report
  • [ ] Operator contact established — Telegram handle or form URL confirmed with each participant

Week 0: Participant Selection + Invite Code Issuance ​

Selection criteria ​

  • BYO-GPU researcher (has their own GPU hardware or cloud GPU access)
  • Already runs iterative ML/DL experiments (at least 2 per week historically)
  • Willing to install CQ and use it as primary session tool for 4 weeks
  • No prior CQ usage (to avoid training effect)

Invite code issuance ​

  1. Generate 5 invite codes via the invite code admin interface.
  2. Send each participant the invite message from invite-template.md.
  3. Log the mapping: invite_code → participant_handle in a private document.
  4. Record the pilot start date (PILOT_START=YYYY-MM-DD).

Do not share invite codes publicly. Each code is single-use and ties the participant to their own isolated team.

Week 1–4: No-Touch Period ​

The no-touch principle is the single most important rule of the pilot.

What operators MUST NOT do ​

  • Do not send "how is it going?" messages.
  • Do not suggest experiments to run.
  • Do not share other participants' results.
  • Do not deploy any new features that add nudges or suggestions.
  • Do not change onboarding copy or email broadcasts to pilot users.
  • Do not announce pilot results publicly until Week 4 closes.

What operators MAY do ​

  • Answer direct support questions (installation failures, auth issues).
  • Fix critical bugs unrelated to the Compound loop (P0/P1 only — see feature-freeze-checklist.md).
  • Monitor error logs passively.

Weekly operator check (5 minutes, optional) ​

bash
# Confirm pilot users are authenticated and active (no diagnostic data leaked).
cq pilot report --since PILOT_START --out /tmp/pilot-draft.md

Do not share this draft with participants.

Week 4 End: cq pilot report + Interpretation ​

Generate the final report ​

bash
cq pilot report --since YYYY-MM-DD --until YYYY-MM-DD --out pilot-final.md

Replace YYYY-MM-DD with the actual pilot start and end dates.

Interpret the verdict ​

VerdictMeaningNext Step
pass≥ 1 user reached chain depth ≥ 3 (γ met)Proceed to Stage 4/5 (broader rollout)
inconclusiveNo user submitted a 2nd experimentRe-examine Compound hypothesis; consider Stage 1 (entry) rework
failActivity recorded but γ not metStage 1 rework is a required prerequisite before re-running

Verdict actions ​

pass ​

  • Archive pilot-final.md in the project knowledge base.
  • Tag the pilot cohort for Stage 4/5 early access.
  • Remove the feature freeze.
  • Announce internally.

inconclusive ​

  • Do not proceed to Stage 4/5.
  • Run a structured retrospective: why did users not run a 2nd experiment?
    • Was the entry experience (T-OBD-1~8) confusing?
    • Was the GPU setup too high-friction?
    • Was the value proposition unclear?
  • Document findings and feed back into the Compound hypothesis.
  • Re-run the pilot only after Stage 1 changes are validated.

fail ​

  • Stage 1 rework is a mandatory prerequisite (R9).
  • Do not attempt partial rollout.
  • Document the specific failure pattern (which chain depths were reached, by how many users).
  • This data is the primary input for Stage 1 redesign.
